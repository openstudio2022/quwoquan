## Context

当前应用已存在小趣相关入口（如悬浮球、会话入口、媒体工具栏入口），但入口位置、触发方式、页面语义不一致。欢迎页当前已具备成熟视觉骨架，发现页当前存在多类内容入口与搜索导向，尚未形成“助理先整理、用户再决策”的统一体验。  
本次变更采用“最小可上线重构”策略：仅升级欢迎页与发现页，并建立小趣统一交互层 V1，其他频道作为后续阶段推进。

## Goals / Non-Goals

**Goals:**
- 建立小趣统一交互心智：全局只强调一个动作“找小趣（什么都可以找Ta）”。
- 欢迎页在不大改结构前提下，升级为“兴趣圈 + 小趣陪伴”的单主句叙事。
- 发现页一级信息架构升级为「帮读 / 美图 / 视频」，将微趣与文章聚合为“帮读”体验。
- 定义小趣三形态（帮读卡/任务抽屉/会话页）与主动触发边界，避免打扰沉浸浏览。
- 输出可实施规范，确保后续实现可按阶段扩展到圈子、创作、趣聊与小趣主页。

**Non-Goals:**
- 不在本次内完成圈子、创作、趣聊、小趣主页、我的页的深度 IA 重构。
- 不在本次内上线第三方 Skill 市场、自动代发、复杂多代理编排。
- 不引入新的后端依赖或账号/权限模型重构（V1 先以前端交互与内容编排规则为主）。

## Decisions

### 决策 1：采用“1 个全局入口 + 3 种形态”统一交互协议
- 方案：在所有场景统一“找小趣”作为唤起语义；按任务复杂度从轻到重切换为帮读卡、抽屉、会话页。
- 原因：相比为每个页面设计独立助手入口，这种方式更易建立用户心智，且降低跨页面学习成本。
- 备选方案：各页面保留独立入口与独立文案。  
  放弃原因：会持续放大“无处不在=到处不一致”的混乱感。

### 决策 2：发现页聚焦三分类，帮读采用“二级分层”承载文本型高认知内容
- 方案：一级分类固定为「帮读 / 美图 / 视频」。帮读内部采用二级分层：上半区为“小趣总结抽屉”，下半区保留“微趣/文章”折叠入口，支持展开微趣列表与进入文章列表。
- 原因：图片/视频消费是沉浸主路径，文本型内容适合助理预处理，能显著降低用户筛选与阅读成本。
- 备选方案：保留微趣与文章独立一级分类。  
  放弃原因：入口过多，弱化助理价值，且与“简单统一”目标冲突。

### 决策 3：欢迎页采取“轻升级”而非版式重构
- 方案：保留主视觉骨架与动效，只替换为单主句叙事并引入小趣轻量形象/语义。
- 原因：在不增加改造成本的情况下，最快建立“兴趣圈 + 助理陪伴”品牌感知。
- 备选方案：完整重做欢迎页流程与多屏文案。  
  放弃原因：影响范围大、回归成本高，不利于本次阶段目标。

### 决策 4：主动式能力默认克制并可反馈
- 方案：主动触发仅限关键时刻提醒、高价值信息提示、单步行动建议；每次主动均可“稍后提醒/不再提醒”。
- 原因：保证“有温度但不打扰”，并用反馈闭环驱动个性化优化。
- 备选方案：更积极的持续主动推送。  
  放弃原因：短期内容易造成干扰，损害沉浸体验与信任。

### 决策 5：帮读抽屉采用“高级感分层交互”而非普通弹层
- 方案：抽屉采用 Peek/Half/Full 三段吸附状态，配合弹簧曲线、轻量触觉反馈、动态圆角与层次阴影；在展开时遵循“容器先到位、内容后渐显”的双阶段动效。
- 原因：该设计可提供接近系统级产品（如 iOS 原生）的一致操控预期，提升品质感并减少突兀打断。
- 备选方案：仅使用普通 BottomSheet（开/关两态）。  
  放弃原因：状态过少、反馈生硬，无法承载“总结抽屉 + 列表入口”的层次信息。

## Risks / Trade-offs

- [Risk] 发现页改为三分类后，老用户对“微趣/文章”入口路径感知变化。  
  → Mitigation：在“帮读”首屏采用“上总结抽屉 + 下微趣/文章折叠入口”的二级分层；保留渐进迁移提示与原文回跳路径。

- [Risk] 小趣入口统一后，部分页面短期会出现入口语义与旧文案并存。  
  → Mitigation：以常量统一“找小趣”命名，短期将“问小趣”作为兼容别名，分批替换并在验收中加入一致性检查。

- [Risk] 抽屉动效复杂度提升可能导致低端机掉帧或手势冲突。  
  → Mitigation：定义统一吸附点与手势阈值，提供降级动画策略，并在关键机型做帧率回归测试。

- [Risk] 主动提醒边界过窄导致“智能感”不足。  
  → Mitigation：V1 先稳态上线，后续通过用户设置中的主动级别逐步放开。

- [Risk] 本次仅改欢迎页与发现页，用户会预期全站已完成升级。  
  → Mitigation：在文档与任务中明确存量范围与下一阶段路线，避免范围误解。
