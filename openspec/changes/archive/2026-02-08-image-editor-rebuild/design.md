## Context

- 当前创作流中的图片编辑为 CreatePage 内某一 Tab（ImageEditor），布局与能力较简单，无统一的三段式布局、无 Snapseed 式记录、无完整工具链与标准化操作面板。
- **参考图**：图一为整体布局（顶栏返回/序号/完成、中部图片、底栏编辑按钮：裁剪、模板、滤镜、文字、画笔、马赛克等）；图二为选中「滤镜」后的操作面板（分类栏、滤镜模板列表、底部取消/保存与强度滑块）；图三为选中「裁剪」后的界面（裁剪框与九宫格、比例选项如自由/原图/正方形/3:2、底部取消/撤销/更多）。实现时以三图布局与层级为准。
- 产品要求对齐 Snapseed、醒图等专业修图工具：顶栏（返回、序号、蓝色完成）、中部图片、底栏工具；完成前可查看/删除/重做步骤；底栏工具包含旋转、裁剪、滤镜、专业修图（曲线/白平衡/局部/修复/美丽光晕/色调对比度）、相框、文字、涂鸦、马赛克；选中某工具后展开「分类 + 模板/内容 + 取消/保存 + 比例或强度」的操作面板。
- 约束：须使用 AppColors、AppSpacing、UITextConstants；完成按钮必须为主色蓝；与 create-flow 集成时由创作页图片 Tab 承载同一套重建后的编辑器。

## Goals / Non-Goals

**Goals:**

- 实现图片编辑器的三段式布局（顶栏、中部、底栏），顶栏含返回、图片序号、蓝色完成按钮。
- 实现完成前的编辑记录：记录每步操作，支持查看步骤列表、删除某步、对某步重做。
- 底栏提供旋转、裁剪、滤镜、专业修图、相框、文字、涂鸦、马赛克入口；专业修图内含曲线、白平衡、局部、修复、美丽光晕、色调对比度等子工具。
- 非「专业修图」的工具选中时，展开统一结构的操作面板：顶部分类、中间模板/内容、底部取消（叉）与保存（勾），以及可选的比例/强度等控件。
- 裁剪时展示可调裁剪框与比例选项（自由、原图、正方形、3:2 等）；滤镜时展示分类、滤镜模板与强度滑块。
- 与 create-flow 集成：创作页「图片」Tab 使用本重建后的编辑器。

**Non-Goals:**

- 本设计不规定具体滤镜算法或第三方图像库选型，仅约定 UI 结构与数据接口。
- 视频编辑、文章编辑不在本次范围内。
- 云端同步编辑记录、多设备协同为后续考虑，不在本设计内。

## Decisions

- **编辑状态与记录模型**：采用步骤栈（step stack）存储编辑操作序列；每步包含类型（旋转/裁剪/滤镜/…）、参数与缩略图或预览引用。删除步骤即从栈中移除该步并重算其后步骤；重做即对该步重新应用并允许调整参数。可选：步骤列表以底部抽屉或顶栏入口展开，参考 Snapseed 的「查看修改内容」。
- **顶栏完成按钮颜色**：完成按钮使用 `AppColors.primaryColor`（蓝色），禁止使用红色，与设计语义一致；文案使用 UITextConstants 或等价常量。
- **操作面板层级**：底栏选中某工具（除专业修图）后，在底栏上方以滑动面板或固定区域展示「分类 + 模板/内容 + 取消/保存 + 比例/强度」；专业修图选中后进入子界面，内部再按子工具（曲线、白平衡等）呈现类似面板结构。这样避免与底栏平级的多个面板同时展开。
- **专业修图入口**：作为底栏一个入口，点击后进入「专业修图」子模式，子模式内提供曲线、白平衡、局部、修复、美丽光晕、色调对比度等入口；每个子工具使用与「滤镜/裁剪」一致的面板结构（分类、参数、取消/保存），便于复用组件与交互。
- **多图序号**：顶栏图片序号（如 1/5）在多图场景下显示当前第几张、共几张；单图时可为 1/1。序号仅展示，切换逻辑（如有）由上层或路由决定。
- **集成方式**：create-flow 的 CreatePage 在「图片」Tab 下挂载重建后的图片编辑器组件（全屏或占满 Tab 区域），传入初始图片列表或单图；编辑器内部不关心来源，只负责布局、记录与工具，完成后通过回调或状态将结果回传给创作页。

**替代方案简述：**

- 记录用「全局撤销/重做」而非步骤列表：未采用，因需求明确要求可查看、删除单步、重做单步。
- 完成按钮用红色：未采用，需求明确要求蓝色。

## Risks / Trade-offs

- **[性能]** 多步滤镜/曲线等实时预览可能卡顿 → 对重操作采用缩略图或降分辨率预览，或限制单会话步数并提示。
- **[包体积]** 引入图像处理库会增大包体 → 优先选用轻量或可按需加载的库，并在设计上预留「仅下载必要滤镜包」的扩展点。
- **[兼容]** 替换现有 ImageEditor 可能影响已有草稿或入口 → 迁移计划中保留旧编辑器路由或版本开关，逐步切量；草稿格式需兼容或做一次性迁移。
- **[复杂度]** 专业修图子工具多，一次性实现工作量大 → 可先实现核心子工具（如曲线、白平衡、局部），其余按优先级迭代。

## Migration Plan

1. 在 feature 或 create 模块内新增重建后的图片编辑器页面/组件，按 design 与 specs 实现布局、记录与工具。
2. 在 CreatePage 的图片 Tab 中，将原有 ImageEditor 替换为新编辑器组件，保证入口与返回、完成回调不变。
3. 若存在草稿或编辑中状态，定义新旧格式兼容策略（或仅新会话使用新编辑器，旧草稿仍用旧编辑器打开一次并导出后再用新编辑器）。
4. 上线后通过配置或灰度控制使用新编辑器的比例；保留回滚到旧编辑器的开关。
5. 全量后移除旧 ImageEditor 与相关死代码。

## 草稿与兼容策略（6.3）

- **过往版本图片草稿格式**：CreateEditImagePage 为无状态编辑器，不持久化编辑步骤；草稿仅存在于 CreatePage 的 `_currentData['photo']['images']`，存储图片路径。
- **兼容策略**：重建后 RebuiltImageEditorPage 同样以 `context.pop(editedPath)` 返回编辑后路径，CreatePage 将返回的路径替换列表中对应项。无需迁移，新旧编辑器接口一致。
- **编辑中步骤栈**：当前步骤栈仅存在于单次编辑会话内存，完成/返回后不持久化；后续若需草稿恢复可扩展。

## Open Questions

- 多图编辑时，记录是「每张图独立步骤栈」还是「全局共享步骤」？建议每张图独立步骤栈，序号切换时切换当前步骤栈。
- 专业修图各子工具（曲线、局部等）的默认参数与预设是否由设计提供？可先定交互与占位参数，预设后续补充。
