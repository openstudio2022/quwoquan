## Context

当前图片编辑器已经完成「基础」分组的专业调节（亮度、对比度等）和统一底部操作框架，但 HSL 分组仍为占位，尚不具备商用调色能力。用户已经明确要求：

- 黑色编辑背景下的高可读中性视觉，不使用干扰性的红色。
- 专业工具交互对齐行业主流（醒图、Lightroom/Camera Raw）：
  - 颜色通道选择（红/橙/黄/绿/青/蓝/紫 + 取色器）
  - H/S/L 三参数调节
  - 会话内 undo/redo
  - 长按或点击对比原图
- 与当前「基础」分组共存，且状态隔离，避免互相污染。

项目约束：

- 必须使用设计系统语义 token（`AppColors`、`AppSpacing`、`UITextConstants`），禁止硬编码。
- 现有编辑器结构需保持可扩展（后续还会继续补齐曲线等专业能力）。
- 当前主编辑器在单页内聚合较多状态，新增 HSL 时要控制复杂度。

## Goals / Non-Goals

**Goals:**

- 提供可交付的 HSL 完整面板（非占位）并接入现有专业修图入口。
- 定义稳定参数模型：`channel × {hue, saturation, luminance}`，范围统一为 `-100..100`，默认 `0`。
- 支持取色器：从图片采样颜色并映射到目标通道，提升调色效率。
- 支持 HSL 会话级撤回/重做与原图对比，不影响全局历史语义。
- 抽象出系统级会话操作条，支持其它工具（如曲线、局部、曝光）复用同一交互范式。
- 保持底部 `X / ✓` 与基础分组一致的尺寸、样式、点击区域。

**Non-Goals:**

- 本次不实现曲线分组完整能力（曲线维持独立后续变更）。
- 本次不实现跨分组统一时间线编辑器（只做 HSL 会话内 undo/redo）。
- 本次不引入新的三方图像处理依赖（优先使用 Flutter 图像管线与现有处理方式）。

## Decisions

### 1) HSL 数据模型采用「按通道存储 + 三轴参数」

**Decision**

- 新增 `HslChannelType`（red/orange/yellow/green/cyan/blue/purple）。
- 新增 `Map<HslChannelType, HslAdjustValue>`：
  - `hue: double`
  - `saturation: double`
  - `luminance: double`
- 参数范围全部 `-100..100`，默认 `0`。

**Rationale**

- 与 Lightroom/Camera Raw/醒图用户心智一致，学习成本最低。
- 可天然支持“切换通道时恢复该通道当前值与滑线位置”的需求。

**Alternatives**

- 方案 A：统一单通道临时值 + 提交时写回。  
  缺点：切换通道时状态复杂，容易出现值错位。
- 方案 B：HSV 角度制（0..360）存储。  
  缺点：与现有 UI 滑线（中位 0）语义冲突，调参不直观。

### 2) 取色器采用「点采样 -> 最近通道映射」策略

**Decision**

- 进入取色器后，图片区显示十字指针与放大镜预览。
- 在触点采样像素的 HSV Hue，映射到七个通道预设区间：
  - red/orange/yellow/green/cyan/blue/purple
- 通道选中态采用外圈强调（非蓝色主视觉），并提供通道名/色块语义提示。

**Rationale**

- 用户不需要理解色轮细节，也能快速定位目标颜色。
- 七通道映射符合主流 HSL 分组习惯。

**Alternatives**

- 方案 A：自由 hue 连续选择（无分组）。  
  缺点：操作复杂，不符合当前产品的“易用优先”定位。
- 方案 B：仅按钮切换通道，无取色器。  
  缺点：复杂图片中效率明显不足。

### 3) 会话级 undo/redo 与全局历史分层

**Decision**

- HSL 面板内维护局部操作栈（通道/参数修改快照）。
- `X`：回滚到进入 HSL 面板时快照并退出。
- `✓`：将最终 HSL 参数写入全局编辑历史一步（`subType: hslAdjustments`）。

**Rationale**

- 兼顾专业调色高频试错与主编辑器历史整洁。
- 避免每次微调都污染全局历史。

**Alternatives**

- 方案 A：每次滑动都写全局历史。  
  缺点：历史爆炸、性能差、用户难以管理。

### 4) 原图对比采用「按压临时旁路」预览

**Decision**

- 右侧对比按钮按下时临时禁用 HSL 调整层，抬起恢复。
- 点击切换可选（按产品手势偏好配置），默认优先按压对比。

**Rationale**

- 主流工具一致体验，快速判断调色收益。
- 不改动当前参数，不影响 undo/redo 栈。

### 5) 会话操作条系统级抽象并放置在图片底部

**Decision**

- 把 `undo/redo + compare` 从 HSL 面板内部逻辑抽离为 `EditorSessionOpsStrip`（共享组件）。
- 组件挂载在图片内容区底部（不与 `X / ✓` 底栏冲突），由工具会话注入能力开关：
  - `supportsUndoRedo`
  - `supportsCompare`
  - `sessionBaselineProvider`
- 对比基线统一定义为“进入当前工具会话时的图像状态”，而不是全局初始原图。

**Rationale**

- 后续工具可直接复用，避免重复实现和行为不一致。
- 用户在不同工具下形成稳定心智：左撤回/重做，右对比基线图。

**Alternatives**

- 方案 A：每个工具自己实现一套按钮和状态。  
  缺点：样式和行为容易漂移，维护成本高。

### 6) 渲染策略采用「GPU 预览 + 导出时落地」

**Decision**

- 预览阶段：通过颜色变换层实时作用于预览图。
- 导出阶段（完成返回上层时）：将 HSL 参数渲染到输出图，避免仅 UI 预览生效。

**Rationale**

- 保证交互流畅与最终结果一致。

**Alternatives**

- 方案 A：每次滑动都生成新文件。  
  缺点：IO 频繁，卡顿明显。

## Risks / Trade-offs

- **[风险] HSL 精确数学模型复杂，简单矩阵难以覆盖所有边缘颜色**  
  → **Mitigation**：首版采用稳定可控映射，关键样图回归；后续预留更高阶色域映射升级点。

- **[风险] 取色器采样在缩放/旋转预览下坐标换算易错**  
  → **Mitigation**：统一采样坐标变换函数，新增专门单测与手测清单（缩放、旋转、横竖图）。

- **[风险] 会话 undo/redo 与全局历史语义冲突**  
  → **Mitigation**：严格限定会话栈只在 HSL 面板内生效，`✓` 时仅产出单步全局记录。

- **[风险] 黑底下色彩通道可读性不足**  
  → **Mitigation**：通道选中采用亮度/描边/外圈组合语义，并保持文字对比度达标。

- **[风险] 系统级会话操作条接入多个工具后状态边界不清**  
  → **Mitigation**：统一定义会话生命周期（enter/update/commit/cancel）与基线快照时机，组件只读工具提供的状态接口。

## Migration Plan

1. 扩展专业修图状态结构，加入 HSL 参数模型与会话快照。
2. 新增系统级会话操作条组件（undo/redo/compare），并先接入 HSL。
3. 新增 HSL 面板组件（通道栏、三参数滑线、底部操作栏）。
4. 接入取色器采样逻辑与通道映射。
5. 接入预览渲染与导出落地流程。
6. 更新 OpenSpec 规格与任务，执行回归测试。
7. 灰度发布：先对内部测试渠道开启，验证性能与视觉一致性。

**Rollback**

- 保留现有基础分组路径，HSL 能力通过 feature flag 或入口开关可快速回退为占位。

## Open Questions

- 取色器交互是“长按持续采样”还是“点击单次采样”为主？是否双模式并存。
- 对比原图按钮最终是“按压生效”还是“点按切换”作为默认行为。
- HSL 与基础分组叠加顺序是否固定（先基础后 HSL）或可配置（影响视觉结果）。
