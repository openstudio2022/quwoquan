## Context

当前图片编辑器已具备专业修图入口与二级分组，但分组语义是“曝光/色彩/光影/质感”，且面板模板偏统一化，不适合承载 Snapseed/醒图“先选调节项，再用统一滑动线精调”的交互模式。  
本次目标是先完成可交付的“基础”调节面板，并把分组架构重构为“基础/HSL/曲线”，其中 HSL 与曲线先完成分组入口与面板隔离框架，复杂编辑逻辑后续增量实现。

约束：
- 保持现有图片编辑器主流程与返回链路稳定，不影响裁剪/旋转/滤镜等已上线能力。
- 继续使用项目语义 token（`AppColors`/`AppSpacing`/`UITextConstants`）。
- 工具面板交互采用“底部 X/勾 + 中间控件区”统一范式，但分组间面板内容完全隔离。

## Goals / Non-Goals

**Goals:**
- 将专业修图二级分组切换为“基础/HSL/曲线”，并与工具列表/面板状态联动。
- 交付“基础”面板完整交互：
  - 横向滚动调节项（图标+名称+`-100~100`值，垂直布局）
  - 底部中间统一调节线，默认 `0`
  - 长按调节线周边并左右拖动，更新当前选中调节项值
  - 调节线位置、当前选中项、当前值实时同步
- 为 HSL 与曲线预留独立面板容器和状态入口，避免后续实现时重构成本过高。

**Non-Goals:**
- 本次不实现 HSL 细粒度通道编辑（色相/饱和/明度分色轮等）。
- 本次不实现曲线控制点、通道曲线与直方图等复杂交互。
- 本次不新增图像处理算法，仅实现面板和参数状态流转（应用算法后续接入）。

## Decisions

### 1) 分组与面板状态采用“分组驱动 + 面板类型分发”

- 决策：专业修图状态由 `group = base|hsl|curve` 驱动，渲染层按 group 分发到独立 panel widget（`BaseAdjustPanel`、`HslPanelPlaceholder`、`CurvePanelPlaceholder`）。
- 原因：避免在单个大面板中通过 `if/switch` 累积复杂分支，降低功能耦合，便于后续并行开发 HSL/曲线。
- 备选方案：继续沿用统一面板模板，仅替换内容区。
  - 不选原因：后续 HSL/曲线差异化大，统一模板会导致状态耦合和回归风险。

### 2) 基础调节项采用“配置驱动”

- 决策：基础项定义为配置数组（id、icon、label、defaultValue、min/max），首批固定七项：亮度、对比度、饱和度、氛围、高光、阴影、暖色调。
- 原因：后续扩展（如去朦胧、黑色阶）只需增配置，不改核心交互逻辑。
- 备选方案：每个调节项单独写状态字段。
  - 不选原因：字段数量膨胀，难以统一渲染与复用。

### 3) 调节线采用“当前项单值模型”

- 决策：底部中间只绑定“当前选中调节项”的值，值域 `-100..100`，中心点 `0`；拖动时仅更新当前项。
- 原因：符合 Snapseed“先选参数再调值”的认知模型，降低误操作。
- 备选方案：每个调节项独立 slider。
  - 不选原因：面板信息密度过高，违背目标交互样式。

### 4) 手势策略：横向列表滑动与调节线拖动分区

- 决策：调节项列表和调节线分别拥有独立手势区域；调节线支持长按后水平拖动，避免与列表横向滚动冲突。
- 原因：提升可控性，减少误触。
- 备选方案：调节线直接水平拖动（无需长按）。
  - 不选原因：容易与页面横向滚动/系统手势冲突。

### 5) HSL/曲线先占位但可进入

- 决策：HSL/曲线分组先提供可进入的独立面板占位视图（含取消/保存），并写入“未生效参数”保护逻辑（无改动不写记录）。
- 原因：先打通完整导航和结构，后续只替换面板实现。
- 备选方案：暂时隐藏 HSL/曲线 tab。
  - 不选原因：与本次分组重构目标不一致，也不利于产品验收分组结构。

## Risks / Trade-offs

- [Risk] 调节线拖动手势与系统边缘返回手势冲突  
  → Mitigation：限制拖动激活区在底部中间安全区域，使用长按激活，再进入水平拖动。

- [Risk] 当前只实现基础面板，用户可能误认为 HSL/曲线不可用  
  → Mitigation：HSL/曲线占位面板显示“即将支持”文案与禁用态提示，不允许产生假记录记录。

- [Risk] 参数状态与记录步骤同步不一致  
  → Mitigation：仅在点击勾确认时落盘步骤；取消/X始终回滚本次 panel 会话内改动。

- [Risk] 面板拆分后状态分散，后续维护成本上升  
  → Mitigation：使用统一的 pro-tool state model（group、selectedItem、values map），面板只负责视图与输入。

## Migration Plan

1. 新增专业修图分组枚举与映射常量：`base/hsl/curve`。  
2. 落地基础调节配置表与状态容器（value map + selected item）。  
3. 将现有专业修图面板改造为按分组分发渲染。  
4. 实现基础面板中部横向调节项 + 底部统一调节线交互。  
5. 增加 HSL/曲线占位面板（分组切换可用，保存无参数变更时不写记录）。  
6. 回归测试：分组切换、调节线同步、取消/保存、记录步骤写入、与其他工具切换互不污染。  

回滚策略：保留旧分组常量与旧面板实现一个版本窗口，出现重大回归时切换开关回旧逻辑。

## Open Questions

- “氛围”对应的内部参数名是否统一为 `ambiance`（建议）？  
- 暖色调是单一温度滑杆（`warmth`）还是拆分色温/色调双参数（后续可能影响数据结构）？  
- HSL 与曲线占位阶段是否允许显示预览变化（当前建议不允许）？
