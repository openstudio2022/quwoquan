# 可观测性与治理

## 1. 目标

runtime 横切能力必须具备统一的观测与治理基线，以支撑：

- 跨服务问题定位
- 异常补偿与回滚
- 厂商故障降级
- 大流量场景的稳定性控制

## 2. 关键观测维度

### 2.1 media

- 上传初始化成功率
- 上传完成成功率
- CDN URL 生成失败率
- OSS/COS 请求时延
- 群头像重算成功率与耗时

### 2.2 sync

- patch 生成量
- 实时通知成功率
- gap fill 命中率
- 重复 patch 比例
- cursor 落后深度

### 2.3 realtime

- 在线连接数
- 连接建立成功率
- topic 订阅数
- ping/pong 超时率
- Push 降级命中率

## 3. 日志标准

必须统一记录：

- `traceId`
- `userId`
- `service`
- `operation`
- `status`
- `durationMs`
- `errorCode`

## 4. 指标分层

- `SLA`：上传成功率、同步时延、在线通知成功率
- `SLO`：关键能力的内部目标
- `KPI`：群头像显示成功率、首屏头像命中率等业务结果指标

## 5. 治理策略

### 5.1 限流

- 上传初始化限流
- 群头像重算任务限流
- patch 拉取频率限流

### 5.2 熔断

- OSS/COS SDK 异常持续升高时熔断
- CDN 签名异常时降级为默认图标或跳过更新

### 5.3 重试

- 群头像生成失败允许异步重试
- patch 投递失败允许补偿
- 私有资源签名失败可短时重试

## 6. 告警建议

- 群头像重算失败率超过阈值
- 用户同步流积压
- 在线连接异常下降
- object storage 回源延迟异常升高

## 7. 回滚原则

任何 runtime 新增能力都必须定义：

- 回滚开关
- 降级路径
- 厂商故障 fallback
- 兼容旧字段/旧协议的窗口期
