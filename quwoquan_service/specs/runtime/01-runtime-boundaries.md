# Runtime 边界

## 1. 边界目标

本章用于明确：

- runtime 与业务服务的职责边界
- 业务对象的归属
- 公共能力的沉淀标准

## 2. 业务服务边界

### 2.1 user-service

负责：

- 用户资料主数据
- 用户头像主资源引用
- `avatarVersion`
- 用户头像更新事件发布
- 用户通知偏好、push token

不负责：

- 群头像生成与存储引用
- 会话列表投影
- 消息同步流

### 2.2 chat-service

负责：

- 会话与群组
- 成员关系与排序
- 群头像业务归属
- 消息发送、撤回、已读
- 与群相关的同步 patch 生产

不负责：

- 用户头像原图上传策略
- 底层对象存储细节
- CDN 厂商适配

### 2.3 content-service

负责：

- 帖子图片与视频的业务语义
- 内容媒体与内容对象的绑定
- 转码、封面、发布状态对业务的影响

不负责：

- 会话消息同步
- 群头像

## 3. runtime 边界

runtime 负责：

- 统一资产引用协议
- 统一上传会话模型
- 对象存储驱动抽象
- CDN URL 生成与签名
- 用户同步流 envelope
- 实时通知 envelope
- 通用治理与观测

runtime 不负责：

- 用户、群组、内容对象的主业务规则
- 权限判断的最终业务逻辑
- 业务侧的排序、推荐、审核与可见性决策

## 4. 进入 runtime 的准入标准

一个能力只有同时满足以下至少两项时，才应沉淀到 runtime：

1. 被两个及以上业务服务复用
2. 需要统一协议或 URL 规范
3. 未来存在 OSS/COS/其它供应商切换需求
4. 需要统一治理与观测
5. 错误地分散实现会造成跨端不一致

## 5. 不应进入 runtime 的典型示例

- “群头像取前 9 个成员”中的“前 9”规则本身
- “消息撤回 2 分钟”这样的产品规则
- “帖子发布失败是否自动重试”这样的业务流程规则

## 6. 边界判定模板

当新增能力时，按以下问题判断：

1. 它是业务归属，还是底层运行机制？
2. 它是否会被多个服务复用？
3. 它是否需要统一 URL、对象引用或同步协议？
4. 它是否会影响客户端一致性？
5. 若未来切换云厂商，业务层是否需要无感？

若第 2~5 项答案为“是”，优先考虑纳入 runtime。
