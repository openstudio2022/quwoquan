## Context

- 现有图片编辑器与 Snapseed/醒图 级能力目标差距大，需按产品需求重新实现。
- 创作流（create-flow）约定图片 Tab 使用由 image-editor 能力提供的编辑器；需保持入口与数据（图片列表、输出）兼容。
- 项目约束：Flutter、设计系统（AppColors、AppSpacing、UITextConstants）、包引用、无硬编码。

## Goals / Non-Goals

**Goals:**

- 实现严格三段式布局（顶栏 / 中部图片 / 底栏），顶栏含返回、序号、蓝色完成及完成前 Snapseed 式历史（步骤列表、删除、重做）。
- 多图时中部支持左右滑动切换，顶栏序号与当前页联动。
- 底栏工具：旋转、裁剪、滤镜、专业修图、相框、文字、涂鸦、马赛克；专业修图内含曲线、白平衡、局部、修复、美丽光晕、色调对比度、HSL 调色。
- 操作面板统一结构：顶部分类、中间模板/内容、底栏取消/保存及可选比例/强度；滤镜有分类+模板+强度；旋转有旋转仪表与图片上网格；曲线为图片下方蒙皮式调节。

**Non-Goals:**

- 不实现桌面端或 Web 端专用交互；不替代系统相册或第三方修图 App 的全部能力；本阶段不规定具体滤镜/曲线算法实现方（可先占位或对接既有库）。

## Decisions

1. **步骤栈与历史 UI**  
   - 决策：在内存中维护线性步骤栈（每步为工具类型 + 参数快照），顶栏或底栏提供入口打开历史列表；支持删除某步（并重算其后）与重做某步（重新进入该步编辑并保存后替换）。  
   - 理由：与 Snapseed 行为一致，完成前可追溯、可撤销单步、可重做。  
   - 备选：仅支持全局撤销/重做（不按步删除/重做）——不满足需求。

2. **多图与序号**  
   - 决策：中部使用可横向滑动的 PageView（或等价），一页一图；滑动后更新顶栏序号（当前索引+1 / 总数）。  
   - 理由：与「跟随内容区左右滑动浏览所有图片」一致。

3. **完成按钮颜色**  
   - 决策：顶栏「完成」按钮使用 AppColors.primaryColor，禁止红色或硬编码色值。  
   - 理由：需求与设计语义明确。

4. **旋转交互**  
   - 决策：选中旋转时，图片下方展示旋转仪表（约 ±45° 或更大范围的可拖/刻度盘），调节时在图片上叠加对齐网格（如 3×3 或 6×6）；底部保留取消/保存及可选「向左 90° / 向右 90° / 水平翻转 / 垂直翻转」等快捷操作。  
   - 理由：对齐参考图与「便于对齐主体与角度」的需求。

5. **曲线调节 UI**  
   - 决策：选中曲线（或专业修图内曲线子工具）时，在图片下方以蒙皮/浮层形式展示曲线图与通道选择（RGB 或 R/G/B 等），不占用底栏工具位；底部仍为取消/保存。  
   - 理由：对齐「曲线调节相关设置在图片下方设置蒙皮」的表述。

6. **专业修图子工具**  
   - 决策：专业修图作为底栏单一入口，进入后展示子工具列表（曲线、白平衡、局部、修复、美丽光晕、色调对比度、HSL 调色等）；每种子工具可拥有自己的面板或蒙皮布局，但统一取消/保存与步骤入栈。  
   - 理由：与 Snapseed/醒图 结构一致，需求明确包含 HSL 调色。

7. **滤镜面板**  
   - 决策：选中滤镜时，面板顶部为分类 Tab、中间为滤镜模板列表（带预览）、底部为取消/保存及强度滑块，与 proposal 一致。  
   - 理由：与参考图及「顶部分类、中间模板、底部取消/保存与比例」一致。

## Risks / Trade-offs

- **[性能]** 多图 + 多步历史 + 实时预览（滤镜/曲线/旋转）可能带来内存与 GPU 压力。  
  → 缓解：限制历史步数或图片分辨率用于预览；导出时再按全分辨率重放步骤。

- **[实现复杂度]** 曲线/局部/HSL 等专业工具涉及像素级或区域级处理。  
  → 缓解：优先实现交互与布局，算法可对接 image 等库或先占位，后续迭代。

- **[与 create-flow 集成]** 创作流传入图片列表与接收导出结果的接口需稳定。  
  → 缓解：在 design 与 tasks 中明确入参（如 List<XFile> 或 AssetEntity）、出参（如临时文件或内存字节）及完成/取消回调。

## Migration Plan

- 用新实现的编辑器替换 CreatePage 图片 Tab 当前使用的旧编辑器入口；保留同一路由或入口点，仅替换内部 Widget/页面。
- 若存在旧版 ImageEditor 组件，可保留为废弃代码路径一段时间，通过 feature flag 或直接切换至新实现；无长期双轨运行需求时可删除旧实现。
- 回滚：恢复旧编辑器入口引用即可回滚。

## Open Questions

- 历史步骤上限数量是否在产品侧有要求（若无则可由实现自定，如 30 步）。
- 专业修图中「局部」「修复」等是否首版就要可用的算法，还是先做 UI 与步骤栈占位。
