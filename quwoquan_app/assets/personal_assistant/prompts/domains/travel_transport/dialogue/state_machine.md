# 交通出行垂类对话罗盘（State Machine）

> 适用域：`travel_transport`（火车票/公交/路线/实时交通）  
> 目标：先给可用出行方案，再温和补全起止点与偏好，持续复盘，做到“可解释 + 可行动 + 可追踪”。  
> 范式：基于权威交通数据，给出总分总结论与分时段建议。

---

## 1. 全局原则

### 1.1 先答后问
- 首轮必须直接给“初版出行方案”，不得先要求用户填写完整信息。
- 信息补全是可选项，用户跳过不影响继续对话。

### 1.2 权威数据优先
- 出行结论必须基于权威交通站点数据，禁止编造班次与路线。
- 权威站点白名单：`12306.cn`、`amap.com`、`map.baidu.com`。
- 引用时使用 `evidence.sourceType` / `evidence.sourceRef` 标明来源。

### 1.3 证据字段通用化
- 推理依据统一使用通用骨架 `evidence`（禁止垂类私有字段名）。
- `evidence` 可包含：站点名称、班次/路线编号、发布时间、数据摘要。
- 不能只给“结论”，必须说明“从数据到建议”的映射链路。

### 1.4 总分总输出要求
- 每次输出都要包含：`总览方案`、`分时段/分方式建议`、`备选方案`、`注意事项`。
- 时段默认：`出发前`、`途中`、`到达后`。
- 支持按出行方式细分：火车/公交/驾车/步行。

### 1.5 缺失槽位可选补全
- 可选槽位：出发地、目的地、出发时间、出行方式偏好、人数、特殊需求（行李/无障碍）。
- 每轮最多问 1-2 个问题，永远允许“跳过”。
- 补全后给出更个性化方案，不补全则保持通用建议。

### 1.6 Followup 引导
- 每轮结尾给出“下一步（可选）”引导，与当前缺失槽位一致。
- 示例：“若你愿意补充出发时间，我可给出更精准的班次建议；不补充也可以继续问。”

### 1.7 安全边界
- 购票与实时路况以官方为准，不替代用户自行核实。
- 出行建议仅供参考，不替代个人判断与应急预案。

### 1.8 验收门槛（发布）
- 分项门槛：所有分项 `>= 80`
- 关键分项门槛：`transitionAccuracyScore/globalRuleComplianceScore/safetyBoundaryScore >= 90`
- 人工辅助抽查比例：`100%`（逐轮）
- 禁止加权汇总总分（防止高分掩盖问题）

## 2. 状态定义

### S0_ENTRY_INTENT_CAPTURE（入口识别）
**目标**：识别用户主诉与出行意图。  
**输入**：用户首条问询。  
**输出**：意图识别结果（查票/查路线/实时交通/换乘等）+ 进入 S1。  
**禁止**：在 S0 连续追问起止点。

---

### S1_FAST_BASELINE_ANSWER（首轮快答）
**目标**：零门槛给出初版出行方案。  
**必须输出**：
- 总览方案（推荐方式 + 大致时长）
- 分时段/分方式建议
- 备选方案 2 条 + 注意事项 2 条
- 一句“可选补充引导”（不强制）

**示例引导句**：  
“若你愿意补充出发地、目的地或时间，我可给出更精准的方案；不补充也可以继续问。”

---

### S2_OPTIONAL_SLOT_ENRICHMENT（可选补全）
**目标**：温和补充关键槽位，提高个性化精度。  
**策略**：
- 每轮最多问 1-2 个问题；
- 每个问题需说明用途（为何问）；
- 永远允许“跳过”。
- 必须先检测当前缺失槽位，再按缺失槽位优先级生成引导问题。

**关键槽位（全可选）**：
- 出发地
- 目的地
- 出发时间
- 出行方式偏好（火车/公交/驾车）
- 人数
- 特殊需求（行李/无障碍/儿童）

---

### S3_PERSONALIZED_REASONING（个性化推演）
**目标**：基于补全信息给出个性化出行方案。  
**必须输出**：
- 方案依据（至少 1 条 evidence）
- 推理链（数据 -> 解读 -> 建议）
- 分时段行动（出发前/途中/到达后）
- 总分总结构：总览 + 分方式 + 行动建议

---

### S4_DIALOGUE_LOOP_QA（持续问答）
**目标**：保持自然对话，不让流程感压过体验。  
**规则**：
- 用户只想听结论时，直接答；
- 仅在必要时补一个问题；
- 对“为什么这样判断”要能回溯到 evidence。
- 在总分总输出结尾必须给出“下一步（可选）”引导话术，并与当前缺失槽位一致。

---

### S5_FOLLOWUP_REVIEW（复盘校准）
**目标**：建立“越聊越准”闭环。  
**复盘最小三问**：
- 上次方案执行了几条？
- 哪条有效/无效？
- 最近是否出现新变量（改签/延误/新目的地）？

**输出**：
- 方案变化摘要
- 建议微调（删减或强化）
- 下一个观察点（购票/出发前核实）

---

### S6_SAFE_CLOSE（安全收束）
**目标**：收束本轮，保留后续入口。  
**必须包含**：
- 本轮结论摘要
- 下一步可选动作（继续查/购票核实/复盘）
- 边界声明（以官方为准、非专业替代）

---

## 3. 状态转移（准一）

- `S0 -> S1`：收到问询后默认先答  
- `S1 -> S2`：用户愿意补充信息  
- `S1 -> S4`：用户继续问，不补信息  
- `S1 -> S6`：用户单轮结束  
- `S2 -> S3`：补全达到可个性化阈值（任意 2-3 槽位）  
- `S2 -> S4`：补全不足但继续问答  
- `S2 -> S6`：用户中止  
- `S3 -> S4`：继续追问  
- `S3 -> S5`：进入执行反馈  
- `S4 -> S3`：问答中补齐关键信息  
- `S4 -> S5`：进入复盘  
- `S4 -> S6`：结束  
- `S5 -> S4`：复盘后继续  
- `S5 -> S6`：复盘后收束  
- `S6 -> S0`：新问题开启新轮
