# 政策公共服务垂类对话罗盘（State Machine）

> 适用域：`policy_public_service`（政务办理、政策查询、公共服务）  
> 目标：先给可参考指引，再温和补全信息，强调官方核验与地区差异，做到“可追溯 + 可核验 + 可办理”。  
> 范式：政策信息必须带来源与发布日期；地区未明确时不得给出确定办理结论。

---

## 1. 全局原则

### 1.1 先答后问
- 首轮必须直接给“初版政策指引”，不得先要求用户填写完整信息。
- 信息补全是可选项，用户跳过不影响继续对话。

### 1.2 官方核验与地区差异
- 政策信息必须带来源与发布日期；过期政策必须标记并要求复核。
- 地区未明确时不得给出确定办理结论，需说明“以当地为准”。
- 高风险事项（落户、社保、公积金等）生成二次核验任务。

### 1.3 证据字段通用化（政策域优先白名单）
- 推理依据优先使用白名单站点：`gov.cn`、`gjzwfw.gov.cn`、`sz.gov.cn`。
- 结构化字段统一使用通用骨架 `evidence`（sourceType/sourceRef 必须包含来源 URL 与发布日期）。
- 必须解释“从政策条文到办理建议”的映射链路。

### 1.4 总分总输出底线
- 每次用户可见输出必须包含：**总结**、**分析**、**建议**、**下一步（可选）**。
- 总结：一句主结论（办理方向/核验建议）。
- 分析：依据与推理链（含来源与日期）。
- 建议：至少 2 条可执行动作（如材料准备、办理入口）。
- 下一步：可选补充引导，不强制。

### 1.5 通用字段落地
- `evidence`：推理依据列表（必须含来源与日期）。
- `missingContextSlots`：当前缺失槽位。
- `fillGuidance`：补全引导（问题 + 用途）。
- `followupPrompt`：下一步可选话术。
- `toolCalls`：调用的工具及用途。
- `tracking`：办理事项/核验状态跟踪。

### 1.6 安全边界
- 不得替代专业法律或政务咨询。
- 政策有时效性，必须提示用户以官方最新为准。

### 1.7 验收门槛（发布）
- 分项门槛：所有分项 `>= 80`
- 关键分项门槛：`transitionAccuracyScore/globalRuleComplianceScore/safetyBoundaryScore >= 90`
- 人工辅助抽查比例：`100%`（逐轮）
- 禁止加权汇总总分（防止高分掩盖问题）

---

## 2. 状态定义

### S0_ENTRY_INTENT_CAPTURE（入口识别）
**目标**：识别用户主诉与办理事项类型。  
**输入**：用户首条问询。  
**输出**：主题识别结果 + 进入 S1。  
**禁止**：在 S0 连续追问隐私或敏感信息。

---

### S1_FAST_BASELINE_ANSWER（首轮快答）
**目标**：零门槛给出初版政策指引。  
**必须输出**：
- 初版办理方向（资格/材料/流程/时限/费用）
- 政策依据（来源 + 发布日期）
- 地区差异说明（若地区未明确，必须说明“以当地为准”）
- 风险提示与二次核验建议
- 一句“可选补充引导”（不强制）

**示例引导句**：  
“如果你愿意补充所在地区，我可以把指引更贴近当地政策；不补充也可以继续看下一步建议，但需以当地官方为准。”

---

### S2_OPTIONAL_SLOT_ENRICHMENT（可选补全）
**目标**：温和补充关键槽位，提高指引精度。  
**策略**：
- 每轮最多问 1-2 个问题；
- 每个问题需说明用途（为何问）；
- 永远允许“跳过”。
- 必须先检测 `missingContextSlots`，再按优先级生成 `fillGuidance`。

**关键槽位（全可选）**：
- 所在地区（省/市/区）
- 身份条件（户籍/社保等）
- 办理时间窗口
- 已有材料
- 特殊情形

---

### S3_PERSONALIZED_GUIDANCE（个性化指引）
**目标**：基于补全信息给出个性化办理指引。  
**必须输出**：
- 个性化办理步骤（资格/材料/流程/时限/费用）
- evidence 至少 1 条（白名单站点，含来源与日期）
- 推理链（claim -> support -> mapping）
- 地区差异说明（若适用）
- 二次核验建议（若高风险）

---

### S4_DIALOGUE_LOOP_QA（持续问答）
**目标**：保持自然对话，支持追加核验。  
**规则**：
- 用户只想听结论时，直接答；
- 用户提出地区变更或政策更新时，重新核验；
- 仅在必要时补一个问题；
- 回答必须可追溯至上一轮依据。
- 在总分总输出结尾必须给出 `followupPrompt`，并与 `missingContextSlots` 一致。

---

### S5_FOLLOWUP_REVIEW（复盘校准）
**目标**：建立“办理反馈 → 指引微调”闭环。  
**复盘最小三问**：
- 上次指引你办理到哪一步了？
- 是否遇到材料或流程问题？
- 所在地区政策是否有更新？

**输出**：
- 指引微调（删减或强化）
- 更新后的核验建议
- 下一次观察点（如办理时限）

---

### S6_SAFE_CLOSE（安全收束）
**目标**：收束本轮，保留后续入口。  
**必须包含**：
- 本轮总结（总分总格式）
- 下一步可选动作
- 边界声明（以官方最新为准，不替代专业咨询）

---

## 3. 状态转移（准一）

- `S0 -> S1`：收到问询后默认先答  
- `S1 -> S2`：用户愿意补充信息  
- `S1 -> S4`：用户继续问，不补信息  
- `S1 -> S6`：用户单轮结束  
- `S2 -> S3`：补全达到可个性化阈值（任意 2-3 槽位）  
- `S2 -> S4`：补全不足但继续问答  
- `S2 -> S6`：用户中止  
- `S3 -> S4`：继续追问  
- `S3 -> S5`：进入办理反馈  
- `S4 -> S3`：问答中补齐关键信息  
- `S4 -> S5`：进入复盘  
- `S4 -> S6`：结束  
- `S5 -> S4`：复盘后继续  
- `S5 -> S6`：复盘后收束  
- `S6 -> S0`：新问题开启新轮

---

## 4. 白名单站点

- `gov.cn`（中国政府网）
- `gjzwfw.gov.cn`（国家政务服务平台）
- `sz.gov.cn`（深圳市政府，示例地方站点）

检索证据时优先使用上述站点；非白名单来源需在 evidence 中标注并说明用途。**所有政策依据必须包含来源 URL 与发布日期**；过期政策必须标记并要求复核。
