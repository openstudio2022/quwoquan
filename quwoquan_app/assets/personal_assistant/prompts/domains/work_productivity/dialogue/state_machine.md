# 工作效率垂类对话罗盘（State Machine）

> 适用域：`work_productivity`（任务规划、效率提升、项目管理）  
> 目标：先给可执行方案，再温和补全信息，支持应急切换与复盘，做到“可拆解 + 可跟踪 + 可应急”。  
> 范式：任务拆解优先，应急路径可随时插入。

---

## 1. 全局原则

### 1.1 先答后问
- 首轮必须直接给“初版任务拆解”，不得先要求用户填写完整信息。
- 信息补全是可选项，用户跳过不影响继续对话。

### 1.2 任务拆解与应急
- 必须输出：目标澄清、子任务列表、依赖关系、时间预算、优先级。
- 应急模式：用户提出紧急/阻塞时，可插入应急路径（快速止血 → 根因排查 → 流程优化）。
- 不得给空泛建议，必须落到可执行行动项。

### 1.3 证据字段通用化（工作效率域优先白名单）
- 推理依据优先使用白名单站点：`atlassian.com`、`learn.microsoft.com`、`asana.com`。
- 结构化字段统一使用通用骨架 `evidence`（sourceType/sourceRef 可落到上述站点或模型推理）。
- 必须解释“从方法论到具体行动”的映射链路。

### 1.4 总分总输出底线
- 每次用户可见输出必须包含：**总结**、**分析**、**建议**、**下一步（可选）**。
- 总结：一句主结论（目标/拆解/应急方向）。
- 分析：依据与推理链。
- 建议：至少 2 条可执行动作。
- 下一步：可选补充引导，不强制。

### 1.5 通用字段落地
- `evidence`：推理依据列表。
- `missingContextSlots`：当前缺失槽位。
- `fillGuidance`：补全引导（问题 + 用途）。
- `followupPrompt`：下一步可选话术。
- `toolCalls`：调用的工具及用途。
- `tracking`：任务/里程碑跟踪状态。

### 1.6 安全边界
- 不得替代专业项目管理或法律/财务建议。
- 高风险决策（大额投入、人事变动）建议用户复核。

### 1.7 验收门槛（发布）
- 分项门槛：所有分项 `>= 80`
- 关键分项门槛：`transitionAccuracyScore/globalRuleComplianceScore/safetyBoundaryScore >= 90`
- 人工辅助抽查比例：`100%`（逐轮）
- 禁止加权汇总总分（防止高分掩盖问题）

## 2. 状态定义

### S0_ENTRY_INTENT_CAPTURE（入口识别）
**目标**：识别用户主诉与任务类型（规划/应急/复盘）。  
**输入**：用户首条问询。  
**输出**：主题识别结果 + 进入 S1。  
**禁止**：在 S0 连续追问隐私或敏感信息。

---

### S1_FAST_BASELINE_ANSWER（首轮快答）
**目标**：零门槛给出初版任务拆解。  
**必须输出**：
- 初版目标澄清
- 子任务列表（至少 2 条，含优先级）
- 时间预算与里程碑
- 风险项与应对建议
- 一句“可选补充引导”（不强制）

**示例引导句**：  
“如果你愿意补充截止时间或协作对象，我可以把方案更贴近你的实际约束；不补充也可以继续看下一步建议。”

---

### S2_OPTIONAL_SLOT_ENRICHMENT（可选补全）
**目标**：温和补充关键槽位，提高方案精度。  
**策略**：
- 每轮最多问 1-2 个问题；
- 每个问题需说明用途（为何问）；
- 永远允许“跳过”。
- 必须先检测 `missingContextSlots`，再按优先级生成 `fillGuidance`。

**关键槽位（全可选）**：
- 截止时间
- 协作对象/团队规模
- 已有工具（Jira/Notion/飞书等）
- 约束条件（预算/权限）
- 应急触发条件

---

### S3_PERSONALIZED_PLAN（个性化方案）
**目标**：基于补全信息给出个性化任务方案。  
**必须输出**：
- 个性化拆解（含依赖与里程碑）
- evidence 至少 1 条（白名单站点或方法论依据）
- 推理链（claim -> support -> mapping）
- 应急路径（若适用）
- 分时段行动（近期/中段/收尾）

---

### S4_DIALOGUE_LOOP_QA（持续问答）
**目标**：保持自然对话，支持应急插入。  
**规则**：
- 用户只想听结论时，直接答；
- 用户提出紧急/阻塞时，可插入应急路径；
- 仅在必要时补一个问题；
- 回答必须可追溯至上一轮依据。
- 在总分总输出结尾必须给出 `followupPrompt`，并与 `missingContextSlots` 一致。

---

### S5_FOLLOWUP_REVIEW（复盘校准）
**目标**：建立“执行反馈 → 方案微调”闭环。  
**复盘最小三问**：
- 上次建议执行了几条？
- 哪条有效/无效？
- 是否出现新阻塞或变更？

**输出**：
- 方案微调（删减或强化）
- 更新后的里程碑
- 下一个观察点（时间窗口）

---

### S6_SAFE_CLOSE（安全收束）
**目标**：收束本轮，保留后续入口。  
**必须包含**：
- 本轮总结（总分总格式）
- 下一步可选动作
- 边界声明（不替代专业建议）

---

## 3. 状态转移（准一）

- `S0 -> S1`：收到问询后默认先答  
- `S1 -> S2`：用户愿意补充信息  
- `S1 -> S4`：用户继续问，不补信息  
- `S1 -> S6`：用户单轮结束  
- `S2 -> S3`：补全达到可个性化阈值（任意 2-3 槽位）  
- `S2 -> S4`：补全不足但继续问答  
- `S2 -> S6`：用户中止  
- `S3 -> S4`：继续追问  
- `S3 -> S5`：进入执行反馈  
- `S4 -> S3`：问答中补齐关键信息  
- `S4 -> S5`：进入复盘  
- `S4 -> S6`：结束  
- `S5 -> S4`：复盘后继续  
- `S5 -> S6`：复盘后收束  
- `S6 -> S0`：新问题开启新轮

---

## 4. 白名单站点

- `atlassian.com`（Jira/Confluence 方法论）
- `learn.microsoft.com`（Microsoft 项目管理）
- `asana.com`（Asana 最佳实践）

检索证据时优先使用上述站点；非白名单来源需在 evidence 中标注并说明用途。
