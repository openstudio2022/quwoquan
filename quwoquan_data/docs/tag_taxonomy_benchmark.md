### 设计结论

酒店住宿不能把“豪华酒店、商务酒店、温泉酒店、精品酒店、胶囊旅馆”全部放在同一兄弟层，因为它们分别属于档次、功能、设施卖点与空间形态。最终设计采用：

- `Entity/地点/住宿/住宿业态`：酒店、民宿、青年旅舍、客栈、度假村、公寓式酒店、服务式公寓、胶囊旅馆、旅馆宾馆、农家乐、营地住宿。
- `Entity/地点/住宿/档次等级`：豪华型、高端型、舒适型、经济型、一星级至白金五星级。
- `Entity/地点/住宿/功能定位`：商务住宿、出差住宿、会议会展住宿、度假住宿、温泉住宿、亲子友好住宿等。
- `Entity/地点/住宿/设施服务`：早餐、停车、接送、泳池、健身房、会议室、温泉汤池、SPA、自助洗衣、充电桩等。
- `Entity/地点/住宿/房型空间`：大床房、双床房、家庭房、亲子房、套房、景观房、山景房、私汤房等。
- `Entity/地点/住宿/房源形态`：整套房源、独立房间、合住房间、独栋别墅、树屋、帐篷营地、藏式民居、庭院客栈等。

配套 Topic / Format：

- `Topic/旅行/住宿`：住宿攻略、酒店体验、民宿体验、出差住宿、商旅住宿、川西住宿、高原住宿、温泉住宿、亲子住宿、住宿避雷、住宿比价。
- `Format/内容角度`：酒店探店、度假村探店、住宿测评、酒店横评、住宿攻略、住宿避雷。

## 七、关键设计决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 根分组数量 | 4（Topic/Audience/Format/Entity） | 比 20-100 一级更利于工程消费，与业界广告/推荐体系对齐 |
| Entity 是否实例化 | 否，只承载类型骨架 | 实例数量无上限，强行列举会变成半实时更新的"黄页"，不适合标签树；实例进 entities/ |
| Format 子维度数 | 6 个互斥 | 消除"故事/评论"等重名，TikTok 的挑战等手法分入表现手法而非内容角度 |
| 内容角度"文化"删除 | 是 | "文化"是主题（Topic），不是内容视角；历史解读/民俗探秘 = 历史文化主题 + 叙事/科普角度组合 |
| 推荐打分模型 | 本期不实现 | 只提供 inverted_index + cooccur 简单召回，留接口签名 stub 等后续推荐工程接入 |
| appliesTo 字段 | 完全去除 | 默认所有对象都可引用任意标签，复杂约束放 tag_policy.yaml，避免通用 schema 变权限表 |

### 维度边界与正交化决策矩阵

| 易混淆对 | 归属维度 | 区分规则 | 示例 |
|----------|----------|----------|------|
| 温泉度假 vs 温泉泡汤 | 旅行主题 vs 玩法 | 主题=旅行整体定位（1-2个），玩法=具体子活动（可多个） | ❌旅行主题/温泉度假 → ✓玩法/温泉泡汤 |
| 自驾游 vs 自驾 | — vs 出行方式 | "自驾游"混合了出行方式与行程形态，拆为出行方式/自驾 | ❌旅行主题/自驾游 → ✓出行方式/自驾 |
| 亲子游 vs 亲子家庭 | — vs Audience/用户 | "亲子游"混合了受众与主题，受众归 Audience，主题用旅行主题组合 | ❌旅行主题/亲子游 → ✓Audience/用户/婚姻家庭/亲子家庭 |
| 探亲访友 vs 旅行筹备 | — | 非旅行消费体验，不纳入旅行标签体系 | ❌旅行主题/探亲访友 |
| 住宿主题 vs Entity/地点/住宿 | Topic vs Entity | Topic=主观偏好（"我想住什么风格"），Entity=客观实体类型骨架（"这是什么住宿设施"） | 住宿主题/精品民宿 vs Entity/地点/住宿/民宿 |
| 摄影旅拍 vs 摄影摄像 vs 摄影艺术 | 玩法 vs 数码科技 vs 艺术创作 | 玩法=旅行中拍照体验, 数码=器材技巧, 艺术=美学属性 | Topic/旅行/玩法/摄影旅拍 |
| 攻略(Format) vs 旅行筹备(Topic) | Format vs Topic | Format=内容呈现角度（怎么讲），Topic=话题主题（讲什么） | Format/内容角度/攻略 vs Topic/旅行/旅行筹备/行前规划 |
| 价格区间/评分 | 结构化字段 | 不纳入标签体系，作为 entity 结构化属性 | entity._entity.json.priceRange / rating |

### 不纳入标签的决策

| 候选维度 | 决策 | 理由 |
|----------|------|------|
| 价格区间 | 不纳入 | 携程/Booking 的价格筛选为结构化字段，标签化会导致维护成本过高 |
| 用户评分 | 不纳入 | 评分为动态数值，属于 entity 属性而非标签分类 |
| 天气/气候 | 不纳入 | 动态变化的实时信息，通过 API 获取而非静态标签 |
| 具体品牌酒店 | 不纳入 | 如"万豪""希尔顿"属于 entities/ 中的实例，不进标签骨架 |
| 目的地国家/城市 | 由行政区管理 | Topic/地理/行政区/ 已覆盖，不在 Topic/旅行/ 中重复 |
