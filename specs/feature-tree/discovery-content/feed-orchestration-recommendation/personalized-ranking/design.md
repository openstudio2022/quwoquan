# 设计：personalized-ranking 端云协同分页

## 背景
当前发现流需要同时满足两类诉求：
- 推荐排序实时响应用户反馈。
- cursor 分页跨页不重复，回滚历史不抖动。

过去“有 cursor 就跳过重排”的做法能止住重复，但会导致第一页和后续页排序语义不一致。

## 设计目标
- 同时满足“推荐排序 + cursor 分页”。
- 端云职责清晰：历史稳定性在端侧，未来动态性在云侧。
- 维持低耦合：端侧不解析 cursor token。

## 方案

### 1) 历史在端侧，未来在云侧
- 端侧维护 seen window queue（已看队列）。
- 云侧只保证 future window（cursor 之后）产出。
- 用户回滚读取端侧队列，不请求历史回放。

### 2) cursor 协议（opaque token）
- 编码：base64(JSON)。
- 最小字段：
  - `v`: token 版本
  - `sid`: 会话标识
  - `off`: future offset（下次起点）
  - `exp`: 过期时间戳
- 端侧仅透传，不依赖内部结构。

### 3) 反馈分级
- 强反馈：`dislike/report/block`
  - 立即应用于未来窗口过滤。
- 弱反馈：`click/like/favorite/dwell`
  - 进入未来窗口重排权重计算。
  - 不回写已看历史。

## 数据流
1. 首次请求：无 cursor，云侧生成排序结果并返回第一页 + token。
2. 翻页请求：携带 token，云侧解码 offset，在最新反馈下重算 future window 并切片返回。
3. 端侧合并新页进入已看队列；回滚时直接读已看队列。

## 失败与回退
- token 无效/过期：按新会话从 offset=0 重新开始。
- 排序候选不足：返回剩余 items，`nextCursor` 置空。
- 兼容策略：按 `v` 做向后兼容解析。

## 取舍
- 不在云侧保存完整历史，降低会话状态复杂度与存储压力。
- 代价是端侧需承担历史窗口缓存责任，但更符合交互体验。
