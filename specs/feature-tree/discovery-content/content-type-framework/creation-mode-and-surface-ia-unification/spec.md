# L3 Story：creation-mode-and-surface-ia-unification（商用上线版 v1.1）

## 节点定位

- `L1_capability`：`discovery-content`
- `L2_feature`：`content-type-framework`
- `L3_story`：`creation-mode-and-surface-ia-unification`

本 Story 负责定义趣我圈内容从创作入口到发现、圈子、主页与小趣助手消费的统一信息架构，解决当前“内容身份”和“内容格式”混用导致的心智割裂问题。

本版 `v1.1` 在统一 IA 的基础上，补齐商用上线所必需的六类冻结项：跨 Story 继承关系、数据生命周期合同、分享产品契约、小趣权限边界、商用品质指标、迁移灰度与回滚要求。凡涉及创作入口、发现、圈子、主页、小趣消费的用户可见内容语义，与本 Story 冲突时以本 Story 为准。

## 背景与动机

当前内容链路存在六类割裂：

1. **入口分层错误**：创作入口过早要求用户在 `微趣/作品/图片/视频/文章` 间做分类判断，用户在开始表达前就要回答产品内部的结构题。
2. **页面语义不一致**：发现页用 `微趣/作品` 双轨，主页改造成 `创作/圈子/互动/生活`，圈子页又使用自己的创作语义，用户难以形成稳定心智。
3. **身份与格式混用**：同样是“多图 + 少量文字”内容，在点滴分享和精品沉淀之间没有清晰边界，导致编辑入口、卡片展示和后续消费都缺乏统一规则。
4. **小趣消费链路不清**：高价值作品尚未被清晰定义为可沉淀、可检索、可引用的长期知识资产，点滴内容也未被明确定义为短中期上下文来源。
5. **分享契约缺失**：用户希望像微博、微信朋友圈、小红书一样自然分享，但当前“分享”仍混在 IA 讨论里，没有冻结为动作层产品契约。
6. **商用发布缺口**：当前规格尚未明确跨 Story 覆盖顺序、SLO、灰度步进、回滚条件、权限撤销时效与历史内容迁移策略，不能直接作为上线版 PRD。

本 Story 的目标不是再新增一种内容类型，而是先统一整条链路的“内容语法”：

- 用户先开始表达，而不是先做格式分类
- 内容先确定“身份”，再决定“展示与分发”
- 各展示页只回答自己的场景问题，不再复用别的页面的分类层
- 分享、小趣消费、灰度发布都围绕同一套内容身份合同工作

## 目标用户

- 想快速记录当下、带图或视频轻量表达的用户
- 想沉淀攻略、经验、图文内容、系列视频的创作者
- 通过圈子与趣聊进行关系传播和同频互动的用户
- 需要理解用户长期偏好、短期状态与可复用内容资产的小趣助手

## 功能范围

### F1. 统一内容语法

定义四层统一语法，作为入口、编辑、展示、分享和助手消费的共同真相源：

1. **内容总类**：`创作`
2. **内容身份**：`点滴` / `作品`
3. **内容格式**：`图片` / `视频` / `笔记`
4. **内容沉淀层级**：普通内容 / 精选内容 / 攻略清单等高价值资产

约束：

- `点滴/作品` 是身份层，不是格式层
- `图片/视频/笔记` 是承载形式，不是一级世界观
- 个人主页中的 `创作` 是总集合；发现页中的 `作品` 仅指“精品沉淀内容”消费轨
- 对外统一术语使用 `点滴`；代码、旧文档、旧埋点中的 `微趣`、`micro`、`moment` 只允许保留在兼容层，不得继续作为新的用户可见 IA 文案

### F2. 入口动作优先

创作首层入口只回答“如何开始表达”，不再暴露格式分类：

- `从相册选`
- `写点什么`
- 可选：`拍一下`

要求：

- 首层入口不再区分 `视频/图片/笔记`
- 首层入口不再区分 `微趣图片/作品视频/作品文章`
- 用户进入编辑器前，主决策不超过 1 次

### F3. 统一编辑器与身份切换

统一编辑器根据用户素材自动适配：

- 选图片：进入图文编辑态
- 选视频：进入视频编辑态
- 先写文字：进入文字编辑态，并允许继续补图/视频

编辑器中提供轻量身份切换：

- `点滴`
- `作品`

要求：

- 模式切换不丢失已选素材、正文、圈子、位置与可见性
- `作品` 模式可在当前 `article` 契约上先以 `笔记` 作为 UI 术语过渡，不立即引入新的后端内容类型
- 身份切换只改变包装深度和分发方式，不重置草稿

### F4. 发布前身份确认与默认策略

发布前由系统给出默认建议，但允许用户显式确认：

- `作为点滴发布`
- `整理为作品`

默认策略：

- 无标题、短文案、快速发布，优先建议 `点滴`
- 有标题、封面、摘要、标签或较强沉淀意图，优先建议 `作品`

其中：

- `点滴`：不强制标题，强调即时表达、社交互动、关系流分发
- `作品`：支持标题、封面、摘要、标签，强调收藏、搜索、沉淀、小趣复用

### F5. 发现页展示语法

发现页继续保留消费双轨，但只承担“内容气质分流”职责：

- `点滴`
- `作品`

要求：

- `点滴` 卡片以“人、时间、互动、当下感”为主
- `作品` 卡片以“封面、标题、内容价值、收藏潜力”为主
- `作品` 内允许二级筛选 `全部 / 图片 / 视频 / 笔记`
- `分享` 是动作层，不是发现页一级分类层

### F6. 圈子页与主页展示语法

圈子页和主页不复用发现页的消费双轨，而使用“创作容器”视角：

- 一级：`创作`
- 二级：`全部 / 点滴 / 作品`
- 进入 `作品` 后，可继续筛选 `图片 / 视频 / 笔记`

要求：

- 圈子页优先表达“主题与关系容器”，内容筛选从属于圈内创作区
- 主页优先表达“个人资产管理”，不直接使用发现页的 `点滴/作品` 一级双轨
- 同一条内容在发现、圈子、主页中身份不变，只因场景不同而改变卡片结构

### F7. 点滴升级为作品

支持从轻表达自然升级为沉淀内容：

- 先发点滴
- 后续进入编辑或一键整理
- 保留原素材、正文、圈子、位置与可见性
- 补充标题、封面、摘要、标签后转为作品

该链路是连接“社交表达”与“精品沉淀”的关键能力。

### F8. 小趣助手消费规则

定义内容对小趣的消费语义：

- `点滴`：进入短中期上下文记忆，用于理解用户最近状态、兴趣、情绪和社交语境
- `作品`：进入长期知识资产，用于检索、引用、总结、生成建议
- `精选/攻略/清单`：作为高优先级可信来源，但必须同时满足权限可见、未被撤销、内容新鲜度合格

### F9. 分享产品契约

分享始终属于动作层，不进入一级 IA 分类：

- `点滴` 分享强调“即时、关系、当下感”
- `作品` 分享强调“标题、封面、摘要、沉淀价值”
- 不同平台复用同一套内容身份合同，不因外部平台差异新增新的站内一级分类

## 数据生命周期合同

1. `点滴 -> 作品` 默认采用**原地升级**：`postId`、deeplink、分享链接保持不变，内容版本号递增并记录升级事件。
2. 评论、点赞、收藏、转发、审核记录默认延续到升级后的作品，不额外复制出第二条主内容。
3. 升级默认继承原素材、圈子、位置、可见性；用户可在升级时显式修改，修改后以最新权限为准。
4. 升级后必须重算发现、圈子、主页投影与助手索引；作者主页和圈子收敛 `P95 <= 5s`，发现页与助手索引收敛 `P95 <= 30s`。
5. `作品 -> 点滴` 本期不支持，以避免链接、审核、助手引用与分享语义倒退。
6. 删除、转私密、退出圈子、作者撤销授权时，分享落地页、检索资格与助手引用资格都必须在 `P95 <= 5 分钟` 内失效。

## 小趣消费与隐私边界

| 可见性 | 小趣可用范围 | 保留策略 |
|------|------|------|
| `仅自己可见` | 仅作者自己的小趣会话可用；禁止进入跨用户检索与公共知识池 | `点滴` 默认保留 30 天上下文；`作品` 对作者长期保留，直到删除或显式排除 |
| `圈内可见` | 仅作者会话与通过圈子权限校验的场景可用；禁止进入公共知识池 | 必须按圈子成员资格实时校验；退圈或改权限后立即撤销资格 |
| `公开可见` | `作品` 可进入公开知识资产；`点滴` 默认仍只作为上下文，不自动进入高优先级公共知识 | 必须标注来源、时间与作者；若作者关闭授权则撤销公开知识资格 |

补充要求：

- 必须提供“小趣使用我的创作内容”的总开关，以及单条内容“不给小趣使用”的排除能力。
- 小趣回答引用用户内容时，必须明确标注来源类型，例如“来自你的作品”“来自你最近的点滴”“来自你所在圈子的公开内容”。
- `精选/攻略/清单` 只有在权限允许、未被撤销、内容新鲜度满足要求时，才可作为高优先级引用来源。

## 分享产品契约

| 内容身份 | 默认分享载体 | 默认落地页 | 失败回退 |
|------|------|------|------|
| `点滴` | 首图或拼图 + 短文案 + 作者 + 时间/圈子语境 | 点滴详情页，保留评论与互动线程 | `复制链接` / `保存海报` / `系统分享` |
| `作品` | 封面 + 标题 + 摘要 + 作者 + 标签 | 作品详情页，保留收藏与继续浏览入口 | `复制链接` / `保存海报` / `系统分享` |

补充要求：

- 分享动作不改变站内 IA；平台差异仅体现在分享模板与目标能力，不新增站内一级分类。
- `仅自己可见` 内容不允许对外分享；`圈内可见` 内容只允许生成权限受控链接。
- 本 Story 冻结分享 payload 最小字段：标题、摘要、封面、作者、deeplink、权限态、失败回退策略；平台 SDK 级接入在后续 Story 实施。

## 非功能目标（商用品质指标与上线门槛）

| 指标 | 商用门槛 |
|------|------|
| 入口到编辑器可交互 | warm `P95 <= 1.2s`，cold `P95 <= 2.0s` |
| 身份切换反馈 | `P95 <= 150ms`，且关键字段零丢失 |
| 草稿自动保存与恢复 | 本地自动保存成功率 `>= 99.9%`；杀进程后恢复 `P95 <= 2s` |
| 发布成功率 | 健康网络下 `>= 99.5%` |
| 跨面收敛时延 | 作者主页/圈子 `P95 <= 5s`；发现页/助手索引 `P95 <= 30s` |
| 创作链路稳定性 | 创作与发布关键路径 `crash-free sessions >= 99.8%` |
| 分享完成率 | 已支持路径下 `>= 98%` |
| 权限撤销时效 | 删除/转私密/退圈后，分享与助手资格撤销 `P95 <= 5 分钟` |

未满足任一 blocker 指标，不得扩大灰度或视为“商用上线就绪”。

## 迁移、灰度与回滚

1. **读路径兼容阶段**：历史内容先按现有契约推导默认身份，`micro -> 点滴`，`image/video/article -> 作品`；旧草稿按同一规则读出。
2. **写路径灰度阶段**：动作优先入口、统一编辑器、跨面 IA、助手索引、分享模板必须分别受独立 feature flag 控制。
3. **字段正式化阶段**：若 `/design` 确认新增 `contentIdentity`，必须执行 `metadata -> codegen -> 双读双写 -> 回填 -> 单写` 顺序，禁止一步切断旧读路径。
4. **灰度步进**：内部灰度 `5%` → 白名单 `20%` → `50%` → `100%`，每阶段至少观察 `24h` 且核心指标稳定。
5. **观测指标**：创作启动率、发布成功率、草稿恢复率、跨面身份一致率、分享完成率、助手撤销时延、关键路径 crash-free。
6. **回滚策略**：任一能力出现指标越线、隐私泄露、跨面身份错乱或草稿丢失，必须支持按能力单独回退入口、编辑器、展示、分享或助手索引，不要求整包回滚。

## Out of Scope

- 本 Story 不直接落具体编辑器 UI 细节、交互动画和字段布局，这些在 `/design` 阶段展开
- 本 Story 不新增新的后端内容实体或独立表结构；是否新增 `contentIdentity`、`expressionMode` 字段，需在 `/design` 阶段 metadata-first 评审
- 本 Story 不直接实施外部平台 SDK 级分享链路（微信、朋友圈、微博、小红书），仅冻结产品契约与 payload 结构
- 本 Story 不定义推荐算法、排序权重和助手检索策略实现细节
- 本 Story 不处理富文本排版、攻略模板市场、AI 自动美编等增强能力

## 约束

### 元数据与契约约束

- 现有 `Post` 单聚合和 `contentType`（`micro / image / video / article`）仍是当前系统主契约
- UI 层可使用 `笔记` 作为 `article` 的展示术语，但不能绕过 metadata 私自引入第二套内容类型真相源
- 若后续需要引入“内容身份”字段，必须遵循 `metadata YAML -> make -C quwoquan_service verify-metadata -> make codegen -> make codegen-app -> 业务逻辑` 顺序

### 页面语义约束

- 发现页的 `作品` 是消费轨，不等同于主页和圈子中的 `创作`
- 主页与圈子中的 `创作` 必须满足 `创作 ⊃ 点滴 + 作品`
- 圈子页中的内容筛选必须挂在圈内创作区，而不是上升为圈子首页全局主导航
- 用户可见 IA 不得继续混用 `微趣/点滴`、`文章/笔记`、`作品/创作` 等同层概念

### 分享与助手约束

- `分享` 只能作为动作层，不得再次成为一级 IA 分类
- 小趣对内容的使用必须区分“上下文记忆”和“知识资产”，且权限校验必须在读取时再次执行，不能只在写入时判断
- 删除、改权限、退圈、关闭授权后，助手与分享资格必须按时撤销，不允许“删除后仍被引用”

### 商用发布约束

- `/prd` 阶段必须冻结跨 Story 覆盖矩阵、数据生命周期、权限边界、分享契约、SLO、灰度与回滚策略
- `/design` 阶段必须把上述冻结项落实为 metadata 方案、feature flag 方案、观测方案与测试证据矩阵；否则不得进入 `/dev`

## 对标输入与吸收结论

### 外部对标

| 产品 | 借鉴点 | 不借鉴点 | 吸收结论 |
|------|--------|----------|----------|
| 小红书 | 入口按动作开始（相册/写作）、作品包装感强、图文沉淀心智清晰 | 不直接照搬全站为“笔记平台” | 借鉴“先开始表达，再决定内容包装”的入口策略与作品包装方式 |
| 微博 | 点滴型轻表达、单列社交信息流、图文轻互动 | 不借鉴全部内容平铺为同一类动态 | 借鉴点滴在社交场中的表达与互动语义 |
| 微信朋友圈 | 熟悉的关系链点滴表达、低门槛发布、分享心智自然 | 不借鉴封闭传播和弱沉淀结构 | 借鉴点滴的关系流心智与分享动作层习惯，不用其作为 IA 主轴 |
| 抖音 | 灰度发布、权限过滤、回退路径明确，创作链路商业化成熟 | 不借鉴单一视频内容世界观 | 借鉴商用上线所需的灰度、回滚、权限和稳定性治理标准 |

### 内部对标

| 现状 | 结论 |
|------|------|
| 发现页 `微趣/作品` 双轨 | 架构可保留，但用户可见语义要统一为 `点滴/作品` |
| 主页 `创作/圈子/互动/生活` 方向 | 继续保留为主页一级 IA |
| 圈子页“作品”改“创作”趋势 | 与本 Story 一致，应统一收口到 `创作` 容器 |
| 创作入口 6 宫格分流 | 应废弃，改为动作优先入口 |
| `Post + contentType` 现有契约 | 继续兼容，必要时在 `/design` 中新增身份字段并双读双写过渡 |

## 角色分工

- **产品**：冻结 `点滴/作品/格式/创作` 的术语边界，明确跨 Story 覆盖顺序、分享契约与商用上线门槛
- **架构/设计**：定义“内容身份”与现有 `contentType`、`ui_config.yaml`、各页面 surface 的映射关系，收口迁移、灰度、回滚与观测方案
- **端侧开发**：落统一入口、统一编辑器、各展示页身份一致性改造，以及分享模板与权限态展示
- **云侧开发**：评审是否需要新增身份字段、投影字段或助手消费契约，并确保 metadata-first 与双读双写过渡
- **测试**：建立 `T1~T4` 的跨页面一致性、权限撤销、灰度回滚与弱网恢复证据
- **发布/灰度**：控制入口、展示、分享、小趣索引的灰度范围，执行逐级放量与回滚演练

## 验收重点

- A1：统一内容语法冻结并映射到各页面，不再混用“身份层”和“格式层”
- A2：创作首层入口改为动作优先，不再首屏暴露格式分类
- A3：统一编辑器支持自动适配和 `点滴/作品` 身份切换，且草稿不丢
- A4：发布前身份确认与默认建议规则明确，点滴与作品包装深度差异清晰
- A5：发现页消费双轨与用户可见文案统一收口
- A6：圈子页、主页在各自语义层上保持一致而不混层
- A7：支持 `点滴 -> 作品` 升级链路，且 `笔记` 作为 `article` 的 UI 术语过渡清晰
- A8：小趣对点滴与作品的消费规则明确，可进入后续设计和实现
- A9：商用品质指标、SLO 与上线 blocker 冻结
- A10：既有 Story 覆盖顺序与继承矩阵冻结，不再存在第二套 IA 词表
- A11：`点滴 -> 作品` 的数据生命周期合同冻结，`postId`、互动、链接与索引语义明确
- A12：小趣权限边界、保留策略、撤销时效与引用标注冻结
- A13：分享产品契约冻结，平台差异下仍保持动作层定位
- A14：历史内容迁移、feature flag、灰度步进与回滚方案冻结
