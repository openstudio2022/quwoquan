# L3 Scenario: creation-mode-and-surface-ia-unification

## 节点定位

- `L1_capability`: `discovery-content`
- `L2_journey`: `content-type-framework`
- `L3_scenario`: `creation-mode-and-surface-ia-unification`

本次重定稿用于替换该场景旧有的“身份切换 + 四 Tab + 六宫格入口”基线，冻结一版可直接进入 `/dev` 的创作体验规格：

- 创作首屏只回答“如何开始表达”
- 创作过程只保留两个编辑器
- 标题变成按需出现的升级入口，而不是默认重表单
- 发布设置继续保留，但默认轻量折叠，不再压在首屏

本场景与既有内容展示、发布链路、圈子分发、小趣使用能力保持兼容；若与过往版本 `spec.md/design.md/tasks.md` 冲突，以本次重定稿为准。

## 背景与动机

当前创作链路的主要问题已经从“功能不够”变成“认知过载”：

1. **入口过早让用户回答结构题**：用户一点击 `+`，就会被要求理解 `点滴/作品/图片/视频/文章` 等内部分类，而不是顺手开始表达。
2. **编辑器同时承载太多世界观**：当前 `CreatePage` 同时维护 `moment/photo/video/article` 四种状态，并叠加身份建议、格式切换、草稿、设置、图片编辑等能力，导致结构臃肿。
3. **轻内容被重表单拖慢**：图片、视频、文章路径都会默认露出标题输入框，轻量表达也显得像在填写后台表单。
4. **视觉语义不够统一**：全局 `+`、发现页 `+`、创作入口抽屉、编辑器顶部与设置区之间的 iOS 语义不统一，缺少一眼可懂的清爽感。
5. **已有方案虽然“功能完整”，但不够极致**：现有基线更像“兼容旧世界的新架构”，而不是为用户直接设计的第一性创作体验。

这次重定稿的目标不是继续往旧编辑器上叠能力，而是先收敛创作主语法，把整条链路压缩到“一个入口、两个编辑器、一套渐进规则”。

## 目标用户

- 想快速从相册或相机开始发图、发视频的用户。
- 想先写几句，再决定要不要加标题、加图片、写成长文的用户。
- 希望创作过程像 iOS 原生能力一样清爽、顺手、低打扰的用户。
- 对发布结果有一定质量诉求，但不愿在开始阶段做复杂分类判断的用户。

## 核心体验原则

### P1. 动作优先

创作入口只回答“怎么开始”，不回答“你要发成什么”。

### P2. 渐进披露

标题、包装、发布设置等重能力只在需要时出现，不默认打满首屏。

### P3. 两个编辑器就够

相册/相机进入统一媒体编辑器，写文字进入文字为主编辑器；不再在一级体验中暴露四种创作页。

### P4. iOS 风格优先

入口、导航、标题入口、设置入口、取消行为都遵循简洁的 iOS 语义，不做信息密度过高的功能面板。

### P5. 第一版以结构正确为先

第一版允许后续继续微调视觉和动效，但不允许保留会导致后续结构性返工的旧心智。

## 功能范围

### F1. 全局 `+` 入口统一为三段式 iOS 动作菜单

全局 `+` 按钮统一收敛为 iOS 风格动作菜单，结构固定为三段：

1. 创作动作
   - `从相册选择`
   - `相机`
   - `写文字`
2. 社交动作
   - `发起群聊`
   - `添加同好`
3. 取消动作
   - `取消`

要求：

- 创作动作与社交动作必须分组呈现。
- 首层不再出现 `点滴/作品/图片/视频/文章` 字样。
- 不使用复杂卡片网格、预览图、瀑布式说明或多列布局。
- 全局 `+` 和发现页 `+` 使用同一入口语义，不再维护两套首层创作 UI。

### F2. 创作过程只保留两个编辑器

创作首层进入后，只允许进入以下两类编辑器：

1. `媒体编辑器`
   - 由 `从相册选择` 或 `相机` 进入
2. `文字为主编辑器`
   - 由 `写文字` 进入

约束：

- 不再在一级创作流程中保留 `moment/photo/video/article` 四个用户可见编辑页。
- 不再让用户先做 `点滴/作品` 身份切换，再进入正文编辑。
- 同一路由体系可保留现有 `create-entry` / `create` 路由 ID，但用户可见心智统一为“双编辑器”。

### F3. 媒体编辑器

媒体编辑器负责承接所有“先选媒体再表达”的场景。

媒体规则冻结如下：

- 图片与视频互斥，不能共存。
- 视频最多 1 个。
- 图片可多选，并可在编辑器内继续追加。
- 图片数量沿用当前图片作品上限，不回退到 9 张限制。
- 从相册进入后可继续添加；从相机进入后也可继续补选相册内容。
- 用户可以删除视频后改为图片，也可以删空图片后改为视频。
- 媒体主预览采用图片作品式的大图承载方式。
- 下方展示横向缩略列表，最后一个入口用于继续添加。

页面结构要求：

- 顶部仅保留 `关闭 / 发布` 主导航动作。
- 首屏主体为大媒体预览。
- 标题入口与配文输入位于媒体预览下方。
- 发布设置作为次级入口，不常驻占满首屏。

### F4. 文字为主编辑器

文字为主编辑器负责承接“先写，再决定是否补图、是否长文化”的场景。

要求：

- 默认进入正文输入，不要求先填标题。
- 标题是可选项，不是默认必填项。
- 支持从纯文字自然演进为带图内容。
- 短文和长文使用同一个编辑器，不再让用户先选“点滴”还是“文章”。

正文编辑优先级：

1. 正文输入
2. 可选标题
3. 插图/补图
4. 发布设置

### F5. 标题策略采用渐进披露

标题的产品定位改为“升级入口”，而不是默认字段。

默认行为：

- 标题区域默认展示为轻量入口：`添加标题（可选）`
- 用户主动点击后，再展开为真实标题输入框
- 若已有标题、来自草稿恢复，或用户已明确进入标题编辑态，则直接展示标题输入框

重内容提示规则：

- 当内容明显变重时，系统可以轻量提示“补个标题会更清楚”
- 提示只能是非阻塞的 inline hint 或轻提醒
- 不允许用强拦截弹窗迫使用户补标题后才能发布

系统建议的第一版判定：

- `媒体编辑器`
  - 当前为视频，建议补标题
  - 图片数 `>= 4`，建议补标题
  - 配文长度 `>= 80` 字，建议补标题
- `文字为主编辑器`
  - 正文长度 `>= 140` 字，建议补标题
  - 正文出现 `>= 2` 个自然段，建议补标题
  - 已插入图片，建议补标题

发布规则：

- 轻内容可以无标题直接发布
- 重内容缺标题时，只允许做一次非阻塞提醒，用户仍可直接发布

### F6. 发布设置保留能力但轻量化

本次不移除以下能力：

- 位置
- 圈子
- 可见性
- 小趣是否可使用

但必须调整为更轻量的呈现方式：

- 主编辑页默认不长期展开完整设置表单
- 首屏只展示紧凑入口或摘要态
- 进入设置页或展开区后，才显示完整配置
- 设置能力复用既有位置/圈子选择器与现有发布契约，不在本轮新增第二套设置真相源

### F7. 草稿与连续性

创作重构不能破坏连续性。

要求：

- 媒体编辑器与文字编辑器都支持草稿保存与恢复
- 旧草稿进入新编辑器时需要可读，不允许直接丢弃
- 自动保存、杀进程恢复、继续编辑的主路径保持可用
- 路由切换、媒体继续追加、删除后改类型等操作不得造成草稿结构性损坏

### F8. 发布结果与内部兼容

本次重构的重点是端侧体验，不要求首批次同步引入新的持久化身份字段。

第一版对现有发布契约的映射策略冻结为：

- `媒体编辑器`
  - 仅图片：发布为现有图片内容契约
  - 单视频：发布为现有视频内容契约
- `文字为主编辑器`
  - 无标题、无图片、正文较短：允许映射为轻内容契约
  - 有标题、插图、正文明显较长：允许映射为现有文章契约

补充约束：

- “轻内容 / 长文”的自动判断属于系统内部兼容逻辑，不得再次上升为一级用户决策
- 若设计阶段确认需要新增 metadata 字段，必须另开 metadata-first slice，不在本次 UX 第一版中强绑定

## 用户可见术语冻结

本场景在创作阶段冻结以下用户可见术语：

- `从相册选择`
- `相机`
- `写文字`
- `添加标题（可选）`
- `发布设置`

在创作一级流程中，以下词表不得再作为首层心智暴露：

- `点滴`
- `作品`
- `图片`
- `视频`
- `文章`
- `笔记`

这些概念如需保留，只能下沉为系统内部兼容语义、发布结果语义或展示页消费语义。

## 对标输入与吸收结论

### 外部对标

| 产品 | 借鉴点 | 不借鉴点 | 吸收结论 |
|------|--------|----------|----------|
| 小红书 | 动作优先进入、图片内容包装感强、媒体主预览明确 | 不让首层编辑变成重表单和平台化运营面板 | 借鉴“先进入媒体，再慢慢补包装” |
| 微信朋友圈 | 低门槛关系流表达、熟悉的相册/相机起点 | 不借鉴封闭传播与弱沉淀结构 | 借鉴“开始极轻、表达顺手” |
| iOS Notes | 标题是自然升级入口而非强制字段 | 不借鉴纯文档工具语义 | 借鉴标题渐进披露方式 |
| iOS Photos / Action Sheet | 首层动作菜单简洁、分组清楚、取消语义明确 | 不借鉴过度工具化或多层弹板叠加 | 借鉴创作入口的系统感 |

### 内部对标

| 现状 | 结论 |
|------|------|
| `GlobalQuickActionSheet` 与 `CreateEntrySheet` 并存 | 必须统一为一套入口语义 |
| `CreatePage` 四态并存 | 必须收敛为双编辑器，不再继续叠层兼容 |
| 现有图片作品大预览 + 缩略图区 | 继续沿用，作为媒体编辑器主骨架 |
| 现有位置/圈子/可见性/小趣开关 | 保留能力，但改为次级设置呈现 |

## 权限边界与数据生命周期

### 权限边界

- 位置、圈子、可见性、小趣使用开关继续沿用现有权限语义。
- `仅自己可见` 内容仍不得对外分享。
- 单条“小趣不可使用”设置继续优先于小趣全局开关的放开语义。
- 媒体互斥、标题提示、自动判定仅影响编辑体验，不得绕过原有权限校验。

### 生命周期

- 草稿必须支持本地自动保存与恢复。
- 已发布内容不在本轮内重新定义“发布后全面可编辑”规则。
- 删除媒体、替换媒体、从图片改视频、从视频改图片均属于草稿态行为，不改变已发布内容生命周期合同。

## 非功能目标

| 指标 | 门槛 |
|------|------|
| `+` 菜单弹出反馈 | `P95 <= 150ms` |
| 入口到编辑器可交互 | warm `P95 <= 1.2s`，cold `P95 <= 2.0s` |
| 草稿自动保存成功率 | `>= 99.9%` |
| 杀进程恢复到可继续编辑 | `P95 <= 2s` |
| 发布成功率 | 健康网络下 `>= 99.5%` |
| 创作关键路径稳定性 | `crash-free sessions >= 99.8%` |

补充要求：

- 不允许出现“入口简洁了，但进入编辑器仍需重新做分类判断”的假收敛。
- 不允许为了追求能力完整，牺牲首屏清爽度。

## 灰度、迁移与回滚

### 灰度能力

第一版至少拆分以下运行时能力开关：

1. `simple_create_action_sheet`
2. `create_editor_v2`
3. `progressive_title_prompt`

### 迁移策略

- 旧入口先并行兼容，但用户可见只保留一套新入口。
- 旧草稿通过兼容适配读入新编辑器。
- 媒体选择与图片编辑子流程优先复用，不在第一版同时重写所有子工具。

### 回滚策略

- 入口体验异常时，可单独回退 `simple_create_action_sheet`
- 编辑器异常时，可单独回退 `create_editor_v2`
- 标题提示体验异常时，可单独关闭 `progressive_title_prompt`

要求：

- 任一能力关闭后，不得造成草稿不可读或发布路径中断。
- 入口与编辑器必须支持分开回退，不要求整包回滚。

## Out of Scope

- 本轮不新增富文本块编辑、攻略模板、AI 自动包装等增强能力。
- 本轮不直接接入新的外部平台分享 SDK。
- 本轮不在首批方案中强绑定新的 metadata 字段设计。
- 本轮不重做图片编辑器本体，只复用现有图片编辑能力。
- 本轮不在创作阶段重新引入 `点滴/作品` 的一级切换心智。

## 约束

- 路由层尽量保留现有 `create-entry` / `create` 标识，避免无必要路由漂移。
- 现有位置选择器、圈子选择器、图片编辑器、发布仓储优先复用。
- 媒体编辑器必须严格执行“图片与视频互斥、视频最多 1 个”。
- 标题提示只能做轻提醒，不能以“世界级质量”为理由重新把轻内容做成重表单。
- 进入 `/dev` 前，本 `spec.md`、对应 `design.md`、`acceptance.yaml`、`plan.yaml`、CR 必须全部同步完成。

## 验收重点

1. 全局 `+` 入口已统一为 iOS 风格三段式动作菜单，且创作首层不再暴露内容分类。
2. 创作一级流程只保留媒体编辑器和文字为主编辑器两条主路径。
3. 媒体编辑器的图片/视频互斥、单视频、多图片、继续追加与类型切换规则已经冻结。
4. 文字为主编辑器的“正文优先、标题可选、长内容轻提醒”规则已经冻结。
5. 标题渐进披露方案明确，可直接进入交互和开发实现。
6. 发布设置保留但轻量化，不再压垮创作首屏。
7. 旧草稿兼容、灰度开关、回滚边界和关键 SLO 已冻结，可直接进入 `/design` 落实现主线。
