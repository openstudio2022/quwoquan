# feed-item-dto-contract 设计

## 设计动因

当前端侧字段映射散落在三个位置：
1. `discovery_page.dart`：四个 `_toXxxItem()` 函数，每个含 5～10 条 `a ?? b ?? c` 别名兜底链
2. `data/mock/prototype_mock_data.dart`：字段名不统一（`likesCount`/`likes`、`avatarUrl`/`avatar` 混用）
3. `cloud/runtime/generated/`：平铺结构，`post_runtime_metadata.g.dart` 不含类型化 DTO

这三处散乱导致：新增字段需在三处同步改动、Mock 与 Remote 输出结构不一致、端云字段漂移无法被 gate 发现。

## 当前态 → 目标态

| 维度 | 当前态 | 目标态 |
|------|--------|--------|
| DTO 来源 | `Map<String, dynamic>`，手工别名 | `FeedItemDto`（codegen，`DO NOT EDIT`） |
| 字段权威 | 无，分散在各 `_toXxxItem` 函数 | `_projections/discovery_feed.yaml` → `client_projection.fields` |
| mock 数据 | `data/mock/prototype_mock_data.dart`，非规范字段名 | `cloud/services/content/mock/content_mock_data.dart`，100% 规范字段 |
| Repository 输出 | `Map<String, dynamic>` | `FeedItemDto`（Mock/Remote 两端一致） |
| `discovery_page.dart` | 4 个映射函数，大量别名链 | 直接 `dto.likeCount`、`dto.authorId` 等 |
| `generated/` 结构 | 平铺 | 按域：`content/`、`user/`（与 `cloud/services/` 对齐） |

## 方案对比

### 方案 A：端侧 normalizer 函数（不改 codegen）
- 在 `ContentRepository` 手写 `_normalize(Map<String, dynamic>)` 函数解析别名
- **优点**：快速，不动 codegen 工具
- **缺点**：normalizer 仍需手写字段列表；字段增减不受 gate 保护；换人维护时无法知道哪些是规范字段

### 方案 B（选定）：metadata codegen 生成 FeedItemDto
- `_projections/discovery_feed.yaml` 增加 `client_projection.fields` 节
- `codegen_app_metadata/main.go` 读取投影定义，生成 `FeedItemDto`（含 `fromMap` + alias resolver）
- **优点**：metadata 单一权威，`make gate` 可验证端云字段一致性，字段变更自动传播，DO NOT EDIT 保护
- **缺点**：需改 codegen 工具，一次性成本；但后续字段维护成本趋近于零

**选择方案 B**，理由：`metadata-first` 是本项目核心约束，codegen 保护是防止端云漂移的关键机制；方案 A 在下一次字段变更时仍会产生维护负担。

## `cloud/runtime/generated/` 目录重组

```
重组前（平铺）：
generated/
  post_runtime_metadata.g.dart

重组后（按域，与 cloud/services/ 对齐）：
generated/
  content/
    feed_item_dto.g.dart          ← 新增，从 _projections/discovery_feed.yaml 生成
    content_metadata.g.dart       ← 重命名自 post_runtime_metadata.g.dart
  user/                           ← 预留，author_dto.g.dart 待 user_profile 完善后生成
```

**设计原则**：generated 按域组织后，`cloud/services/content/content_repository.dart` 只需 import `generated/content/`，不会误引其他域产物，职责边界清晰。

## FeedItemDto 生成结构（目标 dart 类骨架）

```dart
// Code generated by codegen_app_metadata from discovery_feed.yaml. DO NOT EDIT.

class FeedItemDto {
  final String id;
  final String type;           // photo / video / article / moment
  final String authorId;
  final String displayName;
  final String avatarUrl;
  final String coverUrl;
  final String thumbnailUrl;
  final String title;
  final String body;
  final int likeCount;
  final int commentCount;
  final int favoriteCount;
  final List<String> imageUrls;
  final DateTime createdAt;

  const FeedItemDto({...});

  factory FeedItemDto.fromMap(Map<String, dynamic> m) {
    return FeedItemDto(
      id: m['id']?.toString() ?? '',
      authorId: m['authorId']?.toString() ?? m['userId']?.toString() ?? '',
      displayName: m['displayName']?.toString() ?? m['username']?.toString() ?? '',
      avatarUrl: m['avatarUrl']?.toString() ?? m['avatar']?.toString() ?? '',
      likeCount: (m['likeCount'] as num?)?.toInt()
               ?? (m['likesCount'] as num?)?.toInt()
               ?? (m['likes'] as num?)?.toInt() ?? 0,
      // ... 其余字段同理，alias resolver 由 codegen 生成
    );
  }
}
```

> alias resolver 列表来自 `client_projection.fields[*].aliases`，codegen 自动生成，无需手写。


## 适用场景与约束

- **适用**：`content-display-journey-consistency` 所有子旅程
- **不适用**：写操作（like/save/follow）——另见 `content-action-intent-contract`
- **约束**：
  - `generated/content/feed_item_dto.g.dart` 标记 `DO NOT EDIT`，字段变更走 metadata 流程
  - `generated/` 目录变更需同步更新所有 import 路径（C2 任务负责）
  - `content_metadata.g.dart` 重命名后旧 import 路径需全局替换

## 未来演进

- `user/author_dto.g.dart`：当 `user_profile/fields.yaml` 完善后，由 codegen 生成 `AuthorDto`
- FeedItemDto 扩展为端侧 ViewModel（含 UI 计算字段如 `timeAgoText`、`formattedLikeCount`）：在 DTO 基础上包装 ViewModel 层，DTO 保持 DO NOT EDIT
- `make gate-full` 增加 FeedItemDto 字段 vs. `post/fields.yaml` 自动比对，彻底封死端云漂移
