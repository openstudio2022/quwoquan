# L1 规格：共享主页网络（v1）

## 节点定位

- `L1_capability`: `shared-homepage-network`

该节点是 App 内“具体事物长期主页”的唯一能力归属。  
它承接车型、酒店/民宿、餐厅、景点等具体事物的主页建档、展示、口碑、认领、下线与内容挂载，不承接群组/关系主页本体的详细治理。

## 背景与动机

当前产品在“具体事物如何被建立为长期主页”上存在五个结构性问题：

1. **主页缺位**：车型、酒店、景点、餐厅等具体事物缺少统一、长期、可复用的主页，内容挂载和口碑沉淀没有稳定落点。
2. **来源碎片化**：网络抓取、用户补充、后续官方认领之间没有统一主线，导致冷启动、补录、维护和治理无法闭环。
3. **口碑无锚点**：用户可以围绕具体事物讨论，但没有稳定的主页锚点承接评分、标签、精选内容和记录记录。
4. **生命周期缺位**：酒店、餐厅、景点等现实世界对象会关闭、停业、迁移，但当前没有统一的“已下线但保留记录”的合同。
5. **命名与边界混乱**：前台不应教育用户“实体”或“对象平台”，但内部又需要稳定的领域服务、主档模型和挂载契约。

本次 PRD 的目标，是把产品冻结为一套统一的“共享主页”能力：

- 前台统一让用户看到“具体事物的主页”
- 产品层统一把这套能力称为 `共享主页`
- 云侧继续由 `entity-service` 承载主档、主页读模型、认领和治理
- 主页成为内容挂载、口碑沉淀、搜索发现和后续商业承接的统一锚点

## 产品定位

`shared-homepage-network` 的产品定位不是黄页目录，也不是群组关系网络，更不是交易平台。

它的定位是：

- **具体事物的长期主页网络**
- **内容挂载与口碑沉淀的统一锚点**
- **搜索发现与后续商业承接的基础设施**

因此：

- 用户理解的是“某个具体事物有一个主页”
- 口碑依附主页，而不是独立漂浮
- 内容围绕主页传播，但主页不替代内容域
- 群组围绕主页沉淀关系，但主页不替代群组
- baseline 只做主页网络，不做交易闭环与组织型关系主页治理

## 目标用户

- **浏览用户**：查看某个具体事物的主页、口碑、内容、问答和相关群组。
- **内容发布者**：在发布笔记、作品、提问、口碑时关联具体事物主页。
- **社区贡献者**：补充缺失主页、纠错、报告下线、上传真实内容和口碑。
- **认领者/维护者**：代表品牌、门店、景区或经营主体申请认领主页并维护基础信息。
- **平台运营与审核**：做抓取建档、归类、发布、认领审核和下线治理。

## 能力边界

`shared-homepage-network` 负责：

- 共享主页的概念、类目范围、模板和生命周期。
- 主页主档、主页读模型、口碑模板、认领、下线治理。
- 发布内容时的主页搜索、主页挂载与缺失主页补充。
- 主页与内容、群组、搜索之间的统一集成契约。
- 冷启动阶段从网络抓取、AI 抽取、用户补充到主页发布的主线设计。

`shared-homepage-network` 不负责：

- 群组总入口、关系主页模板、群角色和组织节点治理细节。
- 聊天、群资料、组织结构权限体系。
- 交易、预约、支付、团购闭环。
- 搜索底层引擎、倒排/向量检索实现。
- 学校、公司、院系、班级等组织型主页的详细建模与治理。

## 关键 Journey

本 L1 当前冻结 3 个 `L2_journey` 方向，现已全部正式拆出：

| L2 Journey | 说明 | 当前状态 |
|---|---|---|
| `homepage-discovery-and-attach-journey` | 用户从搜索、内容发布、主页详情中发现和挂载共享主页 | 已拆出 |
| `homepage-review-and-content-journey` | 用户围绕主页浏览口碑、内容、问答并发布新内容 | 已拆出 |
| `homepage-claim-maintain-and-offline-journey` | 平台、贡献者、认领方围绕主页完成补录、认领、维护和下线治理 | 已拆出 |

## 领域服务与业务对象

| 领域 | 业务对象 | 本 L1 角色 |
|---|---|---|
| `entity-service` | `EntityProfile`、`EntityTypeTemplate`、`EntityReviewTemplate`、`EntityRelation`、`EntitySourceEvidence`、`EntityClaimRequest`、`EntityStatusReport` | 共享主页主档、模板、认领、状态治理与主页读模型的唯一归属 |
| `content/post` | `Post`、`Comment`、内容投影 | 作为挂载共享主页的公开内容来源 |
| `social/circle` | `Circle` 或后续关系主页读模型 | 作为相关群组的消费方与反向推荐来源 |
| `_shared` metadata | `app_routes`、`ui_surfaces`、`request_context` | 作为主页路由、surface 与请求上下文真相源 |

## 功能范围

### R1：共享主页模型与页面骨架

- R1.1：每个已发布的具体事物都应有一个长期主页。
- R1.2：baseline 阶段只支持 `车型 / 酒店民宿 / 餐厅 / 景点` 四类主页。
- R1.3：主页统一包含 `总览 / 口碑 / 内容 / 问答 / 相关群组 / 官方或服务信息` 六个模块。
- R1.4：不同类目在统一骨架下允许使用不同的字段模板和模块权重。
- R1.5：前台永远显示具体类目名和具体名称，不向用户暴露“实体”或“directory”。

### R2：主页来源与 0→1 冷启动

- R2.1：主页来源统一支持 `网络抓取建档 / 用户补充 / 内容反向抽取 / 运营导入` 四条路径。
- R2.2：抓取与抽取产物必须先进入候选或待校验状态，不能直接跳过治理发布。
- R2.3：AI 可以参与摘要、标签、结构化字段补全与展示配置建议，但不能直接替代事实校验。
- R2.4：用户在发布内容时搜索不到主页，可触发“补充主页”流程。
- R2.5：用户补充一个主页，不等于拥有它。

### R3：主页搜索、挂载与发布集成

- R3.1：全局内容动作统一叫 `发布`，内容类型统一为 `笔记 / 作品 / 提问 / 口碑`。
- R3.2：`口碑` 必须且只能绑定 1 个共享主页。
- R3.3：`笔记 / 作品 / 提问` 可绑定也可不绑定共享主页。
- R3.4：从全局发布入口进入时，发布页可补选主页；从主页进入发布时，当前主页默认已选中。
- R3.5：绑定主页后，内容自动进入该主页的内容区、问答区或口碑区。

### R4：口碑能力

- R4.1：口碑依附于共享主页，不作为独立漂浮对象存在。
- R4.2：评分采用 `1.0 ~ 5.0`，步长 `0.5`。
- R4.3：总评分与维度评分同时存在，维度模板跟类目走，不跟群组走。
- R4.4：baseline 阶段口碑模板至少覆盖车型、酒店民宿、餐厅、景点四类。
- R4.5：主页必须展示评分摘要、口碑数量、维度摘要和精选口碑。

### R5：认领与维护

- R5.1：已发布主页支持认领申请。
- R5.2：认领采用分层验证：baseline 至少支持基础认领与强验证认领两级。
- R5.3：认领后可维护基础信息、封面图库、营业/开放状态、官方说明和服务入口。
- R5.4：认领方不能直接删除真实用户口碑或覆盖记录用户内容，只能发起申诉或补充说明。
- R5.5：baseline 不支持复杂的总部批量门店运营与多门店后台。

### R6：主页生命周期与下线

- R6.1：主页生命周期至少区分 `候选 / 待校验 / 已发布 / 已下线`。
- R6.2：用户可上报 `已关闭 / 已结束 / 信息错误 / 长期停更`。
- R6.3：baseline 统一把用户可见终局状态收口为 `已下线`，并附原因标签。
- R6.4：主页下线后保留原 URL、记录内容、记录口碑和相关群组关系。
- R6.5：已下线主页在搜索与推荐中降级，但仍可被查看。
- R6.6：baseline 不做主页合并与跳转迁移。

### R7：与群组的集成

- R7.1：共享主页可以推荐相关群组，但不承接群组治理。
- R7.2：群组可关联 1 个主共享主页，但群组只消费主页摘要，不拥有主页主档。
- R7.3：群组中的口碑精选来自主页口碑，不单独维护群组评分。
- R7.4：群组/关系主页详细模板、角色与群设计不属于本 L1 负责范围。

### R8：端云一体化与 iOS 体验基线

- R8.1：端侧新增 `ui/entity/` 与 `cloud/services/entity/` 目录归属。
- R8.2：云侧由 `entity-service` 提供 typed contract，禁止长期停留在松散 `Map` 协议。
- R8.3：主页路由、surface、request context 必须走 metadata 唯一真相源。
- R8.4：主页详情、主页选择器、评分交互必须满足 iOS 原生高品质体验要求。

## baseline 范围冻结

### baseline 必做

- 建立 `shared-homepage-network` 独立能力域
- 冻结四类主页：`车型 / 酒店民宿 / 餐厅 / 景点`
- 冻结主页统一骨架：`总览 / 口碑 / 内容 / 问答 / 相关群组 / 官方或服务信息`
- 冻结发布挂载主页契约与口碑依附主页契约
- 冻结认领材料分层、下线软保留合同与 iOS 基线体验
- 冻结主页与群组的最小集成边界：`群组最多关联 1 个主主页，主页推荐相关群组`

### baseline 不做

- 学校、公司、院系、班级等组织型主页正式模板
- 主页合并、记录迁移和复杂图谱推荐
- 交易、预约、支付、团购和商家经营后台
- 主理人自助创建并立即归属公共主页
- 实时持续全网抓取调度平台

### 后续规划

- `owner-managed homepage`：主理型主页创建与归属
- `organization homepage template`：学校/公司等组织型主页与共享主页基础设施复用
- `homepage relationship graph`：多主页关系图谱、系列/品牌/地理联动
- `commercial entry`：预约、服务、交易等商业承接能力

## 不做什么（Out of Scope）

- OS1：不做学校、公司、院系、班级等组织型主页的正式模板。
- OS2：不做群组详细治理、群模型、角色体系与组织树设计。
- OS3：不做交易、预约、支付、团购和商家后台闭环。
- OS4：不做主页合并、跨主页记录迁移和多主页图谱推荐。
- OS5：不做持续实时全网增量抓取调度系统。

## 约束与适用边界

### 产品约束

- 前台统一说“主页”，不对用户暴露“实体”与“directory”。
- 主页的用户理解必须是“具体事物的主页”，不是抽象对象条目。
- 主页可以挂内容、承接口碑、推荐群组，但不替代群组入口与关系沉淀。

### 业务约束

- baseline 只覆盖四类可强口碑化、可商业承接的主页。
- 认领只能获得维护权与官方展示位，不能获得对用户口碑的控制权。
- 下线优先做记录保留，而不是物理删除。

### 技术约束

- 服务名允许继续使用 `entity-service`，但新增 PRD 与页面文案不再扩散 `entity` 到前台。
- `route / surface / operation / request_context / review_template` 必须以 metadata 为唯一真相源。
- 发布器、主页选择器、主页详情页与群组集成必须走 typed contract 与 codegen 常量。

## 对标输入与吸收结论

| 对标 | 借鉴点 | 本次吸收 | 不吸收 |
|---|---|---|---|
| 小红书 | 内容关联具体事物、达人/圈子二次传播、搜索与推荐联动 | 吸收为“内容先发布，再挂主页”的主线 | 不做纯标签化和强运营黑箱流量叙事 |
| 大众点评 | 围绕具体地点形成稳定主页与口碑心智 | 吸收为主页长期存在、口碑依附主页、下线后保留记录 | 不做只面向本地商户的单一目录站 |
| 汽车之家 | 车型级强心智、结构化信息 + 口碑 + 社区并存 | 吸收为车型主页模板与口碑+内容+群组并存 | 不做重参数站单垂类心智 |
| Booking/Airbnb | 酒店/民宿精确主页、认领与主理方维护 | 吸收为认领与后续主理型扩展边界 | baseline 不做预订支付与重交易后台 |

## 角色分工

| 角色 | 职责 |
|---|---|
| `shared-homepage-network` | 主页定义、口碑、认领、下线、内容挂载与主页读模型 |
| `content` | 提供内容写入、内容投影与主页挂载消费 |
| `circle` | 提供相关群组消费面与主主页关联消费面 |
| `search` | 提供主页搜索壳层和主页结果召回 |
| `_shared` metadata | 承载主页路由、surface、request context 真相源 |

## 数据生命周期合同

- 主页从 `候选` 进入 `待校验`，校验通过后进入 `已发布`。
- 已发布主页可被认领、被挂载内容、被写口碑、被推荐相关群组。
- 已下线主页保留记录内容、记录口碑与原 URL。
- baseline 阶段不提供主页合并与记录迁移，避免在首版把生命周期复杂度做重。

## 权限、认领与分享边界

- 主页浏览默认遵从内容与群组已有可见性范围。
- 认领申请需提供与类目相符的证明材料；审核通过前不显示官方认领标识。
- 主页支持后续分享，但 baseline 不冻结对外分享链路实现细节。
- 主理人自助创建“归属自己”的主页进入后续规划，不纳入 baseline。

## 非功能目标

### SLO / KPI

- 发布时主页搜索命中率：目标主页进入前 5 结果 `>= 90%`
- 口碑挂载准确率：`>= 99%`
- 已发布主页重复率：`< 3%`
- 认领审核 SLA：`<= 3 个工作日`
- 下线状态处理 SLA：`<= 7 天`
- 已下线主页记录保留率：`100%`
- `SearchEntities` 或等价主页搜索接口 `p95 < 800ms`
- `GetEntityPage` 或等价主页详情接口 `p95 < 1.5s`（Wi-Fi）/ `< 2.5s`（4G）

### 弱网 / 并发 / 容量

- 默认按移动弱网场景设计，主页首屏应优先展示骨架和关键摘要。
- 单模块失败不阻塞其他主页模块渲染。
- 认领审核、下线治理与抓取导入可异步化，不阻塞主浏览路径。

## 迁移、灰度与回滚要求

- baseline 按“一把上线”设计，不保留旧主页体系并行治理。
- 若主页搜索、主页详情、口碑挂载或认领链路出现持续性问题，可整版回退到旧路径或隐藏新入口。
- 下线合同一旦对用户可见，禁止在无明确迁移方案下回退为硬删除口径。

## 验收重点

详见 `acceptance.yaml`，核心维度：

1. 共享主页能力域和 `entity-service` 边界冻结完成。
2. 四类主页模板、主页生命周期与认领/下线合同冻结完成。
3. 内容与主页挂载契约、口碑依附主页契约、群组消费主页摘要契约冻结完成。
4. KPI/SLO、iOS 体验和端云一体化基线进入 `/prd` 可继续展开的状态。
