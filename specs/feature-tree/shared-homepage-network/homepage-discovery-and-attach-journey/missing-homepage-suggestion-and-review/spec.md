# L3 Scenario: missing-homepage-suggestion-and-review

## 节点定位

- `L1_capability`: `shared-homepage-network`
- `L2_journey`: `homepage-discovery-and-attach-journey`
- `L3_scenario`: `missing-homepage-suggestion-and-review`

## 背景与动机

如果用户搜索不到目标主页，只给“没搜到”的空态，会直接中断发布链路。  
因此必须提供一条“补充主页 -> 进入候选审核 -> 回到当前发布”的连续路径。

## 目标用户

- 发布内容时找不到目标主页的用户。
- 愿意帮助平台补充具体事物主页的社区贡献者。

## 功能范围

- 搜索无结果或结果不匹配时的“补充主页”入口。
- 最小必要信息提交。
- 候选/待校验状态写入。
- 提交后返回当前发布或搜索上下文。

## Out of Scope

- 候选主页发布审核后台。
- 认领申请。
- 主页合并。

## 约束

- 补充主页不能直接生成可公开浏览的正式主页。
- 提交失败不能丢失当前已填写内容。
- 不允许用自由文本长期替代主页绑定。

## 对标输入与吸收结论

- 借鉴平台型产品的“补充缺失对象”能力。
- 吸收“先补录，再审核，再公开”的治理顺序。

## 角色分工

- `shared-homepage-network`：补充主页表单、候选状态、返回链路
- `product-ops`：审核策略

## 数据生命周期合同

- 补充主页提交后进入 `candidate / pending_verify`
- 审核通过前不公开展示为正式主页

## 非功能目标

- 提交流程低打断、可恢复
- 表单失败可重试，已填内容保留

## 迁移、灰度与回滚要求

- 不保留“先发内容再人工补挂主页”的模糊旧方案。
- 如候选审核链路不稳定，可暂时隐藏补充主页入口，但不能用自由文本替代。

## 验收重点

1. 搜不到主页时有明确补充路径。
2. 提交后能回到当前上下文。
3. 候选主页不会绕过审核直接公开。
