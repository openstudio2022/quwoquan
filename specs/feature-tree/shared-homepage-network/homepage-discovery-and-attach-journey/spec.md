# L2 Journey: homepage-discovery-and-attach-journey

## 节点定位

- `L1_capability`: `shared-homepage-network`
- `L2_journey`: `homepage-discovery-and-attach-journey`

该 Journey 冻结“用户如何发现一个具体事物的主页，并在发布内容时把它挂上去”的完整链路。  
它承接主页搜索、主页选择器、缺失主页补充、主页预览与发布挂载，不承接口碑详情消费、认领维护和下线治理。

## 背景与动机

当前共享主页在用户主路径上存在四个断层：

1. **找不到**：用户想给内容关联车型、酒店、景点、餐厅时，没有统一、稳定、可预期的主页搜索与选择体验。
2. **选不准**：即使搜到结果，也缺少足够的摘要信息帮助用户区分同名主页、系列型号、门店或地理位置。
3. **补不上**：搜不到目标时，没有一条明确的“补充主页 -> 继续发布”的连续路径。
4. **挂不稳**：全局发布入口和主页内发布入口容易形成两套逻辑，导致挂载主页的规则、默认值和回流行为不一致。

本次 Journey 的目标，是把“主页发现与挂载”收成一条统一主路径：

- 用户从搜索、发布器或主页详情进入同一套主页发现与选择流程
- 找到主页后能用最少认知负担完成挂载
- 找不到主页时能补充主页并继续当前发布
- 挂载后的内容能稳定回流到主页聚合面

## 目标用户

- **发布用户**：希望在发布笔记、作品、提问或口碑时快速关联正确主页。
- **内容浏览用户**：从内容卡片、搜索结果或推荐入口进入某个主页继续浏览。
- **社区贡献者**：在搜索不到主页时补充一个缺失主页。
- **搜索型用户**：直接搜索具体事物并查看主页，而不是先经过群组或复杂中间页。

## 核心旅程

1. 用户从全局搜索、内容详情、发布入口或主页入口触发主页搜索。
2. App 打开统一的主页选择器或主页搜索页，按用户输入返回候选主页结果。
3. 用户在结果列表中通过名称、类目、位置、品牌/系列、封面和评分摘要判断目标主页是否正确。
4. 若找到目标主页，用户可：
   - 直接进入主页详情继续浏览
   - 或把该主页作为当前内容的主主页/关联主页
5. 若未找到目标主页，用户可以发起“补充主页”，填写最小必要信息后回到当前发布链路。
6. 用户完成内容发布后，该内容自动进入被挂载主页的相应聚合面：
   - `笔记 / 作品` 进入内容区
   - `提问` 进入问答区
   - `口碑` 进入口碑区

## 特性树拆分

本 Journey 冻结为 4 个 `L3_scenario` 方向：

| L3 Scenario | 负责的问题 | 归属域 |
|---|---|---|
| `homepage-search-and-picker` | 主页搜索、结果排序、选择器与主页摘要预览 | `shared-homepage-network` |
| `missing-homepage-suggestion-and-review` | 搜索不到主页时的补充主页与候选审核语义 | `shared-homepage-network` |
| `homepage-entry-and-preview` | 从结果、内容、发布器进入主页详情与摘要预览 | `shared-homepage-network` |
| `homepage-attach-in-publish-flow` | 在发布笔记/作品/提问/口碑时挂载主页并回流聚合 | `shared-homepage-network` + `content` |

## 功能范围

### In Scope

- 主页搜索输入、结果列表、结果摘要与主页选择器。
- 主页选择结果在发布器中的主主页/关联主页写入语义。
- 从主页详情页发内容时自动带入当前主页上下文。
- 搜不到主页时的“补充主页”入口与最小信息提交流程。
- 口碑内容必须先选主页的强约束。
- 发布后内容自动进入主页聚合面的回流语义。

### Out of Scope

- 主页详情页内部的完整口碑消费和问答浏览细节。
- 认领申请、认领审核与维护台。
- 主页下线、恢复、历史保留和治理后台。
- 主页合并、去重后台操作和复杂图谱推荐。
- 群组模板、群治理与角色体系。

## 约束

- 前台统一使用“主页”和具体类目名，不对用户显示“实体”。
- 主页搜索结果必须提供足够的区分信息，避免同名主页造成误挂载。
- `口碑` 必须且只能绑定 1 个主主页。
- `笔记 / 作品 / 提问` 可不绑主页，也可绑定 1 个主主页；baseline 不冻结多主页挂载。
- 全局发布入口与主页内发布入口必须共用同一套发布页，只改变默认上下文。
- 补充主页成功提交后，用户必须能回到原发布链路，不允许丢失当前编辑上下文。
- 主页搜索、主页详情路由、发布上下文与请求头 page context 必须以 metadata 为唯一真相源。

## 对标输入与吸收结论

| 对标 | 借鉴点 | 本次吸收 |
|---|---|---|
| 小红书 | 发布时挂载具体事物、内容与搜索联动 | 吸收为“发布时搜索并关联主页”的主路径 |
| 大众点评 | 找到具体地点后进入稳定主页 | 吸收为“搜索 -> 主页详情”的直接链路 |
| Booking/Airbnb | 房源/酒店主档搜索与结果区分信息完整 | 吸收为搜索结果里必须展示足够摘要 |

## 角色分工

| 角色 | 职责 |
|---|---|
| `shared-homepage-network` | 搜索主页、补充主页、主页选择器、主页详情预览、主页挂载契约 |
| `content` | 发布内容写入、主页挂载字段持久化、发布后回流聚合 |
| `search` | 主页结果召回与结果排序壳层 |
| `_shared` metadata | 主页选择器、主页详情、发布上下文的 route/surface/request context 真相源 |

## 既有 Story 覆盖矩阵

| 当前实现 / 历史能力 | 处理方式 | Journey 内新归属 |
|---|---|---|
| 全局搜索里的模糊搜索原型 | 继续沿用壳层，但主页结果独立建模 | `homepage-search-and-picker` |
| 全局发布入口里的已有内容编辑流程 | 保留编辑器，统一改为在发布页补选主页 | `homepage-attach-in-publish-flow` |
| 主页缺失时的临时手写文本方案 | 删除历史临时方案 | `missing-homepage-suggestion-and-review` |

## 数据生命周期合同

- 主页搜索结果只消费已发布主页，不把候选主页直接暴露给普通浏览用户。
- 用户补充主页后，记录进入 `candidate / pending_verify`，审核通过前不能作为正式主页公开展示。
- 发布器中的主页选择结果在用户未点击发布前只保存在本次编辑上下文中。
- 发布成功后，内容记录写入稳定的主页引用字段，并驱动主页聚合面更新。
- 口碑因依附主页存在，若未选主页则不得发布成功。

## 权限与异常边界

- 用户在当前可见范围内都可以搜索和挂载主页；主页本身不在本 Journey 内增加额外可见性模型。
- 搜索结果为空时，优先引导补充主页，而不是允许自由文本替代主页绑定。
- 补充主页失败、审核中或网络超时，不应导致当前内容编辑上下文丢失。
- 主页详情加载失败时，用户仍应可以回到结果列表或发布器，不能卡死在中间页。

## 非功能目标

### SLO

- 主页搜索首批结果 `p95 < 800ms`
- 主页选择器可交互时间 `p95 < 1.2s`
- 从主页详情进入发布器并完成默认主页预填的额外开销 `p95 < 300ms`

### KPI

- 发布场景下目标主页进入前 5 结果命中率 `>= 90%`
- 主页挂载成功率 `>= 99%`
- “搜索不到主页 -> 补充主页 -> 返回发布” 链路完成率 `>= 85%`

### 弱网与恢复

- 弱网下优先显示搜索框、最近搜索与结果 skeleton。
- 搜索失败时允许重试，不清空用户已输入 query。
- 补充主页提交失败时保留用户已填写内容。

## 迁移、灰度与回滚要求

- baseline 按统一发布页和统一主页选择器设计，不保留“旧编辑器直接文本写对象”的并行方案。
- 若主页搜索召回、主页挂载写入或主页回流聚合出现持续性问题，可先隐藏主页挂载入口或回退到“无主页挂载”的旧发布路径。
- 口碑类型一旦对外冻结为必须绑定主页，不允许在无迁移方案下回退为可不绑定。

## 验收重点

1. 用户可以从搜索、发布器和主页详情中稳定找到并选择正确主页。
2. 搜索不到主页时，能通过补充主页继续当前发布，不丢失上下文。
3. 主页挂载规则在全局发布入口与主页内入口完全一致。
4. 发布成功后，内容能正确回流到主页对应聚合面。
