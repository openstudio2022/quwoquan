# L2 Journey: homepage-claim-maintain-and-offline-journey

## 节点定位

- `L1_capability`: `shared-homepage-network`
- `L2_journey`: `homepage-claim-maintain-and-offline-journey`

该 Journey 冻结“主页如何从候选进入已发布、如何被认领维护、如何在现实世界消亡后软下线保留历史”的完整链路。  
它承接候选建档、认领申请、认领后基础维护和下线治理，不承接主页搜索与普通浏览消费细节。

## 背景与动机

共享主页如果只有浏览面，没有治理面，就无法长期成立。当前至少存在四个问题：

1. **建档不成体系**：抓取、导入、用户补充和内容反向抽取没有统一的候选 -> 发布流程。
2. **认领无合同**：经营主体、品牌方、门店方或主理人想维护主页时，没有统一的材料、审核和维护边界。
3. **维护无边界**：认领方一旦接管，容易越权影响用户口碑和历史内容。
4. **消亡无处理**：酒店、餐厅、景点会关闭、停业、结束，如果没有“已下线但保留历史”的规则，主页会失去可信度。

本次 Journey 的目标，是建立共享主页的长期治理闭环：

- 主页可从候选进入正式发布
- 正式主页可被认领并进入维护
- 已认领主页可维护基础信息但不能改写用户评价
- 主页可被软下线并保留历史

## 目标用户

- **平台运营与审核**：负责候选建档、校验发布、认领审核和下线处理。
- **社区贡献者**：补充主页、纠错、上报已关闭/已结束/信息错误。
- **认领者/维护者**：经营主体、品牌方、门店授权方或后续主理人。
- **普通浏览用户**：在主页已下线时仍能查看历史并理解状态。

## 核心旅程

1. 候选主页通过抓取、导入、用户补充或内容反抽进入候选池。
2. 平台对候选主页做归类、补全、校验并发布为正式主页。
3. 认领方提交认领申请和证明材料。
4. 审核通过后，主页进入已认领/可维护状态。
5. 认领方维护基础信息、封面、营业/开放状态与官方说明。
6. 社区或认领方可上报已关闭、已结束、信息错误或长期停更。
7. 审核通过后，主页进入 `已下线`，但保留原 URL、历史内容和历史口碑。

## 特性树拆分

本 Journey 冻结为 4 个 `L3_scenario` 方向：

| L3 Scenario | 负责的问题 | 归属域 |
|---|---|---|
| `homepage-candidate-intake-and-publish` | 候选主页 intake、校验、发布与发布前状态管理 | `shared-homepage-network` |
| `homepage-claim-request-and-review` | 认领申请、分层材料、审核状态与认领结果 | `shared-homepage-network` |
| `claimed-homepage-basic-maintenance` | 已认领主页可维护字段、维护边界、官方说明与状态更新 | `shared-homepage-network` |
| `homepage-offline-report-and-history-retention` | 下线报告、已下线状态、历史保留合同与搜索降级 | `shared-homepage-network` |

## 功能范围

### In Scope

- 候选主页从 intake 到发布的治理主线。
- 认领申请、材料提交、审核与认领结果。
- 已认领主页的基础信息维护与维护边界。
- 已下线状态、原因标签、历史保留与搜索降级。

### Out of Scope

- 主页搜索与发布挂载。
- 主页口碑读取体验。
- 主页合并与历史迁移。
- 总部多门店后台、批量运营工具。
- 交易、预约、支付。

## 约束

- 候选主页在审核通过前不能作为正式主页对普通浏览用户公开展示。
- 认领采用分层验证，类目和风险等级不同，材料强度可不同。
- 认领后只能维护基础信息、封面、状态和官方说明，不能直接改写用户口碑。
- 下线统一采用软下线，保留 URL、历史内容、历史口碑和相关群组关系。
- baseline 阶段不做主页合并和合并跳转。

## 数据生命周期合同

- 候选主页状态至少区分：`candidate / pending_verify / published / offline`
- 认领状态至少区分：`unclaimed / pending_claim / claimed`
- 已发布主页可被认领、可被下线上报、可继续承接内容和口碑
- 已下线主页停止新的运营性入口，但保留浏览历史和关系锚点
- baseline 阶段不提供“合并后迁移历史到新主页”的能力

## 权限与异常边界

- 普通用户可补充主页、纠错和上报下线，但不能直接发布正式主页或修改主页基础信息。
- 审核未通过的认领申请必须返回明确原因或可重提状态。
- 认领审核超时或材料不全时，不影响主页正常浏览。
- 下线状态错误时，应支持恢复流程，但不在本 Journey 冻结复杂恢复后台。

## 非功能目标

### SLO

- 认领审核 SLA `<= 3 个工作日`
- 下线状态处理 SLA `<= 7 天`
- 候选主页发布审核链路可支持批量处理，不阻塞用户主浏览路径

### KPI

- 候选主页审核通过后的发布成功率 `>= 95%`
- 认领审核完成率 `>= 95%`
- 已下线主页历史保留率 `100%`

### 弱网与恢复

- 认领材料上传失败时保留已填表单上下文。
- 下线上报失败时允许重试，不丢失已填原因。
- 审核结果异步返回，不阻塞普通浏览链路。

## 迁移、灰度与回滚要求

- baseline 不保留硬删除主页的旧口径，应统一切到软下线。
- 若认领链路或下线合同上线后出现严重治理风险，可先隐藏认领入口或下线上报入口，但不得破坏已下线主页历史保留。
- 主页一旦对外展示“已下线”合同，不允许回退为直接删页。

## 验收重点

1. 候选主页可进入正式发布状态，形成统一建档主线。
2. 认领申请、审核和基础维护边界清晰，认领方不能控制用户口碑。
3. 主页下线后仍保留历史内容、历史口碑和原 URL。
4. 共享主页生命周期主线形成闭环，可支撑后续 design 深化。
