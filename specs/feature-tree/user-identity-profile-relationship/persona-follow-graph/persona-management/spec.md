# L3 Scenario: persona-management

## 节点定位

- `L1_capability`: `user-identity-profile-relationship`
- `L2_journey`: `persona-follow-graph`
- `L3_scenario`: `persona-management`

本场景冻结“用户及分身”模型下的分身管理台基线。产品与接口统一使用“用户 / 分身 / 主分身 / 用户号 / 用户ID”语义，不再使用账号、主账号、子账号。 本场景只负责分身管理面的创建、切换、继承/同步建议、停用/删除保护、配额和管理入口，不负责公开主页壳层与跨域透传实现。

## 背景与动机

当前分身管理在主链路已基本打通后，仍有三类必须冻结的生命周期语义：

1. **退役语义**：`retire` 不能停留在 guard 视图，必须把分身写入真实 `retired` 状态，并阻断后续编辑、激活与同步。
2. **记录归因语义**：`hasAttributedHistory` 不能写死，必须由真实分身级归因来源决定 delete vs retire 分流。
3. **删除语义**：`delete-empty` 必须只对空白分身生效，不能再以“泛删除”名义绕过有记录分身的保留要求。

本场景的目标不是设计“作者主页”，而是设计“一个用户如何安全、低打扰地管理多个分身”。

## 目标用户

- 需要维护多个创作或社交身份的重度用户。
- 只有一个默认身份，但未来可能扩展更多分身的普通用户。
- 需要审计用户与分身映射、但又不能把该映射暴露给外部用户的平台治理团队。

## 功能范围

### F1. 统一分身管理入口

- 入口一：`我的主页` 动作区中的 `分身管理`。
- 入口二：设置页中的 `分身管理` 次入口。
- 单分身用户默认不被强提示打扰，但在上述两处始终能找到管理入口。
- 管理台是用户私有页面，不允许通过公开路由直接访问。

### F2. 分身列表与状态卡

管理台列表至少展示：

- 当前激活分身
- 主分身
- 分身显示名
- 用户号
- 手机号 / 邮箱同步状态
- 隔离等级
- 是否继承主分身资料基线
- 当前数量与配额上限

管理台可以展示面向当前用户的管理摘要，但这些信息不属于公开资料字段：

- 用途备注 `purposeHint`
- 最近激活时间
- 是否已有记录内容/关系，决定删除语义

### F3. 创建分身与默认继承

创建分身的第一版冻结规则：

- 默认上限：每个用户最多 5 个分身。
- 创建时最少采集：
  - `displayName`
  - `userHandle`
  - `isolationLevel`
  - 可选 `purposeHint`
- 新分身默认继承主分身的公开资料基线，以及当前 `phone / email`。
- 创建完成后，允许用户保留继承态，或进入资料编辑页改写继承值。
- 创建成功后，系统允许用户选择：
  - 立即切换到新分身
  - 保持当前 active persona 不变，并进入该分身资料编辑入口

### F4. 分身资料同步建议

- 当某个分身修改 `displayName / userHandle / phone / email` 时，系统可建议：
  - 同步到全部其它分身
  - 同步到指定分身
  - 仅保留当前分身修改
- 其它分身进入管理台时，应能看到“可同步更新”的待处理提示。
- 同步建议属于用户私有管理状态，不属于公开资料字段。

### F5. 激活与切换

- 同一用户同时只能有 1 个 active persona。
- 切换必须是强一致操作，不允许出现前端已切换、下游主体仍旧是旧分身的中间态。
- 切换成功后，本场景只负责更新 active persona 真相源；下游透传由 `persona-context-propagation` 负责。
- 切换失败时，UI 必须明确展示“仍停留在原分身”，不允许出现假成功。

### F6. 删除与停用语义

用户可见动作保留“删除分身”，但领域语义区分两类：

- **空白分身**：没有内容、评论、聊天、圈子、邀请等记录归因时，可直接物理删除。
- **有记录归因的分身**：第一版按“停用/退役”处理，禁止继续作为新动作主体，但保留记录归因与审计链。

强保护规则：

- 主分身不可删除。
- 最后一个剩余分身不可删除。
- 正在激活的分身若执行停用，必须先切换到其他可用分身。

### F7. 失败恢复与用户可控性

- 创建失败不产生半成品分身。
- 切换失败不改变当前 active persona。
- 删除/停用失败不影响记录归因和列表状态。
- 用户可在管理台中看到当前配额占用、不可删除原因、同步建议和恢复建议。

## 领域边界

### 本场景负责

- `user` 域的分身列表、创建、切换、停用/删除保护、配额和管理入口
- 用户可见但不对外暴露的管理字段，如 `purposeHint`
- `displayName / userHandle / phone / email` 的继承与同步建议管理
- active persona 真相源的用户私有管理操作

### 本场景不负责

- 分身公开资料、继承/覆写、公开可见性：归 `persona-profile-subject-and-visibility`
- 发帖、评论、聊天、圈子、助手上下文切换：归 `persona-context-propagation`
- 主页壳层、滚动与视觉规格：归 `profile-homepage-redesign`

## 权限边界与数据生命周期

- 只有当前用户可创建、切换、停用或删除自己的分身。
- 管理台可以看到 `userId -> persona` 映射；普通读接口不可见。
- 删除/停用后不得把记录内容、评论、消息重绑到 `userId` 或其它分身。
- 记录归因的保留策略由 user 域主导，具体公开读取策略由 `persona-profile-subject-and-visibility` 定义。

## 不做什么（Out of Scope）

- 公开资料页的视觉与交互。
- 下游 domain 的透传细节。
- 粉丝/关注图谱分页与读接口。
- 合规层面的物理清除与客服申诉流。

## 对标输入

| 对标 | 吸收点 |
|---|---|
| 微信 | 登录容器与应用身份分离，管理台不干扰单用户单分身路径 |
| 小红书 | 分身作为不同创作语境的身份入口 |
| 微博 | 运营身份与公开社交身份可分离 |

## 非功能目标

- 管理台首屏列表加载 P95 < 800ms。
- 分身创建、切换、停用/删除保护主路径 P95 < 1.5s。
- 因管理台操作导致的 active persona 串号事故为 `0`。

## 验收重点

1. 用户只能看到一个统一的分身管理台，不再在 mock 页与真实页之间切换。
2. 用户可以完成创建、切换、继承/同步建议、停用/删除保护和配额管理，且单分身路径不被打扰。
3. 删除语义对“空白分身”和“有记录归因分身”区分清楚，不破坏记录数据归因。
