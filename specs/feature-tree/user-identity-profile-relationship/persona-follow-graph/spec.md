# L2 特性：persona-follow-graph

## 说明
- 该节点沿用历史路径名 `persona-follow-graph`，但本次 PRD 基线将其语义正式升级为“子账号隔离与关系网络”。
- 后续 `/design` 与 metadata 拆分时，如需将 `persona` 重命名为 `SubAccount` 相关对象，以本节点定义的业务边界为准。

## 功能说明
- 负责 `SubAccount` 的创建、切换、停用、删除、激活与展示边界。
- 负责好友、关注、群、圈子、评论发言、私信主体等关系网络按子账号独立建立。
- 负责保证多个子账号彼此隔离，对外默认无法从社交关系、内容行为、圈层参与或邀请行为推断它们隶属于同一主控账号。

## 目标用户
- 有匿名表达需求，但不希望暴露职业身份或生活身份的用户。
- 需要把职业沟通、私人社交、运营职责、圈主管理等使用场景隔离的重度用户。
- 仅有一个子账号，但未来可能逐步扩展多个身份的普通用户。

## 关键用户故事
- 用户可以创建多个子账号，并为其自行定义用途，不受平台预设角色限制。
- 用户可以选择用某个子账号发评论、进群、加好友、建立圈子关系或承担运营职责。
- 外部用户只看到当前子账号，不会被默认告知它与其他子账号存在主控关系。
- 不同子账号的好友、关注、圈子、群聊、主页、获赞、互动、邀请扩散可以完全不同。

## 范围
- 子账号生命周期：创建、编辑、切换、激活、停用、删除。
- 子账号关系网络：好友、关注、圈子、群、评论、发帖、私信主体。
- 子账号隔离：主页、列表、推荐、关系、通知、搜索与展示层的隔离边界。
- 通讯录匹配后的关系落地：从 `OwnerAccount` 匹配结果转为某个子账号的真实社交连接。

## 约束
- 契约与字段策略必须与 OpenAPI 与 metadata 保持一致。
- 子账号不预设为“公开账号”；是否公开、私密、圈内可见、仅好友可见由产品策略与用户设置决定。
- `OwnerAccount` 只负责管理、切换、恢复与通讯录匹配，不应成为默认的好友、关注、圈子、评论或邀请主体。
- 任一好友、圈子、群、关注、评论、内容、私信关系都必须落到具体 `SubAccount`。
- 默认情况下，外部世界不可通过共同联系人、推荐、圈子列表、评论记录、邀请记录等间接推断跨子账号关联。
- 子账号切换必须把推荐、聊天、评论、圈子、通知、助手、邀请等上下文一起切换。
- 平台内部审计与风控可追踪 `OwnerAccount -> SubAccount` 归属，但普通业务接口不得泄露该映射。

## 体验要求
- 单子账号用户不应被复杂的多账号心智打扰。
- 多子账号用户必须有明确、低成本、安全的切换入口与上下文提示。
- 评论/发帖/加好友/进圈/邀请等动作需要在合适场景支持选择当前子账号。
- 关系建立失败、切换失败、删除失败时应有清晰恢复与兜底方案，避免串号。

## 对标输入
- 微信：强关系链路与工作/生活身份冲突痛点。
- 小红书：人格化表达、主页经营与评论身份感知。
- 微博：公开社交、运营账号、粉丝关系与身份扩散。

## 非功能目标
- 子账号切换到新身份上下文在端侧 P95 小于 1 秒。
- 子账号切换后，评论/私信/圈子/邀请主体不得发生串号。
- 关系网络建立、删除、切换、分页读取都需具备一致性修复与审计能力。

## 不做什么（Out of Scope）
- 不在本节点定义手机号、微信、Apple 登录绑定本身。
- 不在本节点展开增长奖励算法或控制面工作台。
- 不在本节点设计完整恢复工单与客服处置细节。

## 验收标准
- A1：用户可以创建、切换和管理多个子账号，且单子账号路径依旧简单。
- A2：好友、关注、圈子、群、评论、私信、主页与互动数据都按子账号独立归属。
- A3：外部默认无法从应用界面与常规业务接口推断多个子账号归属于同一主控账号。
- A4：通讯录匹配结果来自 `OwnerAccount`，但转化为真实关系时必须显式落到具体子账号。
- A5：评论、发帖、加好友、进圈、邀请等关键动作可以明确绑定当前子账号，不发生串号。
- A6：平台审计与风控能追踪主控与子账号的映射，但普通读接口遵循隔离与最小暴露。
- A7：子账号、关系网络、隔离规则相关契约与 metadata 一致。
- A8：对应自动化测试覆盖子账号生命周期、关系网络分页、切换透传、隔离验证与异常恢复。
