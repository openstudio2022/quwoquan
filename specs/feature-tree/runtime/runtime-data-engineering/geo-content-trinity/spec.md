# L3 规格：geo-content-trinity（实体–标签–图文数据工程）

## 背景与动机

当前 `quwoquan_data` 已具备部分 crawl 能力，但数据工程主线仍存在三个根本问题：

1. **目录层、实体层、标签层、图文层边界不清**：早期实现把“地理目录”“实体”“标签”“图文发现”混在同一口径里，导致字段、命名、验收与门禁持续漂移。
2. **缺少独立的数据工程运行工作流**：现有 `crawl *` 子命令可用，但没有一组显式命名的阶段命令与阶段准出，难以复用到四川、全国与海外不同规模的数据生产。
3. **四川旅游出行样例暴露出质量缺口**：目录中出现非景点实体、英文/符号名、行政切片异常（如广安市 0 命中）、`wiki_expand=full` 回归等，说明仅有脚本不等于可治理的数据工程系统。

本特性的目标，是在**不修改产品研发主线**的前提下，为 `quwoquan_data` 建立一条独立、可审计、可门禁的**应用数据生成工作流**，统一覆盖：

- 地理目录候选层
- 实体层
- 标签层
- 图文发现 / 下载 / 加工 / 发布层
- 权威锚点与真实性门禁

配套文档：

- `[workflow.md](workflow.md)` — 数据工程运行工作流
- `[command-matrix.md](command-matrix.md)` — `data-`* 阶段命令与内部 `crawl` 映射
- `config/geo_catalog_config.sichuan.yaml`
- `config/entity_naming_rules.yaml`
- `config/geo_band_rules.sichuan.yaml`

## 目标用户

- **数据工程执行者**：需要一组显式阶段命令、稳定输入输出与阶段门禁，能够重复生产四川及后续省域/海外数据。
- **数据规格维护者**：需要 `spec.md / design.md / acceptance.yaml / schema / config` 的唯一真相源，避免目录、实体、标签、图文定义分裂。
- **AI Agent / 编程助手**：需要区分“研发工作流”和“数据运行工作流”，避免把 `prd→design→dev` 与 `crawl` 执行混为一谈。
- **质量与验收负责人**：需要 A1~A14 这类可量化验收项、明确脚本门禁与抽样策略。

## 功能范围

### R1：定义独立的应用数据生成工作流

对外使用显式 `data-`* 阶段命令，内部映射到现有 `crawl` 子命令与验证脚本：

- `data explore`
- `data baseline`
- `data build-entities-tags`
- `data download`
- `data process-content`
- `data publish`

该工作流与产品研发 `explore → prd → design → dev → verify → commit` **分离**：

- 研发主线负责**定义和演进**数据工程能力
- 数据工作流负责**运行并产出数据**

### R2：冻结目录层、实体层、标签层、图文层的边界

- `catalog.ndjson` 是**地理目录候选层**，不是实体类型；候选层允许保留成员点、别名候选、并列实体候选与待审点。
- 每条 `catalog` 候选必须先获得语义归宿：`standalone` / `member` / `alias` / `parallel_entity` / `reject` / `pending_review`。
- `entity_catalog` 是规范化实体层，承载 `entityId`、canonical name、类型、权威锚点；仅 `standalone` 与显式 `parallel_entity` 可作为 publishable 顶层实体。
- `tag_catalog` 是共享分类维度，不能代替实体级 `label_zh`。
- `source_pool / publish` 是图文与包层，必须与实体锚点同源可追溯。

### R3：建立中文优先的命名与 label 规则

- **国内与可中文化场景**：`label_zh` 必保。
- `label_en` 为扩展批次，非默认 Gate0。
- 海外无通用中文译名时允许英文主展示，但必须显式打 `display_locale=en`。
- 中文业务类型为统一口径；内部可保留 `_kind` 作为技术映射字段，但不得取代中文业务类型。

### R4：建立地理目录候选层的质量约束

- 省 / 市州 / 县切片结果必须可出 `raw/kept/reject` 报告。
- 无权威或名称不合规的候选行，可在目录层直接剔除。
- `wiki_expand` 等关键参数必须有回归用例，避免 filtered / full 行为漂移。

### R5：建立实体–标签–图文三角一致性

- `entity-catalog-build` 后，抽样 `topic_id` 必须能回链到 catalog。
- `semantic_cluster_candidates.ndjson` / `semantic_cluster_pending.ndjson` 必须与 `catalog` 同源，能解释每条候选为何进入顶层实体、成员、并列实体、剔除或待审。
- `tagRefs` 必须能解析到 `tag_catalog`。
- 图文标题、snippet、正文锚点必须能命中 canonical name 或 `label_zh`。
- 内容侧发现的新名称/别名/行政信息应输出 diff 提案，反向修订目录/实体。

### R6：四川旅游出行端到端验证作为首个运行样例

四川（尤其川西）作为首个真实样例，必须至少证明：

- 目录候选层可生成并过滤明显脏数据
- 实体与标签可从目录候选层构建
- 图文来源可下载、审核、加工、发布
- authenticity / package gate 可通过

### R7：四川省域「景点 + 打卡地 / 观景台 / 小众秘境」与双轨实体准入（本期需求冻结）

**地理与类型范围（目录候选与实体均应能表达，具体技术映射见 `design.md`）**：

- 行政区划锚定 **四川省**（省→市州→县切片策略沿用 `geo_catalog_config`）。
- **传统观光 POI**：旅游景点、自然景观、人文史迹、文博场馆、风景名胜区、国家公园、世界遗产（境内点位）等。
- **轻量/社交向点位**：**打卡地**、**观景台**（与 OSM `tourism=viewpoint` 等可对齐的部分）、**小众秘境**（含尚未商业开发、野生步道可达描述等）；此类点位在百科/维基上**常常无独立词条**。

**权威与图文证据的双轨准入（写入验收 A15～A17）**：

1. **权威轨（直接采纳）**  
   存在合规可用的 **百度百科词条** 与/或 **维基百科**（含延伸：如 Wikivoyage 等在 `design` 中显式列出的白名单）作为命名与锚点来源时：**直接采纳**该标准命名与主页 URL 进入实体层，**不强制**要求额外 post 篇数作为准入条件。  
   若两端均有，以中文用户可读与内部一致性规则做**单主锚 + 互链**（见 `entity_naming_rules` / `authority_pool` 约定）。

2. **图文证据轨（post 互证）**  
   当 **无**上述权威主锚（或权威不足以唯一确定实体）时，仅允许从**可合规抓取/引用**的攻略、游记、日记、摄影向图文中收敛新实体；**准入硬条件**：  
   - **不少于 2 篇相互独立的文章级来源**均明确提及**同一地点**；  
   - 对 **主展示名** 与 **地理归属/相对位置** 的描述须 **无实质冲突**（细则见 `design.md`「冲突判定」）；  
   - 不满足则：**不得**写入 `entity_catalog` 可发布行，至多保留为目录候选 / `source_pool` 聚类待审。  
   「独立」指不同 `sourceUrl`（或同站不同稿件且非明显转载镜像—细则见 design）；转载/洗稿归并规则在加工阶段降权或合并计数。

**图文规模与精品比例（目标型 KPI，非单批次强 Gate0）**：

- 对已准入实体（尤其权威轨）：从权威与**白名单内容平台**发现、下载 **攻略 / 游玩 / 日记** 及 **摄影向图文**，目标 **每实体 20～100 篇**，在合规与平台可得性允许下**尽量多采**。  
- **约 30%** 进入「精品 / 高优发布」子集（`curated` 或等价 tier），具体**打分或人工抽检口径**在 `acceptance` 与加工流水线中冻结，避免只写比例无操作定义。

**诚实边界（与 Out of Scope 不矛盾）**：

- 不承诺全省每一个野生点位均被 OSM 或百科预先枚举；**未开发秘境**主要依赖图文证据轨 + 人工审核。  
- 「≥2 篇互证」的**全自动冲突检测**可分阶段落地：基线要求 **结构化记录证据 URL + 可人工/抽检裁决**；自动化规则迭代不得绕过合规与 TOS。

**R7 补充：语义归并不是文本去重**：

- `member`：明确属于某个主实体的成员点，只能进入根实体 `extensions.members`，不得平铺成顶层实体。
- `alias`：仅作为根实体别名存在，不得再生成第二条顶层实体。
- `parallel_entity`：与候选根实体主题相关但不构成从属，允许保留为独立顶层实体，必须有显式语义判定记录。
- `reject`：名称/类型/证据明显不成立，直接从 publishable 实体层剔除。
- `pending_review`：有聚类/从属信号但证据不足或缺主实体；默认待审，严禁为追求“目录干净”而静默强合并。

## 不做什么（Out of Scope）

- OS1：不修改产品研发 `00_MASTER_DEVELOPMENT_FLOW.md` 主线。
- OS2：不承诺一键零人工覆盖全球所有国家 / 行政层级。
- OS3：不在本轮把 `label_en` 做成与 `label_zh` 对等的强门禁。
- OS4：不绕过第三方站点的 robots / TOS / 权限限制。

## 对标输入与吸收结论

### 对标对象

- 仓库现有 `spec-first / acceptance-first / design-first` 研发主线
- `quwoquan_data` 既有 hybrid crawl 链路

### 借鉴点

- 借鉴研发主线的**阶段治理**与**验收前置**方法。
- 借鉴 `crawl` 现有子命令作为内部执行原语，而不是推翻重做。

### 不借鉴点

- 不把数据运行阶段直接混入研发 `prd / design / dev` 命名。
- 不再用抽象 D0~D8 代号作为对外阶段语言。

## 非功能目标

- 所有关键产物具备 schemaVersion 与可追溯路径。
- 自测清单明确分阶段，不允许静默跳过失败阶段。
- 关键四川样例可在受控范围内重跑，并产出验收证据。
