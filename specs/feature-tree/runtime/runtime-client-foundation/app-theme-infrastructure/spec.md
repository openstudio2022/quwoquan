# L3 规格：app-theme-infrastructure — Cupertino-first 全局视觉运行时

> **层级**：L3_story（隶属 L2 `runtime-client-foundation`）
> **状态**：specified

## 0. 一句话定义

建立以 `Apple HIG / Cupertino-first` 为基线的全 app 视觉运行时，统一主题、字体、图标、表面、状态栏、安全区与多设备适配，并为主题/字号设置提供全局应用能力。

## 1. 背景与动机

当前 App 的视觉体验存在系统性断层：

1. **视觉语法混杂**：Material、Cupertino、自定义样式、旧的 `ScreenUtil` 适配方式并存，导致字体层级、图标笔触、分割线、背景表面和动效反馈不统一。
2. **主题接线不完整**：虽然已有 `AppTheme`、`AppColors`、`AppTypography`、`AppSpacing` 雏形，但未形成真正的全局主题运行时，深浅色、状态栏、沉浸式媒体页仍大量依赖页面内手写分支。
3. **字号与可访问性缺失**：当前未形成真正的 Dynamic Type / App 内字号档位 / 高对比与 Safe Area 协同能力，无法支撑“精品感 + 清晰度 + 多端一致性”。
4. **设备矩阵无统一策略**：手机高低分辨率、刘海/挖孔、平板、横屏、分屏、桌面窗口缺乏统一布局语义，记录页面仍存在固定高宽、百分比布局、局部 overflow 风险。

本 Story 负责把“视觉基础设施”从零散 token 升级为真正的运行时能力；账号级偏好同步由兄弟 Story `appearance-accessibility-settings` 负责。

## 2. 目标用户

- 所有终端用户：在发现、内容、聊天、圈子、个人页、设置、创作等全量页面获得统一、简洁、清晰、苹果风格的视觉体验。
- 多设备用户：在手机、平板、横屏、分屏、桌面窗口下获得一致且稳定的视觉秩序。
- 设计、开发、测试团队：拥有单一真相源的主题/排版/图标/适配规则，避免后续继续产生新的视觉漂移。

## 3. 功能范围

### 3.1 In-Scope

| 编号 | 功能 | 说明 |
|---|---|---|
| UI1 | Cupertino-first 主题运行时 | 建立 `ThemeData + CupertinoThemeData + semantic tokens` 的统一运行时，以 Apple HIG 为默认视觉语法 |
| UI2 | 全局主题模式 | 支持 `system / light / dark` 三种主题模式，并统一状态栏、导航栏、弹层、沉浸式媒体页的明暗策略 |
| UI3 | 统一设计 token | 统一颜色、字体、字号梯度、行高、图标语义、圆角、边框、阴影、表面层级、触控热区 |
| UI4 | 全局组件子主题 | 统一 App Shell、Navigation Bar、Tab、List、Section、Sheet、Dialog、Input、Button、Card、Chat Input 等高频共享组件语法 |
| UI5 | 字号与可访问性应用能力 | 建立 Dynamic Type / App 内字号档位在运行时的应用链路，并为高对比、粗体、减弱动效预留标准接入点 |
| UI6 | 多设备适配 | 统一 `compact / regular / expanded` 断点与布局语义，覆盖手机高低分辨率、刘海/挖孔、安全区差异、平板、横屏、分屏、桌面 |
| UI7 | 系统栏与沉浸式策略 | 统一 status bar / navigation bar / immersive media viewer 的表面与图标对比策略，避免深色界面仍使用错误亮度 |
| UI8 | 全量页面推广基线 | 本次首发范围覆盖全量页面；业务域页面需消费统一主题运行时而非继续自建视觉分支 |

### 3.2 Out-of-Scope

- `owner / sub account` 外观偏好的持久化、继承、同步、审计与冲突处理（由 `appearance-accessibility-settings` 承担）
- 业务信息架构调整、功能入口增删、业务流程重构
- 非 App 主体范围的品牌官网、活动页、外部 H5 视觉体系
- 与主题无关的文案、推荐算法、业务数据契约变更

## 4. 对标输入与吸收结论

| 输入 | 借鉴点 | 不借鉴点 | 本次吸收结论 |
|---|---|---|---|
| Apple HIG | Navigation / Sheet / Grouped List / Dynamic Type / Safe Area / 深浅色 / 触控热区 / 动效克制 | 过度依赖苹果原生限定能力、不适合内容消费产品的过强系统感 | 作为第一视觉基线与交互语法硬约束 |
| Instagram | Chrome 克制、内容优先、留白节奏、全局视觉秩序稳定 | 对创作者与复杂设置页支持较弱 | 用于发现流、沉浸式内容和全局“轻 UI”感 |
| 小红书 | 图文卡片阅读性、信息层级、留白与列表节奏 | 局部卡片密度偏高、部分页面视觉噪声较多 | 用于图文 feed、详情、评论阅读层级 |
| 微信 | 列表、聊天、设置、输入区的一致性与低认知负担 | Android/微信式视觉语义、绿色品牌语义不沿用 | 用于高频工具页的稳定密度和低学习成本 |
| 抖音 | 沉浸式视频页、深色表面控制、边缘控件压缩能力 | 激进动效、过度抢注意力的视觉风格 | 仅用于沉浸式媒体页与全屏观看场景 |

**总原则**：`视觉语法以 Apple HIG 为主，内容消费效率吸收 Instagram / 小红书 / 微信 / 抖音的成熟模式，但不做拼盘式混搭。`

## 5. 约束

- 必须坚持 `Cupertino-first`：Material 仅用于兼容和平台必要能力，不得成为主要视觉语法来源。
- 视觉真相源必须收敛到统一主题运行时：禁止页面继续以 `Colors.*`、裸 `TextStyle`、裸 `Icon`、局部 `isDark` 分支维护第二套规则。
- 全量页面必须纳入本 Story 推广范围，禁止只修壳层或只修单域页面后宣布完成。
- 路由、surface、operation 等业务标识仍必须来自 metadata codegen，主题系统不得引入第二套业务配置表。
- 必须与 `page-layout-semantics`、`dart-semantic-gate` 保持一致，并把苹果风格约束沉淀为长期门禁。
- 任何可交互元素必须满足最小热区 `44x44`，主操作建议 `48x48`。

## 6. 角色分工

- 产品：冻结“精品、简洁、清晰”的体验目标，确认全量页面首发范围与 benchmark 吸收边界。
- 架构/设计系统：定义 Cupertino-first 主题运行时、token 分层、断点语义、组件子主题边界。
- 开发：按 `App Shell -> 共享组件 -> 各业务域页面` 的顺序实施，消除旧的视觉分支与适配方式。
- 测试：建立全量页面、全设备矩阵、深浅色与字号档位的回归证据。
- 发布：以统一生效、简单灰度为主，保障可回滚。

## 7. 非功能目标

- 当前账号本地外观应用体感即时：`<= 100ms`
- 全 app 主题/字号切换稳定完成：`<= 300ms`，且**不闪屏、不出现明暗错帧**
- 全量页面在手机、平板、横屏、分屏、桌面窗口下无系统性 overflow / 裁切 / 不可点击区域
- 深浅色下文本与关键图标满足可读性目标，普通文本对比度满足 `AA`
- 沉浸式媒体页切换状态栏/导航栏时无错误明暗反转
- 共享主题运行时必须支持后续账号同步与离线补同步接入，但本 Story 不直接承载同步逻辑

## 8. 验收重点

### T1

- 主题运行时、token、断点、系统栏策略存在单一真相源
- 语义门禁能够拦截新增的硬编码视觉字面量与 Android-first 语法回流

### T2

- 全量页面在 `system / light / dark` 与不同字号档位下保持苹果风格一致
- 高价值页面（发现、聊天、个人页、设置、创作、沉浸式媒体页）视觉语法统一，无明显 Material-first 残留

### T3

- 主题与字号切换能跨页面、跨导航栈稳定传播
- 与账号偏好设置 Story 对接后，运行时应用链路稳定，不出现“设置值正确但 UI 不更新”的断裂

### T4

- 真机覆盖刘海/非刘海、高低分辨率手机、平板横竖屏、分屏、桌面窗口时，视觉层级、Safe Area、状态栏、沉浸式页面均符合苹果风格基线
