# 设计方案：development-workflow-governance

## 设计动因

本特性不是单一功能设计，而是对整个 SDD 研发治理系统做一致性重构。当前仓库的核心问题不是“缺少规则”，而是“规则、命令、特性树、验收与测试口径彼此错位”：

- `/explore`、`/prd`、`/design`、`/dev` 已存在，但阶段职责仍偏模板化，缺少批判性澄清、对标吸收和上游阻断机制。
- 特性树希望收敛层级，但存量语义仍以 `L3_component / L4_detail / L5_*` 为主，和目标中的“关键能力 / 功能特性 / 子功能或组件 / Story”不一致。
- `tasks.md` 既像树层级又像工程清单，造成“交付单元”与“实施步骤”混淆。
- `.cursor/rules/03-testing.mdc` 已经有较完整的测试金字塔，但产品规格、设计、验收和门禁并没有统一引用同一测试视图。
- 当用户强调“业界一流标准”时，流程缺少系统性的对标输入机制，无法稳定吸收优秀产品、原型、公开代码和公开技术方案。

设计目标不是再加一层新规则，而是建立一套单一叙事：

1. 用户在 `/explore` 阶段能被正确引导说出问题、约束与标杆。
2. `/prd` 能把模糊需求变成高质量规格和四层验收基线。
3. `/design` 能评审上游质量是否足以支撑方案和开发。
4. `/dev` 能以 Story 为单位交付，而不是被任务清单牵着走。
5. `/try` 能被纳入同一治理模型，在快速原型验证时也不脱离核心工程约束。
6. `acceptance.yaml` 能成为“特性树 + 四层测试 + 证据追踪”的唯一真相源。

## 业界对标输入与借鉴

### 对标主题 1：高质量需求探索与规格产出

**典型标杆**：
- Linear / Notion 式问题定义与范围控制
- Amazon PRFAQ / Narrative 式先写清用户价值与成功标准
- Shape Up 式 problem framing、appetite、风险前置

**可借鉴点**：
- 在产出规格前先对问题定义做批判性审查，而不是直接写方案。
- 要求明确“目标用户、核心问题、成功标准、边界和不做什么”。
- 把风险、未知项、假设提前显式化，避免进入设计和开发后补洞。

**不直接照搬的点**：
- 不引入完整 PRFAQ/Narrative 长文模板，避免仓库文档过重。
- 不将 Shape Up 的时间盒完全引入当前 SDD 主线，以免与现有 gate 模型冲突。

### 对标主题 2：业界优秀产品与原型吸收机制

**典型标杆**：
- Figma / Framer / 原型驱动的交互评审流程
- OpenAI / Cursor / Vercel 等文档中“给参考输入，得到更高质量输出”的实践
- 开源框架中通过 RFC/ADR 吸收外部模式的做法

**可借鉴点**：
- 主动引导用户提供标杆产品、原型、视频、公开代码与公开架构文档。
- 每个外部参考都要落成“借鉴点 / 不借鉴点 / 适用边界 / 成本”。
- 对标不只是 UX，也包括协议设计、架构设计、测试治理和交付流程。

**不直接照搬的点**：
- 不要求用户必须有完整原型或代码参考才能进入流程。
- 不把外部实现直接复制进项目；只吸收公开可审计的机制。

### 对标主题 3：Story 驱动交付与可追踪验收

**典型标杆**：
- Jira / Shortcut / Azure DevOps 中 Story 与 Task 分层
- BDD / ATDD 中“验收驱动开发”
- 合规工程中的 requirements traceability matrix

**可借鉴点**：
- 把 Story 作为最小交付单元，把 task 作为 Story 的实施步骤。
- 验收项必须有测试证据，不能只有“完成了”的口头判断。
- 验收、测试、设计和实现要能双向追溯。

**不直接照搬的点**：
- 不引入独立 RTM 工具或第五类文档，避免脱离特性树四件套。
- 不引入过于繁重的 issue workflow，仍以内仓 spec/design/tasks/acceptance 为准。

### 对标主题 4：原型先行再基线化

**典型标杆**：
- 产品和技术团队中常见的 spike / prototype / RFC spike 流程
- “先做小验证，再决定是否正式立项”的工程化试验机制

**可借鉴点**：
- 原型阶段可以豁免完整规格拆解，但不能豁免核心工程约束。
- 试验目标、成功标准、退出条件必须在原型启动时说清楚。
- 原型一旦被证明可行，就要有正式回补路径，而不是长期停在实验态。

**不直接照搬的点**：
- 不引入独立的 spike 文档体系。
- 不把 `/try` 变成绕过 metadata-first、DDD、codegen 的捷径。

## 方案对比

### 方案 A：最小变更方案

只在现有命令文案中追加几条说明，不改变特性树语义，不重构验收模型。

**优点**：
- 成本低，改动范围小。
- 对现有存量节点和脚本最友好。

**缺点**：
- 解决不了“任务、Story、层级、验收、测试”概念混用的问题。
- 对标输入机制只能是提示语，难形成门禁闭环。
- 后续 `/dev` 仍会被旧的 task 叙事牵制。

**适用条件**：
- 只想修正文案，不追求治理模型真正统一。

### 方案 B：统一治理模型方案（本期选型）

在保持四件套框架不变的前提下，统一更新命令、主流程、特性树标准、层级定义、测试规则与目标节点设计文档，形成一套新的治理口径：

- 树上统一到“关键能力 / 功能特性 / 子功能或组件 / Story”
- `tasks.md` 保留但只作为实施清单
- `acceptance.yaml` 强绑定四层测试
- `explore/prd/design/dev` 明确加入批判性澄清、对标输入与上游阻断
- 对历史 `L5_*` 做兼容，不要求一次性全量迁移

**优点**：
- 能同时解决命令、树、验收、测试的多点错位问题。
- 可分阶段落地，不要求立刻重写所有存量节点。
- 与现有四件套和 gate 体系兼容性较好。

**缺点**：
- 需要同时修改多份关键文档，范围较大。
- 仍需后续 `/dev` 阶段配合把脚本和门禁真正固化。

**适用条件**：
- 需要稳定治理模型，但接受先以文档和流程基线推进，再逐步脚本化。

### 方案 C：一次性全面迁移方案

在本次设计阶段就同步重写所有命令、规则、脚本、存量特性树层级和所有 acceptance 模板，并立即移除 L5 历史兼容。

**优点**：
- 理论上口径最统一。
- 无需长期兼容层。

**缺点**：
- 变更面过大，风险极高。
- 会与大量存量节点、脚本、门禁形成直接冲突。
- 设计阶段难以保证所有执行脚本同步跟上。

**适用条件**：
- 仅适用于新仓或冻结开发窗口的集中重构场景；当前仓库不适合。

## 选型决策

**选定方案**：方案 B，统一治理模型方案。

**理由**：
- 它能解决本次最核心的问题：命令、特性树、验收、测试语言不统一。
- 它保留四件套与现有主线，不会把仓库带入一次性大爆炸迁移。
- 它允许先完成“规格与流程统一”，再在后续 `/dev` 中继续固化脚本和门禁。

## 适用场景与约束

### 适用场景

- 适用于全仓 SDD 主流程治理。
- 适用于后续所有新增特性节点。
- 特别适用于个人私人助理、runtime、跨端云能力等复杂度高、需要强验收和强追溯的特性。

### 约束与局限性

- 当前仓库已有大量 `L3_component / L4_detail / L5_*` 节点，短期内必须兼容读取。
- 现有 gate 与脚本仍以旧层级和旧测试桶为主，本次只能先统一设计口径，后续再逐步把脚本升级为新口径。
- 四层测试视图与执行桶不是完全一一对应，本次设计采用“规格视图统一、执行桶兼容保留”的策略。

## 核心设计

### 1. 特性树统一为四层治理视图

目标视图：

1. `L1`：关键能力
2. `L2_feature`：功能特性
3. `L3_subfeature`：子功能或组件
4. `L4_story`：最小交付 Story

设计原则：
- 树上交付单位统一落到 `Story`，而不是 `task`。
- `tasks.md` 继续存在，但不再代表树层级。
- 历史 `L5` 只保留兼容语义，不再作为新增默认层级。

### 2. 四层测试视图统一

规格与验收层统一使用：

1. `T1`：契约与静态层
2. `T2`：模块与交互层
3. `T3`：端云集成层
4. `T4`：端到端旅程层

兼容策略：
- 保留 `.cursor/rules/03-testing.mdc` 当前执行桶 `L1a/L1b/L1c/L2/L3/L4`
- 在文档中新增“视图映射”：
  - `T1` ≈ `L1a + 静态校验`
  - `T2` ≈ `L1b + 部分 L1c`
  - `T3` ≈ `L2 + L3`
  - `T4` ≈ `L1c 主旅程 + Patrol L4`

### 3. 三阶段对标输入机制

#### `/explore`
- 主动追问用户是否有标杆产品、原型、截图、代码、公开文档
- 没有时提供建议维度，不强行指定
- 输出“借鉴方向假设”，而非直接定方案

#### `/prd`
- 将外部参考整理成“借鉴点 / 不借鉴点 / 适用条件 / 成本”
- 吸收到 `spec.md` 的目标、范围、验收重点

#### `/design`
- 将对标对象变成方案对比依据
- 明确“为什么借鉴”“为什么不完全照搬”“当前差距与演进路径”

### 4. 命令职责重构

#### `/explore`
- 不是自由聊天，而是高质量问题建模
- 必须输出 `EXPLORE_READY` 或 `GATE_BLOCK`

#### `/prd`
- 不是仅写文档模板，而是做规格准入评审
- 必须检查需求是否足够清晰、可验收、可分层、可测试

#### `/design`
- 不是仅给方案，而是评审“这个规格能否支撑方案和开发”
- 必须阻断需求不清、契约缺失、测试无法落地的情况

#### `/dev`
- 不是按 task 清单机械推进，而是按 Story 交付
- 每个 Story 都要有用户价值、边界、影响范围和测试计划

#### `/try`
- 是标准 SDD 主链之外的快速验证通道
- 仅豁免特性树四件套前置创建，不豁免核心工程约束
- 成功后必须通过 `/land` 吸收到标准链路

### 5. acceptance.yaml 成为唯一验收真相源

每个验收项除 `criteria / status / linked_tasks / tests` 外，还要具备测试层视图：

```yaml
test_layers:
  T1: required
  T2: required
  T3: optional
  T4: optional
```

后续演进目标：
- `implemented` 必须能校验测试证据存在
- Gate 能识别 Story 与测试层的覆盖缺口
- `/try` 成果一旦 land，也必须补齐到同一验收模型中

## 受影响文档与同步边界

本次设计要求同步更新以下文档：

- `specs/00_MASTER_DEVELOPMENT_FLOW.md`
- `specs/feature-tree/00_FEATURE_TREE_STANDARD.md`
- `specs/feature-tree/01_FEATURE_TREE_LEVEL_DEFINITIONS.md`
- `.cursor/rules/03-testing.mdc`
- `.cursor/commands/explore.md`
- `.cursor/commands/prd.md`
- `.cursor/commands/design.md`
- `.cursor/commands/dev.md`
- `.cursor/commands/try.md`
- `.cursor/commands/verify.md`
- `.cursor/commands/deliver.md`

本次不要求同步完成脚本实现，但要求这些文档先形成单一叙事。

## 关键设计决策

- 决策 1：保留四件套，不新增第五类治理文档。
- 决策 2：树的治理视图收敛为四层，L4 代表 Story，而不是 task。
- 决策 3：`tasks.md` 继续保留，但只作为实施清单。
- 决策 4：验收基线统一对齐四层测试视图 `T1~T4`。
- 决策 5：`03-testing.mdc` 保留执行桶，但必须补上与 `T1~T4` 的映射说明。
- 决策 6：`/explore`、`/prd`、`/design` 全部引入业界对标输入机制。
- 决策 7：`/try` 纳入统一治理模型，但继续保留“先实验、再基线”的独立定位。
- 决策 8：对历史层级采取兼容迁移，而不是一次性清理。

## 当前态到目标态

### 当前态

- 命令职责偏模板化
- 树层级与验收术语混用
- `tasks` 同时承担层级和执行两种语义
- 测试规则与规格验收视图没有统一
- 对标输入机制缺失
- `/try` 与标准链路的关系虽存在，但未被纳入同一治理叙事

### 目标态

- 命令具有明确准入、阻断、评审与输出模型
- 树上统一四层治理视图
- `tasks.md` 与树层级彻底解耦
- `acceptance.yaml` 成为唯一验收真相源
- 四层测试视图贯穿规格、设计、开发和门禁
- 对标输入可贯穿 explore / prd / design / try
- `/try` 成为正式受管的快速验证通道，并能通过 `/land` 回补到标准链路

## 未来演进

- 将 `L4_story`、`T1~T4` 进一步固化到 gate 脚本中，自动检查 Story 覆盖缺口。
- 为 explore / prd / design 增加统一的“对标输入模板片段”或命令内置表单。
- 提供一个真实样例特性的完整演练路径，作为后续团队复用模板。
- 分阶段迁移历史 `L5_*` 节点，最终减少新建 L5 的需求。

## 遗留带规划任务

- 历史脚本和门禁仍依赖旧层级命名，需在 `/dev` 阶段继续做兼容升级。
- 存量 acceptance 模板差异较大，需选择 1~2 个真实特性先试点，再全局铺开。
- `tree_index.yaml` 中仍大量使用旧 level 枚举，短期以内需保留映射表。
