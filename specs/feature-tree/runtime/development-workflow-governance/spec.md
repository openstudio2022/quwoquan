# L2 规格：development-workflow-governance（命令、特性树与四层验收模型）

## 背景与动机

当前仓库已经形成 `explore → prd → design → dev → commit → deploy` 的 SDD 主线（其中 `/dev` 收口四层自验证并自动归档），也已经存在特性树、四类文档、metadata-first、端云一体化测试规则等基础约束，但整体仍存在以下结构性问题：

1. **命令职责不够收敛**：`/explore`、`/prd`、`/design`、`/dev` 更像阶段说明，尚未把“批判性澄清、业界对标、测试可验证性、准入/阻断条件”固化为统一标准。
2. **特性树语义仍有混用**：现有树中同时存在 `L3_component`、`L4_detail`、`L5_leaf`、`L5_cross_cutting` 等旧命名，与目标中的“关键能力、功能特性、子功能或组件、开发 Story”不一致。
3. **`tasks` 语义有歧义**：它既被当作树层级的一部分，又被当作执行清单，导致“树节点”和“工程任务”混在一起，难以形成稳定交付单元。
4. **验收标准与测试层未真正绑定**：现有 `acceptance.yaml` 在不同节点格式不统一，很多验收项仍停留在描述层，没有明确映射到可执行的四层测试与证据。
5. **上游文档质量无法稳定传导到下游实现**：需求规格不够清晰时，设计与开发阶段缺少系统化的“上游批判性评审”，易出现 spec 漂移、设计失真、开发时补洞。
6. **对标输入机制缺失**：当用户希望产品达到业界一流水平时，现有流程没有系统性引导用户提供或共同分析“标杆产品、原型、交互稿、公开代码、公开技术方案”，导致优秀经验难以沉淀为规格和设计质量。

本特性的目标，是把“命令体系 + 特性树 + 验收标准 + 四层测试 + 标杆输入机制”统一重构为一套可复用、可审计、可门禁的研发治理模型，提升后续所有新增特性的规格质量、设计质量与实现质量。

## 目标用户

- **需求提出者 / 产品经理**：希望把模糊想法转成高质量、可落地、可验收的产品规格。
- **架构师 / 技术负责人**：希望设计阶段能识别约束、方案优劣、可演进性和测试可验证性。
- **AI Agent / 编程助手**：需要稳定、明确、可执行的阶段职责与卡点，避免跨阶段误判。
- **前端 / 云端开发者**：需要以 Story 为单位推进开发，并明确知道需要补哪些 metadata、契约、代码和测试。
- **评审者 / QA / 交付负责人**：需要从 `acceptance.yaml` 直接看到交付边界、测试证据、端云联调和端到端旅程是否闭环。

## 功能范围

### R1：重构需求探索（`/explore`）的质量模型

- R1.1：`/explore` 必须从“自由聊天”升级为“高质量问题建模”，强制覆盖问题重述、目标用户、核心痛点、边界、关键未知项、风险假设与进入 `/prd` 的就绪度判断。
- R1.2：`/explore` 必须具备批判性澄清能力，对模糊、跳步、伪需求、隐含前提进行追问，而不是直接顺着用户描述生成方案。
- R1.3：`/explore` 必须支持经典需求分析方法的适配性使用，包括但不限于：头脑风暴、5 Why、JTBD、反例分析、边界场景分析、失败路径分析。
- R1.4：`/explore` 必须引导用户提供业界一流对标输入，至少支持以下五类参考：
  - 标杆产品或竞品
  - 原型或交互稿
  - 公开代码仓库或关键实现片段
  - 公开技术文档、架构文档、设计文档
  - 成功/失败案例与可借鉴机制
- R1.5：当用户未主动给出对标样本时，`/explore` 必须主动追问“希望对标什么”，并可给出参考维度建议，如：产品体验、信息架构、交互细节、协议设计、工程架构、测试治理、发布机制。
- R1.6：`/explore` 结束时必须明确输出：
  - 已澄清事实
  - 仍待澄清问题
  - 建议归属的特性树节点
  - 涉及的 metadata / 契约 / 测试影响
  - `EXPLORE_READY` 或 `GATE_BLOCK`

### R2：重构产品规格（`/prd`）的质量模型

- R2.1：`/prd` 必须在写规格前，先对 `/explore` 输出做准入评审，确认需求是否足够清晰、是否具备可测验收标准、是否明确了范围边界与 out-of-scope。
- R2.2：`/prd` 输出必须对齐特性树四层语义：
  - L1：关键能力
  - L2：功能特性
  - L3：子功能或组件
  - L4：开发 Story / 最小可交付单元
- R2.3：L1 和 L2 的规格必须显式定义用户旅程与端到端价值闭环，不能只有模块罗列。
- R2.4：L3 的规格必须显式定义接口契约页要求，至少覆盖 API、事件、存储、Tool/Prompt/Output Contract 中的一类或多类。
- R2.5：L4 的规格必须显式定义 Story 的完成定义（DoD）、影响范围、依赖边界、端侧与云侧交付面。
- R2.6：`/prd` 必须把验收标准从“描述性清单”升级为“可测量、可追踪、可与四层测试绑定”的验收基线。
- R2.7：`/prd` 中的标杆分析必须沉淀为结构化结论，而不是仅罗列名词。每个外部参考都必须说明：
  - 借鉴点
  - 不借鉴点
  - 适用条件
  - 约束或成本

### R3：重构设计方案（`/design`）的评审与可实施性模型

- R3.1：`/design` 必须先评审上游 `spec.md` 与 `acceptance.yaml` 是否真的足以支撑设计，不足则阻断进入设计。
- R3.2：`/design` 必须至少给出两个方案，并从职责边界、复杂度、演进性、可测试性、观测性、灰度/回滚能力、端云协作成本等维度进行比较。
- R3.3：`/design` 必须对标业界一流实践，但不是“照抄”，而是明确：
  - 对标对象是什么
  - 其优点是什么
  - 为什么适合或不适合本项目
  - 当前选型与目标最佳实践的差距
  - 后续演进路径
- R3.4：`/design` 必须评审实现前置条件是否齐备：metadata、错误码、契约、codegen、测试入口、门禁、灰度策略、回滚方案。
- R3.5：L1 的 `design.md` 必须将架构交付件纳入四件套内部，不得另起附件文档。至少包含：
  - 组件/包图
  - 用例图
  - 主流程/异常流程图
- R3.6：L2 与 L3 的设计必须分别给出：
  - 用户旅程与模块边界映射
  - 契约与实现边界映射
  - 验收项与测试层映射

### R4：重构开发阶段（`/dev`）为 Story 驱动交付

- R4.1：`/dev` 必须从“按 tasks 做事”升级为“按 L4 Story 交付”，`tasks.md` 仅作为 Story 的工程执行清单，不再作为特性树层级语义。
- R4.2：每个 Story 必须显式说明：
  - 用户价值或平台价值
  - 端侧改动范围
  - 云侧改动范围
  - metadata / codegen 影响
  - 错误码与契约影响
  - 四层测试计划
- R4.3：开发前必须评审该 Story 是否满足准入条件：
  - spec 与 design 已冻结到可开发程度
  - metadata-first / contracts-first 路径明确
  - 不存在魔鬼数字、魔鬼字符串、硬编码错误码/文案方案
  - 端侧与云侧编码规范已对齐
- R4.4：开发过程中必须继续遵守现有强制规范：
  - metadata-first
  - contracts-first
  - codegen-first
  - runtime 统一能力
  - Dart 语义 token
  - 错误码生成引用
  - 禁止直接硬编码关键协议字段

### R4b：将原型通道（`/try`）纳入同一治理模型

- R4b.1：`/try` 必须作为标准链路之外的“快速验证通道”被正式纳入命令体系，而不是游离说明。
- R4b.2：`/try` 只豁免特性树四件套的前置创建，不豁免 metadata-first、contracts-first、DDD 分层、runtime 统一、Dart 编码规范、错误码规范与 codegen 保护。
- R4b.3：`/try` 必须同样支持对标输入机制，尤其在“想法尚未明确、需要看原型或参考实现再判断是否立项”时，要求先明确验证目标、成功标准和参考对象。
- R4b.4：`/try` 的验证完成后必须明确分流：
  - 可行：进入 `/land` 基线化，再纳入标准特性树与后续 SDD 链路
  - 不可行：沉淀结论并清理实验性产物
- R4b.5：`/try` 的结果必须能被标准流程吸收，不能形成长期游离于特性树之外的“半成品能力”。

### R5：建立“特性树 + 四层测试 + 验收标准”统一模型

- R5.1：特性树目标模型统一为四层：
  - L1：关键能力
  - L2：功能特性
  - L3：子功能或组件
  - L4：Story / 最小交付单元
- R5.2：历史 `L5` 视为兼容层，仅用于迁移期读取，不再作为新增特性的默认建模层。
- R5.3：`tasks.md` 从树层级概念中移除，仅保留为执行文档，不再与树节点语义混用。
- R5.4：`acceptance.yaml` 升级为交付唯一验收真相源，必须可同时追踪：
  - spec / design / Story 对应关系
  - 四层测试覆盖关系
  - 本地门禁与全量门禁
  - 证据文件与测试函数
- R5.5：验收标准必须统一对齐四层测试视图：
  - T1：契约与静态层
  - T2：模块与交互层
  - T3：端云集成层
  - T4：端到端旅程层
- R5.6：L1 与 L2 必须有用户旅程级验收要求；L3 必须有契约级验收要求；L4 必须有 Story 完成定义与对应的 T1~T4 覆盖计划。
- R5.7：`/try` 虽不要求预先创建特性树节点，但一旦验证成功，必须通过 `/land` 回补到特性树四件套和验收模型中。

### R6：建立“业界对标输入机制”并贯穿 explore / prd / design

- R6.1：`/explore`、`/prd`、`/design` 均必须支持用户提供以下类型的对标输入：
  - 产品链接或产品名
  - Figma / 原型 / 截图 / 视频
  - GitHub / 开源代码 / SDK / API 文档
  - 技术博客 / 架构文档 / 设计说明
- R6.2：三阶段都必须引导用户说明其对标目标，例如：
  - “你最想借鉴哪款产品的体验？”
  - “有没有希望参考的原型、页面或代码仓库？”
  - “你希望对标的是交互体验、信息结构、架构设计还是工程质量？”
- R6.3：若用户提供对标对象，系统必须抽取并沉淀为规格和设计语言：
  - 期望达到的质量上限
  - 明确借鉴的优点
  - 需要规避的问题
  - 本项目的适配约束
- R6.4：若用户未提供对标对象，系统可以主动建议典型标杆方向，但必须区分“建议对标”与“最终采纳”，不得未经确认直接当作强需求。
- R6.5：任何外部对标吸收都必须遵守公开可引用、可审计、避免直接复制私有实现的边界。
- R6.6：在 `/try` 场景下，对标输入可以作为“是否值得立项”的关键判断依据，允许先围绕参考样本验证可行性，再决定是否进入 `/prd`。

### R7：建立迁移与兼容策略

- R7.1：现有 `L1_domain / L3_component / L4_detail / L5_leaf / L5_cross_cutting` 需要有兼容映射表，避免一次性重写全树。
- R7.2：命令文档、主流程文档、特性树标准、测试规则、验收模板必须统一引用同一套术语。
- R7.3：允许分阶段落地：
  - 第一阶段：统一规格口径与模板
  - 第二阶段：统一命令文案与准入 Gate
  - 第三阶段：统一 `acceptance.yaml` 与测试映射
  - 第四阶段：迁移存量节点与脚本校验

## 不做什么（Out of Scope）

- OS1：本次 PRD 不直接实现全部命令脚本、门禁脚本与 codegen 逻辑，只冻结规格基线与验收基线。
- OS2：本次 PRD 不要求一次性迁移仓库内全部历史特性树节点，只要求定义迁移原则与兼容策略。
- OS3：本次 PRD 不直接重写所有 `.cursor/commands/*.md`，只明确后续设计与实施范围。
- OS4：本次 PRD 不强制用户必须提供私有或商业敏感材料作为对标输入，仅支持公开或用户自愿提供的参考。
- OS5：本次 PRD 不改变端云现有核心编码规则本身，只重构其在 explore / prd / design / dev 流程中的承接方式。
- OS6：本次 PRD 不改变 `/try` 与 `/land` 的基本职责，只把它们纳入统一治理叙事与吸收路径。

## 约束

### 技术约束

- 必须与 `specs/00_MASTER_DEVELOPMENT_FLOW.md`、`specs/feature-tree/00_FEATURE_TREE_STANDARD.md`、`specs/feature-tree/01_FEATURE_TREE_LEVEL_DEFINITIONS.md`、`.cursor/rules/03-testing.mdc` 保持一致或推动其统一升级。
- 必须延续 metadata-first、contracts-first、codegen-first、DDD 单向依赖、runtime 统一能力等现有强约束。
- 必须兼容当前仓库仍存在的历史层级命名与验收格式，避免一次性破坏存量特性树。
- 必须考虑 Cursor 命令、规则、spec 文档、acceptance 模板、验证脚本之间的联动，不得只改某一个文档层。

### 业务约束

- 后续所有新增特性都应能复用本模型，不应只服务于个人助理或某个单域。
- 大能力（如个人私人助理）必须能在特性树内完成规格、设计、架构图、验收与测试映射，不依赖散落的附件文档。
- 规格质量提升不仅针对“写得更完整”，还要覆盖“更容易做设计评审、更容易做开发准入、更容易做门禁验收”。

### 对标输入约束

- 对标输入必须区分为“用户明确指定”和“系统建议参考”。
- 对标分析必须说明适用边界，不得把外部产品方案直接当作本项目最终方案。
- 对标公开代码与公开文档时，只能做架构与机制吸收，不得直接复制不可用或不合规实现。

## 验收重点

详见 `acceptance.yaml`，本次规格基线重点检查：

- A1~A2：`/explore` 是否真正具备批判性澄清与对标输入机制。
- A3~A4：`/prd` 是否真正对齐四层特性树与四层测试模型。
- A5~A6：`/design` 是否具备上游评审、方案对比、对标吸收与可测试性评审能力。
- A7：`/dev` 是否完成 Story 驱动交付与契约优先、metadata 优先的强绑定。
- A8：`/try` 是否被纳入统一治理模型，并明确与 `/land`、标准 SDD 链路的衔接。
- A9~A10：特性树、`tasks.md`、`acceptance.yaml` 的语义是否完成解耦与统一。
- A11：四层测试和门禁是否能与验收标准形成闭环。
- A12：业界对标输入机制是否可贯穿 explore / prd / design / try 全流程。
