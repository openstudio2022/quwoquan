# L2 规格：群组体验重构

## 背景与动机

当前群组发现与详情体验仍沿用单一“圈子”模型，带来两类结构性问题：

1. **发现层裂缝**：用户在首页和搜索里既会找兴趣圈，也会找学校、班级、公司、部门，但现有入口与推荐没有把它们收敛成一个统一的群组入口。
2. **详情层裂缝**：现有详情页只适合兴趣型圈子，不适合学校、院系、班级、公司、部门等组织型主页。

本 L2 的目标，是建立统一的 `群组` 入口与模板化详情页，让用户在一个发现入口里找到不同类型的关系主页，并在详情页中获得一致但不违和的参与体验。

## 目标用户

- 新用户：需要在统一入口里快速找到兴趣圈或组织主页。
- 活跃用户：期望推荐结果与自己的兴趣、身份和近期活跃主题同步。
- 助理用户：希望在群组内唤起助理时，依然能承接所属领域上下文。

## 功能范围

### L3: domain-taxonomy-alignment（领域标签对齐）

- 建立统一 domain taxonomy 配置文件，定义领域 ID 及其属性。
- 群组入口的分类、推荐和上下文继承由 taxonomy 驱动。
- 个人助理 domain routing 与群组分类共享同一套领域 ID。
- 领域 ID 至少覆盖：travel, local_life, health_wellness, tech, education, lifestyle, social_meet, culture_arts, automotive, finance, fortune。

### L3: resonance-discovery（同频发现）

- 用户兴趣画像聚合（助理交互领域频次 + 内容浏览标签 + 群组参与历史）。
- 基于兴趣画像的群组推荐排序。
- 群组列表页「推荐」频道由同频推荐驱动，并允许同时返回兴趣型圈子与组织型主页。
- 群组卡片统一使用类型徽章区分 `圈子 / 学校 / 院系 / 班级 / 公司 / 部门` 等详情模板。

### L3: circle-homepage-redesign（群组详情模板基线）

- 建立共享的群组详情基础壳层与模块化首页。
- 冻结 `通用圈子模板`：`首页 / 内容 / 群 / 成员`。
- 冻结 `组织主页模板`：`首页 / 内容 / 组织 / 成员`。
- 冻结组织节点二级页面与通用群主页的差异化规则。
- 保持群组内助理上下文、内容发布、成员摘要和降级策略的一致性。

## 不做什么（Out of Scope）

- 不实现推荐算法训练流水线，仅冻结对象边界、排序规则与降级口径。
- 不修改个人助理引擎核心，只在路由层传入群组领域上下文。
- 不在本 L2 内冻结具体事物主档、抓取、归一和展示模型。

## 约束

- 领域标签配置必须为 metadata YAML，经 codegen 生成端云代码。
- 兼容现有圈子分类——旧分类 ID 映射到新领域 ID，不删除旧数据。
- 首页和搜索的统一用户词必须是 `群组`。
- 组织型详情页前台直接显示具体名称，不强制包装成“圈子”。
- 推荐排序必须支持降级到按热度排序。

## 验收重点

- A1：统一领域标签体系建立，群组入口和助理路由共享至少 5 个领域。
- A2：群组发现可在一个入口中同时推荐兴趣型圈子与组织型主页。
- A3：群组详情模板冻结完成，并区分通用圈子与组织主页。
- A4：群组内助理上下文继承不因详情模板变化而断裂。
