# L3 规格：chat-list-ui-polish — 趣信会话列表对齐微信基线

> **层级**：L3_story（隶属 L2 `chat-experience-optimization`）
> **状态**：specified
> **本期边界**：**密信不纳入本期**，本期只覆盖 `全部 / @我 / 未读` 三个消息胶囊及其对应的列表体验。

## 0. 一句话定义

除密信外，趣信会话列表在视觉层次、行高间距、头像组合、时间展示、未读和 `@我` 交互上对齐微信基线，并统一由每用户 `ChatInbox` 收件箱读模型驱动，消除页面本地拼装的第二真相源。

## 1. 背景与动机

当前趣信会话页与微信基线相比存在六类核心断层：

1. **列表行信息不完整**：部分场景只有标题，第二行最新消息预览和右上角时间没有稳定实现，无法形成微信式的两行信息层级。
2. **头像辨识度不足**：群头像未填充区域与页面白底过近，用户无法快速感知“这是组合头像容器而不是留白”。
3. **分割线节奏缺失**：列表行之间缺少微信式弱分割线或分割线 inset 错误，导致会话边界模糊、页面显得松散而非精致。
4. **滚动交互不顺滑**：选中 `全部` 时二级胶囊上滑隐藏后留白明显，回滑恢复也不具备首页圈子页同等流畅度。
5. **`@我` / `未读` 不一致**：当前数字角标并非统一用户态模型驱动，读完后也没有稳定减数规则。
6. **端云契约未真正收口**：页面仍在直接消费多种 `Map` 形态和本地 mock 字段，列表摘要、时间、未读、头像来源没有统一真相源。

## 2. 目标用户

- 趣信高频沟通用户
- 每天多次查看消息列表、依赖未读提醒与 `@我` 提醒的群聊用户
- 对主流 IM 基线有强心智、会直接拿微信做对比的用户

## 3. 目标与成功标准

### 3.1 产品目标

- 让用户在消息页首屏即可快速完成“看头像识别会话、看标题确认对象、看第二行理解最近消息、看右上角判断新鲜度、看胶囊决定是否先处理 `@我` / 未读”的五步扫描。
- 用统一的 per-user inbox 数据模型承载列表展示与角标逻辑，避免页面本地推断造成的错数、假数、读后不减。

### 3.2 成功标准

- 用户进入趣信页后，首屏会话列表信息密度和可扫读性接近微信消息列表。
- `@我` 和 `未读` 的角标数量来自同一模型，并在阅读后按统一规则递减。
- 在 100 条会话规模下，滚动隐藏/回滑显示二级胶囊不出现留白或明显跳变。

## 4. 功能范围

### 4.1 In-Scope

| 编号 | 功能 | 说明 |
|---|---|---|
| UI1 | 微信式两行列表行 | 每一行统一为“头像 + 两行文字 + 右上时间”，支持标题省略与第二行单行预览 |
| UI2 | 头像与组合头像对齐 | 单聊圆角方形头像；群聊 1~9 宫格组合头像；未填充底色需弱对比可辨识 |
| UI3 | 弱分割线与行节奏 | 列表行使用微信式低干扰分割线和稳定行高/间距 |
| UI4 | 胶囊滚动显隐 | `全部 / @我 / 未读` 所在二级胶囊上滑隐藏、回滑显示，且不留下白色空洞 |
| UI5 | `@我` / `未读` 角标 | `@我` 显示未处理提及数，`未读` 显示未读消息数，二者统一用同一用户态模型驱动 |
| UI6 | 空状态话术 | `全部`、`@我`、`未读` 分别有独立空状态文案，不再复用鼓励语 |
| UI7 | 端云一致数据源 | 会话列表统一消费 `GET /v1/chat/inbox` 对应的 per-user inbox read model，而不是 UI 本地拼装 |
| UI8 | 已读减数一致性 | 进入会话并成功完成已读同步后，会话行未读数、`@我` 胶囊数、`未读` 胶囊数按同一模型更新 |

### 4.2 Out-of-Scope

- 密信列表视觉和交互优化
- 会话详情页气泡样式、输入区、语音/视频通话入口
- 联系人页的索引、排序与群管理
- 自定义群头像上传与群名称编辑流程
- 消息列表之外的新提醒中心、系统通知中心

## 5. 对标

本期以用户提供的微信消息列表示意为主 benchmark。

| 维度 | 微信基线 | 趣信本期目标 | 借鉴/不借鉴 |
|---|---|---|---|
| 单聊行结构 | 头像 + 标题 + 摘要 + 时间 | 完整对齐 | 借鉴 |
| 群头像 | 1~9 宫格，空白区有轻灰底 | 布局与灰底辨识度对齐 | 借鉴 |
| 行分割线 | 低对比、弱存在感 | 低干扰、不脏、不缺失 | 借鉴 |
| 时间显示 | 当天时段、昨天/前天、月日、跨年月日 | 完整对齐 | 借鉴 |
| `@我` / 未读入口 | 胶囊中高优先入口 | 本期只做 `全部 / @我 / 未读` 三个胶囊 | 借鉴并收敛 |
| 密信 | 微信无同形态 | 本期不做 | 不借鉴 |

## 6. 产品规则

### 6.1 会话行内容规则

- 每一行固定由四个信息块组成：
  - 左侧头像区
  - 中间标题行
  - 中间摘要行
  - 右上时间区
- 标题行：
  - 单聊默认显示对方昵称
  - 群聊默认显示群名称
  - 若群名称为空，则回退为按规则拼接的成员名称摘要
  - 标题与时间同一行；标题过长时在接近时间前使用省略号截断
- 摘要行：
  - 固定单行展示，超出宽度省略
  - 文字消息显示最近一条文本摘要
  - 图片/视频/语音/电话等非文本消息显示对应类型预览文案或图文组合
  - 不允许因为没有正确字段而直接空白
- 时间区：
  - 字号与摘要行一致或略小，不得抢标题视觉权重
  - 当天显示时段 + 时间：`凌晨/上午/中午/下午/晚上 hh:mm`
  - 昨天显示 `昨天`
  - 前天显示 `前天`
  - 同年更早显示 `M月d日`
  - 跨年显示 `yyyy年M月d日`

### 6.2 头像与分割线规则

- 单聊头像为圆角方形。
- 群聊头像支持 1~9 宫格：
  - 1 人：单头像
  - 2 人：两格
  - 3 人：1-2
  - 4 人：2-2
  - 5 人：2-3
  - 6 人：3-3
  - 7 人：1-3-3
  - 8 人：2-3-3
  - 9 人及以上：3-3-3，取前 9
- 组合头像中所有子头像尺寸必须一致；不能出现上行/下行大小不一致的凌乱感。
- 组合头像未填充区域使用轻灰底，要求：
  - 与页面白底可区分
  - 对比度不能过强，不可脏、不可沉
- 列表分割线要求：
  - 为弱存在感的分割线
  - 不得全屏满宽穿透头像区，需保持微信式 inset 视觉
  - 不得因为对比度过高而抢主信息层级

### 6.3 胶囊与空状态规则

- 本期消息胶囊只对 `全部 / @我 / 未读` 冻结产品规则；密信不纳入本期验收。
- `全部`：
  - 不显示数字角标
  - 空状态可使用鼓励语
- `@我`：
  - 显示“未处理提及消息总数”
  - 角标上限 `99+`
  - 空状态文案为提醒型，不使用“快去聊天吧”的鼓励语
- `未读`：
  - 显示“未读消息总数”
  - 角标上限 `99+`
  - 空状态文案为收敛型，不使用鼓励语
- 空状态建议口径：
  - `全部`：暂无对话 / 开始和圈友聊起来吧
  - `@我`：还没有人 @ 你 / 群里有人点名你时，这里会立刻提醒
  - `未读`：没有未读消息 / 新消息来了会显示在这里

### 6.4 滚动交互规则

- 在消息主 Tab 且当前胶囊为 `全部 / @我 / 未读` 时：
  - 向上滚动列表，二级胶囊隐藏
  - 向下回滑，二级胶囊恢复
- 显隐过程中内容区必须即时补位，不允许留下胶囊高度的白色留白。
- 显隐动画节奏应与首页圈子页的二级导航显隐语义一致。

### 6.5 已读与减数规则

- 本期统一采用“用户完成已读同步后，角标再递减”的规则。
- 列表行未读数、`未读` 胶囊数、`@我` 胶囊数必须来自同一用户级状态。
- 不允许页面仅靠本地遍历 `Map` 临时计算一个数字，再与服务端返回的数字脱节。

## 7. 端云一致性约束

### 7.1 唯一真相源

- 会话列表唯一数据源：`GET /v1/chat/inbox`
- 操作 / pageId 真相源：`contracts/metadata/messages/conversation/service.yaml` 与 `contracts/metadata/_shared/request_context.yaml`
- 路由真相源：`contracts/metadata/_shared/app_routes.yaml`

### 7.2 App 侧约束

- 列表页必须优先消费 per-user inbox read model，不再把 `Conversation`、本地 mock `Map`、页面内派生逻辑混合作为第二真相源。
- 列表页展示所需字段至少包括：
  - `conversationId`
  - `type`
  - `title`
  - `avatarUrl`
  - `lastMessagePreview`
  - `lastMessageTime`
  - `unreadCount`
  - `pinned`
- 为满足本期 `@我` 和组合头像目标，`/design` 阶段需要明确 inbox contract 是否补齐：
  - `mentionUnreadCount`
  - `avatarUrl`（群聊为服务端预合成图；端侧不再消费 `avatarCompositeUrls`）
- 路由跳转必须继续使用 codegen 的 `AppRoutePaths.chatDetail(...)`，不能在 UI 层硬编码路径。

### 7.3 生命周期

- 未读和 `@我` 都是用户级状态，不是会话公共状态。
- 其递减必须以 `ConversationUserState` / inbox read model 的更新结果为准。
- 本期不改变密信的权限、解锁和生命周期模型。

## 8. 非功能要求

### 8.1 真实网络条件假设

- 4G 常态：RTT 80~150ms，下行 5~20 Mbps
- 弱网兜底：RTT 150~300ms，下行约 1 Mbps
- App 需在本地缓存可用时优先展示缓存，再异步刷新 inbox

### 8.2 容量假设

- 普通用户会话列表规模：20~100
- 重度用户会话列表规模：100~300
- 本期以 100 条列表连续滚动作为主要体验基线

### 8.3 SLO / KPI

- P75 进入趣信页后 300ms 内看到可交互的首屏缓存列表
- P95 inbox 刷新结果在 1.2s 内完成首屏更新（常态网络）
- 100 条会话规模下列表滚动保持可感知流畅，不出现显著卡顿和跳闪
- 已读同步成功后，列表与胶囊数字更新延迟不超过 1s

## 9. 灰度、迁移与回滚

- 本期允许通过数据源开关将列表从旧 `listConversations` 迁移到 `listInbox`
- 灰度顺序：dev → integration → prod
- 若 `listInbox` 返回字段不全、badge 错数或排序异常，必须可回滚到旧列表数据源
- 回滚时应保留：
  - 现有路由不变
  - 详情页不变
  - 非密信消息主链路可继续使用

## 10. 权限边界

- 会话列表只展示当前登录用户可见的会话 inbox 条目
- `@我`、未读、置顶都属于当前用户私有视图，不得向他人透出
- 本期不新增相册、麦克风、定位等系统权限

## 11. 不做什么

- 不在本期做密信行样式与 badge 规则
- 不在本期做会话滑动菜单、批量管理、归档
- 不在本期做联系人页 IA 重构
- 不在本期做新的通知中心

## 12. 验收重点

### T1

- 时间格式化、摘要映射、角标计数和减数逻辑具备可单测的纯函数或 typed view-model

### T2

- 列表行结构、弱分割线、头像灰底、胶囊显隐和空状态在 widget / integration 层稳定可验证

### T3

- inbox contract、已读递减、`@我` 计数来源与 metadata / codegen 一致

### T4

- 真机对比微信消息列表，确认首屏扫读感、头像辨识度、时间层级和分割线节奏达到行业一流基线
