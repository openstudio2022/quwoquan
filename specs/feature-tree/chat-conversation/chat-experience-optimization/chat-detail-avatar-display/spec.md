# L3 规格：chat-detail-avatar-display — 对话页头像展示与用户信息缓存

> **层级**：L3_story（隶属 L2 `chat-experience-optimization`）
> **状态**：specified

## 0. 一句话定义

对话页显示对方用户圆角方形头像，点击可跳转用户主页，基于时间戳驱动的两层缓存确保头像加载不影响页面打开性能。

## 1. 背景与动机

当前 `ChatDetailPage` 存在三个问题：

1. **对话页不显示头像**：AppBar 仅展示会话标题，没有对方用户头像，缺乏直觉感知。
2. **无用户信息缓存**：每次进入对话都重新请求用户信息，浪费流量且增加延迟。
3. **无关系标识**：群聊中无法区分陌生人、关注用户、同好、密友四种关系。

## 2. 目标用户

- 趣聊日活用户（进入 1v1 / 群聊对话页）

## 3. 功能范围

### 3.1 In-Scope

| 编号 | 功能 | 说明 |
|---|---|---|
| AD1 | 对话页 AppBar 头像 | 1v1 对话：显示对方用户圆角方形头像 + 昵称；群聊：显示群名称（头像复用九宫格） |
| AD2 | 消息气泡头像 | 每条消息旁显示发送者圆角方形头像 |
| AD3 | 头像点击跳转 | 点击头像跳转用户主页，区分四种关系（陌生人/关注/同好/密友） |
| AD4 | 内存 LRU 缓存 | 最多 200 条用户信息内存缓存，LRU 淘汰 |
| AD5 | 磁盘永久缓存 | Hive Box 存储用户信息 + 云端时间戳，无 TTL |
| AD6 | 时间戳驱动刷新 | 打开对话时后台批量查用户时间戳，有变化才拉取完整信息 |
| AD7 | WebSocket 刷新 | 收到 `user.profile.updated` 事件时主动更新缓存 |

### 3.2 Out-of-Scope

- 用户主页的具体布局（已有）
- 关系类型的判定逻辑（已有 `RelationshipCapabilityRepository`）
- 消息列表数据加载（已有 `chatMessageProvider`）

## 4. 两层缓存架构

```
┌──────────────────────────────────────────┐
│         内存层 LRU（最多 200 条）          │
│  key: userId → UserProfileCacheEntry     │
│  命中 → 直接返回，≤1ms                    │
│  淘汰：LRU，仅内存，不持久化              │
└──────────────┬───────────────────────────┘
               │ miss
┌──────────────▼───────────────────────────┐
│       磁盘层 Hive Box（永久保存）          │
│  key: userId → UserProfileCacheEntry     │
│  命中 → 写入内存层 → 返回                 │
│  无 TTL，永久保存                         │
└──────────────┬───────────────────────────┘
               │ miss
┌──────────────▼───────────────────────────┐
│          云端请求                          │
│  GET /users/profiles?ids=...             │
│  返回 → 写入磁盘层 + 内存层 → 返回        │
└──────────────────────────────────────────┘
```

## 5. 时间戳驱动按需刷新

### 5.1 进入对话页时的刷新流程

```
端侧                                          云端
  │                                             │
  │ 1. 读本地缓存（内存→磁盘）→ 立即渲染头像/名字  │
  │                                             │
  │ 2. 后台（30s 冷却）：                         │
  │    POST /users/timestamps                   │
  │    body: { userIds: [该会话成员userId列表] }   │
  │ ──────────────────────────────────────────→  │
  │ ←── { timestamps: {uid: ts, ...} }           │
  │                                             │
  │ 3. 比对本地 cloudUpdatedAt：                   │
  │    cloud > local → 收集 staleUserIds          │
  │    cloud == local → 跳过                      │
  │                                             │
  │ 4. 仅拉取有变化的用户：                         │
  │    GET /users/profiles?ids=u1,u2             │
  │ ──────────────────────────────────────────→  │
  │ ←── [完整用户信息...]                          │
  │                                             │
  │ 5. 更新磁盘+内存缓存 → 静默刷新头像/名字        │
```

### 5.2 冷却时间

同一会话 30 秒内不重复检查用户时间戳，避免快速切换会话时反复请求。

### 5.3 云端接口

| 接口 | 方法 | 说明 |
|---|---|---|
| `POST /users/timestamps` | POST | 批量查询用户信息时间戳（**新增**） |
| `GET /users/profiles` | GET | 批量拉取用户完整信息（`?ids=u1,u2`，已有/扩展） |

**`POST /users/timestamps` 响应结构：**
```json
{
  "timestamps": {
    "user_abc": "2026-03-10T10:30:00Z",
    "user_def": "2026-03-09T08:15:00Z"
  }
}
```

## 6. 端侧存储结构

```dart
class UserProfileCacheEntry {
  final String userId;
  final String? avatarUrl;
  final String? displayName;
  final String? bio;
  final String cloudUpdatedAt;  // 云端时间戳 ISO8601
  final DateTime localCachedAt; // 本地写入时间
}
```

## 7. 四种用户关系

点击头像跳转用户主页时，根据 `RelationshipCapabilityDto` 传递关系类型：

| 关系 | 说明 | 跳转目标 |
|---|---|---|
| 陌生人 | 群聊中的非关注用户 | 用户主页（受限模式） |
| 关注 | 单向关注的用户 | 用户主页（标准模式） |
| 同好 | 互相关注的用户 | 用户主页（互关标记） |
| 密友 | 特殊标记的互关用户 | 用户主页（密友标记） |

关系类型判定复用已有 `RelationshipCapabilityRepository`。

## 8. 对标

| 维度 | 微信 | 趣聊目标 |
|---|---|---|
| 对话页头像 | 圆角方形，显示在消息旁 | 圆角方形，消息旁 + AppBar |
| 头像缓存 | 本地永久缓存 | 内存 LRU + 磁盘永久 |
| 点击跳转 | 跳转用户主页 | 跳转用户主页 + 关系标识 |

## 9. 业务约束

- 对话设置页头像尺寸比对话页消息头像大一号（设置页 52px，消息 40px）
- 头像圆角与列表页一致（`AppSpacing.borderRadiusMd` ≈ 8px）
- 时间戳由云端生成和维护，端侧不生成比较用时间戳
- 内存 LRU 上限 200 条，超出淘汰最久未访问的

## 10. 非功能要求

- 内存缓存命中时头像渲染 ≤1ms
- 打开对话页不因后台用户信息刷新而闪屏或重绘整个列表
- 后台刷新失败时静默忽略，以本地数据为准

## 11. 验收重点

### T1
- `POST /users/timestamps` 请求响应结构契约
- `GET /users/profiles` 批量查询响应结构契约

### T2
- 对话页 AppBar 显示对方圆角方形头像（1v1）
- 消息气泡旁显示发送者圆角方形头像
- 点击头像跳转用户主页

### T3
- 时间戳比对：仅拉取有变化的用户信息
- 新用户（本地无缓存）首次进入时从云端拉取并缓存
- WebSocket `user.profile.updated` 事件触发缓存更新

### T4
- 真机：打开对话页无闪屏，头像从缓存加载流畅
- 弱网：本地缓存头像正常显示，后台刷新静默失败
