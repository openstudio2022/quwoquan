# member-add-remove-policy 设计方案

## 设计动因

如果只把建群做出来，而不冻结“建群后还能怎么加人、什么群能解散”，最终仍会在聊天信息页出现另一套规则。这个 Scenario 的作用，就是让后续成员治理复用建群规则，而不是重新发明一套 UI 和 contract。

## 方案对比

### 方案 A：继续让聊天信息页单独维护加人逻辑与解散逻辑

优点：

- 对现有页面侵入少。

缺点：

- 与建群规则重复，容易分叉。
- 圈子群/私建群危险操作边界容易只做前端隐藏。

### 方案 B：后续加人复用建群 candidate engine，解散按 `conversation.type` 统一治理

优点：

- 资格规则、去重规则、500 人上限都只有一套。
- 私建群/圈子群差异可由服务端硬约束。
- 消息列表一致性更容易验证。

缺点：

- 需要把聊天信息页切到新的 candidate contract。

## 选型决策

**选定方案：方案 B**

## 关键设计决策

### KD-1：继续加人复用建群候选成员 contract

- 已有群继续加人时，不再自己组合联系人/成员列表
- 直接复用“来源成员列表 + 互关过滤 + 去重 + 上限校验”的同一 contract 语义

### KD-2：危险操作基于 `conversation.type`

- `group`：允许群主解散
- `circle`：不允许解散

前端只做第一层可见性控制，真正的权限判断和错误码由服务端负责。

### KD-3：消息列表清理由服务端结果驱动

私建群解散后，优先保证：

- 拉取会话列表时该群消失
- 如有本地缓存，则同步移除
- 正在打开该群的页面收到状态变化后可安全退出

### KD-4：失败态保留上下文

- 加人失败：保留当前已选成员
- 解散失败：停留当前设置页并展示明确失败原因
- 圈子群误触解散：以服务端错误码返回，不允许出现“看起来成功但其实没删”的状态

## metadata / codegen 方案

- `messages/conversation/service.yaml`
  - 冻结继续加人的候选成员 operation
  - 冻结 `DissolveConversation` 的群类型约束与错误语义
- `messages/conversation/fields.yaml`
  - 如需新增 `dissolvedAt` / `isDissolved` 等投影字段，在此阶段冻结
- error metadata
  - 新增“圈子群不可解散”“超过群人数上限”“成员不满足互关准入”等错误码

## T1~T4 证据矩阵
