# L3 Scenario: social-relationship-search-contract

## 节点定位

- `L1_capability`: `global-search-experience`
- `L2_journey`: `cross-domain-search-journey`
- `L3_scenario`: `social-relationship-search-contract`

## 背景与动机

历史上“朋友 / 联系人 / 社交关系”被混着使用。最新两段式搜索 UX 已经把首段“人”的结果收敛成可直接发起聊天的“联系人”分段，因此需要冻结：当前 Journey 内展示的是可行动联系人投影，而不是泛化的用户主页搜索列表。

## 目标用户

- 需要查找联系人并直接进入单聊/群聊，而不是先进入用户主页中间页的用户。

## 功能范围

- 冻结“联系人”在当前两段式搜索中的产品命名、对象边界与跳转语义。
- 明确联系人联想结果的主契约由 `messages/conversation` 提供，并可补充 `user` 域身份信息。
- 定义联系人结果项展示的最小身份信息、会话绑定信息与跳转目标。

## Out of Scope

- follow/unfollow 写侧行为本身。
- 私密账号和密信隔离能力本身。
- 泛化用户主页搜索列表与 profile ranking 设计。

## 约束

- 当前 Journey 中“人”的实时联想统一展示为“联系人”。
- “联系人”结果必须携带可落地的 `conversationId / conversationType`，点击后直接进入会话。
- 当前账号或登录子账号可见范围内的对象都允许进入结果。

## 对标输入与吸收结论

- 借鉴微信联系人搜索结果的清晰对象卡片与快捷进入会话方式，但明确当前 UX 优先服务“去聊天”而非“看主页”。

## 角色分工

- `messages`: 联系人搜索主契约、会话绑定与直达语义真相源。
- `user`: 联系人展示所需的公开身份补充信息与后续扩展真相源。
- `global-search-experience`: 分段组织、文案、入口与跳转。

## 既有 Story 覆盖矩阵

| 历史节点 | 处理 |
|---|---|
| `contact-search-index` | 删除历史节点，归并到本 Scenario |
| `search-query-contract` | 删除历史节点，查询契约迁回 user 域与全局搜索 Journey |

## 数据生命周期合同

- 联系人搜索结果为瞬时读模型。
- 历史搜索只记录 query 与上下文，不单独保存联系人对象快照。

## 小趣 / 权限 / 分享边界

- 本 Scenario 不涉及问小趣逻辑。
- 本期不在账号内再做细粒度权限裁剪。

## 非功能目标

- 首批联系人结果与其它联想分段一同在搜索首屏返回。
- 从联系人结果跳转到对应会话的成功率 > 99%。

## 迁移、灰度与回滚要求

- 从特性树和 PRD 治理层彻底移除旧 chat 搜索节点的散乱命名。
- 如联系人结果契约不稳定，整体回退到旧搜索实现，不保留旧节点并行。

## 验收重点

1. 当前搜索 Journey 中“联系人”命名与对象边界冻结。
2. 联系人结果点击后直接进入会话，而不是进入新的用户中间页。
3. 旧 chat 搜索节点被彻底清理，不再出现“朋友 / 联系人 / 社交关系”混用。
