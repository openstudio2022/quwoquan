# L3 Scenario: multi-domain-result-composition

## 节点定位

- `L1_capability`: `global-search-experience`
- `L2_journey`: `cross-domain-search-journey`
- `L3_scenario`: `multi-domain-result-composition`

## 背景与动机

搜索结果不只是“查出来”，还必须“看得懂、点得准、失败可降级”。当前缺少跨域统一结果模型与分组组织，导致内容、消息、圈子和人的搜索体验无法统一。

## 目标用户

- 需要通过一次输入跨域查找内容、消息、圈子和社交关系的用户。

## 功能范围

- 综合搜索结果的跨域组织与分组展示。
- 内容、社交关系、消息、圈子与频道 facet 的展示顺序与分组规则。
- 单域失败的局部降级与空态策略。
- 点击结果进入详情或列表页的统一跳转语义。

## Out of Scope

- 具体搜索索引实现。
- AI 搜索结果混排。
- 复杂排序算法与个性化召回策略。

## 约束

- 综合搜索最多 fan-out 到 4 个域：`content / user-social / messages / circle`。
- 问小趣不进入综合结果混排。
- 频道只允许作为 circle facet，而不是独立结果主域。
- 结果模型必须可类型化，不允许长期停留在松散 `Map` 拼装层。

## 对标输入与吸收结论

- 吸收微信聊天/联系人搜索结果的对象分组方式。
- 吸收微信内容搜索结果里“内容结果 + 分类/频道”的组织思路。

## 角色分工

- `global-search-experience`: 负责分组规则、混排壳层、降级策略。
- `content/messages/user/circle`: 分别提供各自结果对象与导航目标。

## 既有 Story 覆盖矩阵

| 既有实现 | 处理 |
|---|---|
| 当前 `GlobalSearchSheet` 的本地过滤结果 | 不再作为正式结果模型 |
| chat 侧联系人搜索局部结果 | 收敛为社交关系域的一个结果来源参考，不再单独建树 |

## 数据生命周期合同

- 结果是瞬时读模型，不作为用户长期资产持久化。
- 每次 query 的结果只服务于当前搜索会话。

## 小趣 / 权限 / 分享边界

- 不混排问小趣结果。
- 当前账号或登录子账号范围内的对象均允许进入结果。
- 本 Scenario 不定义搜索结果分享。

## 非功能目标

- 首批结果分组 P95 在 1.5s 内可见。
- 任一域失败时，只影响该域分组，不阻塞整页。

## 迁移、灰度与回滚要求

- 本 Scenario 不保留旧的局部结果组织逻辑。
- 若分组编排出现严重问题，整体回退到旧搜索实现。

## 验收重点

1. 结果分组稳定可理解。
2. 单域降级不拖垮整页。
3. 问小趣与频道对象边界不再混乱。
