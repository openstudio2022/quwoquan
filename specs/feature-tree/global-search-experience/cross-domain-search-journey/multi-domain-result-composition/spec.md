# L3 Scenario: multi-domain-result-composition

## 节点定位

- `L1_capability`: `global-search-experience`
- `L2_journey`: `cross-domain-search-journey`
- `L3_scenario`: `multi-domain-result-composition`

## 背景与动机

搜索结果不只是“查出来”，还必须“看得懂、点得准、失败可降级”。最新体验已经从“单页综合结果”改为“两段式结果组织”：输入后的联想页负责快速直达，独立网络结果页负责 assistant 与内容分类结果，因此需要重新冻结结果编排规则。

## 目标用户

- 需要通过一次输入快速直达本地联系人/聊天记录，并进一步进入网络结果页查看内容或 `小趣搜` assistant 结果的用户。

## 功能范围

- 输入后的联想页编排，且严格按 `最常使用 / 联系人 / 聊天记录 / 搜索网络结果` 顺序展示。
- 联系人、聊天记录默认 3 条，并在当前页内联展开更多列表。
- “搜索网络结果”进入独立网络结果页后的 tab 组织：左侧 `小趣搜`，其余为群组分类 facet。
- 单域失败的局部降级与空态策略。
- 点击联系人/聊天记录直达会话、点击网络结果进入独立结果页、点击结果卡片进入内容或引用对象的统一跳转语义。

## Out of Scope

- 具体搜索索引实现。
- assistant 结果混入联想页四段。
- 复杂排序算法与个性化召回策略。

## 约束

- 联想页只允许四段固定顺序，不允许回退到 domain 分组墙。
- 联系人和聊天记录的“更多”只能做当前页内联展开，不能跳到新的中间列表页。
- `小趣搜` 只出现在独立网络结果页左侧 tab，不进入联想页混排。
- 搜索里的用户面向结果类型统一叫 `群组`。
- 群组分类只允许作为 circle facet，而不是独立结果主域。
- 结果模型必须可类型化，不允许长期停留在松散 `Map` 拼装层。
- 联想页中的联系人 / 聊天记录由本地聊天搜索 contract 提供，不再依赖“社交关系搜索”产品语义。

## 对标输入与吸收结论

- 吸收微信聊天/联系人搜索结果的对象分段、更多后原位展开方式。
- 吸收微信内容搜索结果里“内容结果 + 分类/频道”的组织思路，并将 AI 结果收口为独立 `小趣搜` tab。

## 角色分工

- `global-search-experience`: 负责联想页四段规则、网络结果页编排、降级策略。
- `messages`: 提供本地联系人/聊天记录/最常使用的会话相关对象与导航目标。
- `content`: 提供网络结果页中的内容结果对象与详情跳转。
- `circle`: 提供网络结果页分类 tab 真相源。
- `assistant`: 提供 `小趣搜` assistant 摘要与引用。

## 既有 Story 覆盖矩阵

| 既有实现 | 处理 |
|---|---|
| 当前 `GlobalSearchSheet` 的本地过滤结果 | 不再作为正式结果模型 |
| chat 侧联系人搜索局部结果 | 收敛为本地聊天搜索 contract 的一个实现来源，不再单独建树 |

## 数据生命周期合同

- 结果是瞬时读模型，不作为用户长期资产持久化。
- 每次 query 的结果只服务于当前搜索会话。

## 小趣 / 权限 / 分享边界

- 不混排 `小趣搜` 到联想页联系人/聊天记录分段。
- 当前账号或登录子账号范围内的对象均允许进入结果。
- 本 Scenario 不定义搜索结果分享。

## 非功能目标

- 联想页首批结果分段 P95 在 1.5s 内可见。
- 任一域失败时，只影响该段或当前 tab，不阻塞整页。

## 迁移、灰度与回滚要求

- 本 Scenario 不保留旧的局部结果组织逻辑。
- 若分组编排出现严重问题，整体回退到旧搜索实现。

## 验收重点

1. 联想页四段结构与网络结果页 tab 结构稳定可理解。
2. 单域降级不拖垮联想页或网络结果页。
3. `小趣搜` 与群组分类对象边界不再混乱。
