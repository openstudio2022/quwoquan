# 全局产品概念基线

> 适用阶段：`/explore`、`/prd`、`/design`、`/baseline`
>
> 文档目的：在进入各域 PRD 之前，先冻结全 App 的核心对象、关系模型、品牌解释与页面层级，避免同一概念在不同文档、不同页面、不同服务中被重复发明。

---

## 1. 文档定位

本文件定义趣我圈在产品层面的世界观、核心对象与跨域关系，不直接替代：

- metadata 中的 `route / surface / operation / field`
- 各 `L1/L2/L3` 节点的场景行为规格
- 云侧 DDD 与服务分层文档

本文件回答的问题是：

- 用户到底在 App 里会看到哪些核心对象
- 这些对象之间是什么关系
- 整个 App 的品牌、AI、内容、关系、搜索和主页应该如何统一解释
- 后续 `/prd` 应该沿着哪套概念骨架继续展开

本文件与 `00_GLOBAL_TERMINOLOGY.md` 的分工如下：

- `00_PRODUCT_CONCEPT_SYSTEM.md`：定义“有什么、怎么关联、各域分别扮演什么角色”
- `00_GLOBAL_TERMINOLOGY.md`：定义“这些东西应该怎么命名、哪些词禁止前台使用、旧词怎么迁移”

两份文档不建议硬合并：

- 本文件更像“世界观和对象地图”
- 术语表更像“命名规则和迁移字典”

## 2. 品牌定位与产品主张

### 2.1 品牌名

- 品牌名：`趣我圈`
- AI 助手名：`小趣`

### 2.2 产品定位

趣我圈不是单纯的兴趣论坛，也不是单纯的黄页目录，更不是只有聊天的 IM。

当前统一定位为：

**一个以 `小趣` 为连接纽带、以 `群组` 为关系入口、以 `主页与口碑` 承接真实生活信息的 AI 群组社区。**

### 2.3 品牌解释

`趣我圈` 中的“圈”不再等于某个具体产品名词，也不等于一级入口词 `圈子`。  
它代表的是：

- 与我有关的人
- 与我有关的群
- 与我有关的生活、地点、物和组织

因此：

- 品牌层保留 `趣我圈`
- 产品入口层统一使用 `群组`
- 兴趣型详情页继续使用 `圈子`
- 组织型详情页直接显示 `学校 / 院系 / 班级 / 公司 / 部门`

### 2.4 主张

建议统一沿用这条产品主张作为解释口径：

**和你有关的人、群与生活，都在这里。**

## 3. 全 App 领域与业务对象地图

| L1 能力域 | 产品层角色 | 核心业务对象 | 用户前台主要感知 |
|---|---|---|---|
| `discovery-content` | 负责发现、分发、互动与反馈 | Feed、Post、Comment、互动/负反馈、媒体与发布链路 | 发现流、内容、评论、互动、发布 |
| `circle-community` | 负责关系沉淀入口与关系主页治理 | 群组、圈子、组织主页、群、成员、群级协作 | 群组、圈子、学校/公司/班级、公共群、自建群 |
| `chat-conversation` | 负责实时沟通与消息同步 | Conversation、Message、Inbox、ConversationMember、ConversationUserState | 趣聊、会话、消息、聊天记录 |
| `user-identity-profile-relationship` | 负责主体身份、关系与个人主页世界观 | OwnerAccount、SubAccount、Profile、ContactDiscovery、Invite Attribution | 身份、个人主页、联系人、切换身份 |
| `assistant-run-learning` | 负责助手运行、反馈与学习闭环 | Run、Stream、策略模板、学习事件、画像提案 | 小趣、助手回复、反馈 |
| `global-search-experience` | 负责跨域搜索壳层与结果编排 | 全屏搜索、记录、联想、网络结果页、小趣搜、群组结果 | 搜索、最近在搜、群组结果、联系人结果、小趣搜 |
| `shared-homepage-network` | 负责具体事物主页、口碑、认领与下线 | EntityProfile、ReviewTemplate、ClaimRequest、SourceEvidence、StatusReport | 车型主页、酒店主页、景点主页、餐厅主页、口碑 |
| `product-ops-growth` | 负责增长、运营与效果闭环 | 运营事件、增长策略、实验与观测 | 一般不直接作为前台对象出现 |
| `platform-ops-governance / gateway / runtime` | 负责基础设施与治理 | 配置、可观测、流量治理、运行时能力 | 不直接作为前台对象出现 |

结论：

- 用户前台真正会感知的对象，不止 `共享主页 / 群组 / 群 / 内容`
- 还包括 `身份/个人主页`、`会话/消息`、`小趣`、`搜索`
- `发现` 与 `搜索` 是横切层，不是孤立业务对象，但会强烈影响用户对全 App 的理解

## 4. 产品总模型

### 4.1 六类基础对象

趣我圈的产品骨架统一收敛为六类基础对象：

1. `身份`
应用世界里的真实使用主体，默认由 `SubAccount` 承担，前台主要表现为“身份”和“个人主页”。

2. `主页`
主页分三类：
- `个人主页`：围绕身份建立
- `共享主页`：围绕具体事物建立
- `关系主页`：围绕关系沉淀建立

3. `群组`
所有可加入、可沉淀关系的地方的统一入口。

4. `群`
群组中的子单元，承接小范围交流、资料、公告与协作。

5. `内容`
用户发布的公开表达，统一分为 `笔记 / 作品 / 提问 / 口碑`。

6. `会话`
用户与用户、用户与群、用户与小趣之间的实时沟通上下文。

### 4.2 三条横切能力

除上述对象外，全 App 还有三条横切能力：

- `发现`：把内容与对象推荐到用户面前
- `搜索`：让用户跨域查找内容、人、会话、群组与后续主页
- `小趣`：贯穿内容、搜索、会话与关系沉淀的 AI 连接层，也是用户可直接感知的前台助手

### 4.3 一句话总纲

**身份决定“谁在使用”；主页回答“这是谁/这是什么”；群组和群沉淀“人与人的关系”；内容负责表达；会话负责实时沟通；小趣负责跨对象连接与理解。**

## 5. 身份与主体

### 5.1 基本原则

- `OwnerAccount / 主控账号` 负责登录、安全、恢复、设备与通讯录发现。
- `SubAccount / 子账号` 是应用内真实使用主体，承载主页、群组、内容、会话、邀请与关系。
- 前台默认不把 `OwnerAccount` 暴露成用户心智中心。
- 用户前台默认理解的是“身份”，不是“主控账号”。

### 5.2 个人主页

个人主页属于 `SubAccount`，承载：

- 公开身份与资料
- 发布过的内容
- 关系与被关注状态
- 在群组和会话里的活跃痕迹

### 5.3 归属原则

- 通讯录发现归 `OwnerAccount`
- 好友、群组、群、会话、内容、邀请归因归 `SubAccount`
- 子账号切换必须对群组、聊天、助手、推荐、通知上下文立即生效

## 6. 主页体系

### 6.1 主页是总称，不是单一产品

“主页”在趣我圈里是一个总称，至少包含三类页面：

- `个人主页`
- `共享主页`
- `关系主页`

因此，后续 PRD 里如果只写“主页”，必须保证上下文足够明确；否则应写清：

- `个人主页`
- `共享主页`
- `关系主页`

### 6.2 个人主页

个人主页围绕 `SubAccount` 建立，是“谁”的页面。

它解决：

- 身份展示
- 内容展示
- 关系展示
- 个人经营

### 6.3 共享主页

共享主页围绕具体事物建立，是“这是什么”的页面。

baseline 阶段优先支持：

- `车型`
- `酒店 / 民宿`
- `餐厅`
- `景点`

共享主页至少承接：

- 基础信息
- 口碑
- 内容
- 提问
- 相关群组
- 官方/服务信息

共享主页原则：

- 口碑依附于共享主页，不单独漂浮存在
- 内容可以挂载共享主页
- 群组可以关联共享主页
- 共享主页可以推荐相关群组
- 用户补充一个主页，不等于拥有它

### 6.4 关系主页

关系主页围绕“人为什么在一起”建立，是“谁围绕它沉淀关系”的页面。

关系主页分两种模板：

- 兴趣型：`圈子`
- 组织型：`学校 / 院系 / 班级 / 公司 / 部门`

关系主页可以关联一个共享主页，但不等于共享主页本身。

---

## 7. 群组与关系主页

### 7.1 群组是总入口

一级 Tab 冻结为：

- `群组`

它表示“所有可加入、可沉淀关系的地方”，是总入口名，不是某一种详情页名。

### 7.2 兴趣型关系主页

前台名称：

- `圈子`

示例：

- `摄影圈`
- `L9车友圈`
- `露营圈`

角色：

- `圈主`
- `圈管`

### 7.3 组织型关系主页

前台直接显示具体组织或节点名，不再硬套“圈子”。

示例：

- `复旦大学`
- `计算机学院`
- `2020级3班`
- `字节跳动`
- `推荐算法部`

角色：

- `负责人`
- `管理员`

### 7.4 统一原则

- `圈子` 从全局总称降级为兴趣型关系主页的一种类型名
- 组织型主页不再使用“圈主”一类兴趣社区词
- 学校 / 公司在 baseline 阶段优先作为组织型关系主页，而不是共享主页标准类目

---

## 8. 群

### 8.1 定义

群是关系主页内的子单元。

群不是聊天的同义词。聊天只是群的一种能力。

群可以承接：

- `交流`
- `资料`
- `公告`
- `活动`
- 小范围成员协作

### 8.2 群类型

baseline 统一为：

- `公共群`
- `自建群`

但在组织型关系主页中，前台优先显示具体节点名，不强行露出“群”字。

例如：

- `2020级3班`
- `推荐算法部`
- `摄影小组`

底层仍可复用统一的群能力。

### 8.3 群与会话的关系

- 群是成员和协作单元
- 会话是实时沟通单元
- 一个群通常会有自己的主会话或交流能力
- 但“群”不等于“会话”

### 8.4 baseline 规则

- 小圈子或小组织可只有一个公共群
- 大型关系主页可有多个公共群
- 用户可加入多个自建群
- 群不承担全局公开内容的主分发职责

---

## 9. 内容

### 9.1 用户动作

全局统一动作叫：

- `发布`

不再把所有内容动作都粗暴统一成“发帖”。

### 9.2 一级内容类型

一级内容类型冻结为：

- `笔记`
- `作品`
- `提问`
- `口碑`

### 9.3 表达形式

内容编辑入口继续保留自然创作心智：

- `图文`
- `视频`
- `文章`

这里区分两层：

- `内容类型`：这条内容是什么
- `表达形式`：用户用什么方式开始创作

### 9.4 与发现域的映射

`discovery-content` 规格里仍有 `文章 / 微趣 / 美图 / 视频` 这类展示或投影分组。  
全局概念层统一解释为：

- `笔记 / 作品 / 提问 / 口碑` 是业务内容类型
- `文章 / 微趣 / 美图 / 视频` 是发现流、媒体投影或展示家族

后续 PRD 不得把两层概念混成一层。

### 9.5 与主页和群组的关系

- `口碑` 必须绑定一个共享主页
- `笔记 / 作品 / 提问` 可选择绑定共享主页
- 内容可以发布到个人，也可以发布到群组中的关系主页
- 从群组内发内容时，当前关系主页默认已选中
- 从全局发布入口发内容时，可在发布页补选共享主页和关系主页
- 群层不作为公开内容主时间线

### 9.6 互动与反馈

内容默认承接：

- 评论
- 点赞
- 收藏
- 转发
- 负反馈
- 举报

这些属于 `discovery-content` 与 `product-ops` 的共用反馈基线。

---

## 10. 会话与消息

### 10.1 定义

会话是实时沟通上下文，属于 `chat-conversation` 域。

用户前台主要感知为：

- `趣聊`
- `会话`
- `消息`
- `聊天记录`
- `收件箱 / 会话列表`

### 10.2 类型

- `1v1 会话`
- `群会话`
- `圈子 / 群组关联会话`

### 10.3 与群组的边界

- 群组负责关系沉淀
- 群负责小范围协作
- 会话负责实时沟通
- 群和会话常常绑定，但不是同一层对象

---

## 11. 小趣与助手

### 11.1 用户前台对象

用户前台感知到的助手对象主要有：

- `小趣`
- `小趣搜`

### 11.2 产品角色

小趣不是一个孤立功能，而是贯穿全 App 的连接层：

- 在搜索里：以 `小趣搜` 结果页出现
- 在会话里：可作为助手成员参与会话
- 在内容与主页里：提供理解、建议与后续学习闭环
- 在资料与画像里：输出提案，但不绕过治理规则直接改事实主档

### 11.3 内部抽象

内部继续使用：

- `Run / Stream`
- 策略模板
- 学习事件
- 反馈注入
- 画像提案

这些是内部能力名，不作为前台主词。

---

## 12. 搜索与发现

### 12.1 发现

发现不是单独对象，而是把内容、主页、群组和关系推荐到用户面前的横切能力。

发现域负责：

- 发现流
- 推荐排序
- 内容反馈
- 运营与评估数据回流

### 12.2 搜索

搜索是跨域壳层能力，不是单独业务主对象。

当前基线下，全局搜索优先覆盖：

- 内容
- 联系人
- 聊天记录
- 群组
- 小趣搜

共享主页搜索在 `shared-homepage-network` 中单独定义，并在后续与全局搜索进一步收口。

### 12.3 统一原则

- 发现负责“把什么推给你”
- 搜索负责“让你主动找到什么”
- 两者都不能发明第二套对象命名

### 13.1 兴趣型关系主页

- `圈主`
- `圈管`

### 13.2 组织型关系主页

- `负责人`
- `管理员`

### 13.3 群级

- `群主`
- `群管`

### 13.4 共享主页侧

- `创建者`
- `贡献者`
- `维护者`
- `认领方`

### 13.5 角色原则

- `圈主` 只属于兴趣型关系主页
- `负责人` 只属于组织型关系主页
- `群主` 只属于真实的群单元
- 学校、公司、院系、班级等前台不再硬套“圈主”
- 认领共享主页不等于获得用户内容或口碑的控制权

## 15. 前后端一体化约束

### 15.1 产品词与内部词分层

- 前台不使用：`实体`、`目录`、`directory`
- PRD 能力名统一使用：`共享主页`
- 云侧服务名允许继续使用：`entity-service`
- `圈子` 不再作为全局总称
- `群组` 只作为入口词，不是所有详情页的统一页名

### 15.2 单一真相源

- 页面路由、surface、operation：metadata 唯一真相源
- 全局概念与命名：本文件 + 全局术语表
- L1/L2/L3 场景行为：各 feature-tree 节点规格

### 15.3 iOS 原生体验要求

后续所有 `/prd` 与 `/design` 必须默认满足：

- iOS 原生大标题与沉浸式头图体验
- 原生手势返回、sheet、图片浏览、半星评分与轻触觉反馈
- 搜索选择器与发布器流程保持单手友好与低认知负担
- 在概念命名收敛前，不牺牲原生流畅度与理解成本

## 16. baseline 范围冻结

在本概念基线下，baseline 阶段冻结如下：

- 身份：冻结 `OwnerAccount / SubAccount` 双层主体边界
- 主页：冻结 `个人主页 / 共享主页 / 关系主页` 的三类区分
- 共享主页：仅做 `车型 / 酒店民宿 / 餐厅 / 景点`
- 群组：冻结命名体系与兴趣型 / 组织型双模板
- 群：冻结为 `公共群 / 自建群`
- 内容：冻结为 `笔记 / 作品 / 提问 / 口碑`
- 会话：冻结 `会话 / 消息 / 会话列表` 的全局定义
- 助手：冻结 `小趣 / 小趣搜` 的用户词和连接层角色
- 搜索与发现：冻结“发现是横切推荐层、搜索是横切查找层”的定位

以下进入后续规划：

- 多共享主页关联的复杂关系主页
- 学校 / 公司的复杂组织树治理细节
- 主理型主页自助创建与立即归属
- 交易与预约闭环
