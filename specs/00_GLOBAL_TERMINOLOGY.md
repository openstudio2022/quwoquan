# 全局术语与命名规则

> 适用阶段：`/explore`、`/prd`、`/design`、`/baseline`
>
> 本文档是跨产品、跨端云、跨 feature-tree 的统一术语入口。

---

## 1. 文档目的

本文件用于解决以下问题：

- 同一个概念在不同文档、不同页面、不同服务里被多次命名
- 用户语言、PRD 语言、技术语言没有分层
- 记录设计稿、旧 spec、metadata、代码注释对同一对象使用了不同叫法

本文件不替代 metadata 中的字段、route、surface、operation 定义；  
本文件只定义**概念名称、使用边界、优先级和迁移口径**。

与 `00_PRODUCT_CONCEPT_SYSTEM.md` 的分工：

- `00_PRODUCT_CONCEPT_SYSTEM.md` 负责“对象与关系”
- 本文件负责“命名与用词”

两份文档不建议直接合并，因为：

- 概念系统更稳定，更新频率低
- 术语表更偏规则、迁移与纠偏，更新频率高

---

## 2. 优先级规则

后续写 PRD、设计文档、页面文案和服务说明时，统一按下面的优先级执行：

1. `metadata`
仅负责 `path / route / surface / operation / field / enum` 的机器真相源。

2. `00_PRODUCT_CONCEPT_SYSTEM.md + 本术语表`
负责全局产品概念、用户语言、内部抽象词和命名边界。

3. `feature-tree 各 L1/L2/L3 spec`
负责场景行为、流程、验收与页面细节，不得无说明地重新发明全局名词。

4. `记录 openspec / 旧设计文档 / 旧 UI 文案`
仅作背景参考；若与本术语表冲突，以本术语表为准。

## 3. 品牌与产品层术语

| 层级 | 术语 | 是否前台使用 | 定义 | 示例 |
|------|------|-------------|------|------|
| 品牌 | `趣我圈` | 是 | App 母品牌名 | App 名、品牌传播 |
| 产品定位 | `AI 群组社区` | 可用于对外描述 | App 的一句话产品定位 | 应用商店副标题、介绍页 |
| 关系入口 | `群组` | 是 | 所有可加入、可沉淀关系的地方的统一入口词 | 首页 Tab、搜索分类 |
| AI 助手 | `小趣` | 是 | App 内的统一助手名 | 助手页、会话内 @小趣 |
| 搜索结果页助手入口 | `小趣搜` | 是 | 搜索里的 AI 结果入口 | 网络结果页左侧 tab |

### 使用规则

- `趣我圈` 的“圈”是品牌解释，不等于产品层统一入口词
- 一级入口统一叫 `群组`
- `圈子` 不再作为全局总称

## 4. 身份与人

| 层级 | 术语 | 是否前台使用 | 定义 | 示例 |
|------|------|-------------|------|------|
| 用户语言 | `身份` | 是 | 用户在 App 内的真实使用主体心智 | 切换身份 |
| 用户语言 | `个人主页` | 是 | 围绕某个身份建立的长期页面 | 进入个人主页 |
| 设置/管理语言 | `子账号` | 可局部使用 | 多身份管理场景下的产品词 | 子账号管理 |
| PRD/技术术语 | `SubAccount / 子账号` | 否 | App 世界的默认主体 | 发内容、加群、聊天 |
| PRD/技术术语 | `OwnerAccount / 主控账号` | 否 | 登录、安全、恢复、设备与通讯录发现容器 | 登录态与恢复 |
| 搜索/聊天用户词 | `联系人` | 是 | 可用于搜索、聊天、通讯录发现的人 | 搜索联系人 |

### 使用规则

- 前台默认先说 `身份`，不是 `主控账号`
- `OwnerAccount` 不能被当成 App 世界的默认主体
- `联系人` 不等于 `用户主页`，在搜索与聊天语境中优先表示“可发起或进入会话的人”

## 5. 主页体系

| 层级 | 术语 | 是否前台使用 | 定义 | 示例 |
|------|------|-------------|------|------|
| 泛称 | `主页` | 可在上下文清晰时使用 | 长期承接某个主体或事物的信息页面 | 进入主页 |
| 用户语言 | `个人主页 / 用户主页 / 作者主页` | 是 | 围绕身份建立的主页 | 查看作者主页 |
| 用户语言 | `车型主页 / 酒店主页 / 景点主页 / 餐厅主页` | 是 | 围绕具体事物建立的主页 | 认领酒店主页 |
| PRD 术语 | `共享主页` | 否 | 对各种具体事物主页的统一能力名 | 共享主页网络 |
| 内部抽象 | `关系主页` | 否 | 群组详情页的统一抽象 | 兴趣型 / 组织型 |
| 技术术语 | `entity-service` | 否 | 共享主页域的云侧服务名 | `entity-service` |

### 使用规则

- `主页` 是泛称，不独占某一种页面
- 有歧义时必须明确写 `个人主页`、`共享主页` 或 `关系主页`
- 前台不出现 `实体`
- 前台不出现 `directory`
- PRD 不用“对象平台”，统一用 `共享主页`
- 服务层保留 `entity-service`

## 6. 群组与关系沉淀

| 层级 | 术语 | 是否前台使用 | 定义 | 示例 |
|------|------|-------------|------|------|
| 一级入口 | `群组` | 是 | 所有可加入、可沉淀关系的地方的总入口 | 首页 Tab、搜索分类 |
| 详情页类型 | `圈子` | 是 | 兴趣型关系主页 | `摄影圈`、`L9车友圈` |
| 详情页类型 | `学校 / 院系 / 班级 / 公司 / 部门` | 是 | 组织型关系主页的前台名称 | `复旦大学`、`推荐算法部` |
| 子单元 | `群` | 是 | 关系主页内的可加入子单元 | 公共群、自建群 |
| 群类型 | `公共群` | 是 | 由关系主页管理侧创建的群 | `华东公共群` |
| 群类型 | `自建群` | 是 | 成员自组织创建的群 | `摄影小组` |

### 使用规则

- `圈子` 不再作为全局总称
- `群组` 是总入口，不是具体详情页名
- 组织型详情页前台直接显示具体类型或节点名
- 群不是聊天的同义词，聊天只是群的一种能力

## 7. 内容与发布

| 层级 | 术语 | 是否前台使用 | 定义 |
|------|------|-------------|------|
| 动作 | `发布` | 是 | 全局统一内容动作 |
| 内容类型 | `笔记` | 是 | 轻量记录、动态、经验、攻略类内容 |
| 内容类型 | `作品` | 是 | 完成度更高、展示感更强的内容 |
| 内容类型 | `提问` | 是 | 面向求助、咨询、问答场景的内容 |
| 内容类型 | `口碑` | 是 | 绑定共享主页的结构化评价内容 |
| 表达形式 | `图文 / 视频 / 文章` | 是 | 用户开始创作的方式 |
| 发现/投影视角 | `微趣 / 美图 / 视频 / 文章` | 可前台使用 | 发现流里的展示或内容家族视角 |

### 使用规则

- 全局统一叫 `发布`，不全局统一叫 `发帖`
- `发帖` 仅可在明确论坛语境中局部使用，不得作为全局动作名
- `口碑` 必须绑定一个共享主页
- `笔记 / 作品 / 提问` 可选择绑定共享主页
- `笔记 / 作品 / 提问 / 口碑` 是业务内容类型
- `微趣 / 美图 / 视频 / 文章` 是发现或媒体展示维度，不与业务类型混为一层

---

## 8. 会话与消息

| 层级 | 术语 | 是否前台使用 | 定义 |
|------|------|-------------|------|
| 产品词 | `趣聊` | 是 | 聊天与会话能力的前台总称 |
| 用户语言 | `会话` | 是 | 一个实时沟通上下文 |
| 用户语言 | `消息` | 是 | 会话里的单条沟通内容 |
| 用户语言 | `聊天记录` | 是 | 搜索和查看记录消息时的说法 |
| 用户语言 | `会话列表 / 收件箱` | 可前台使用 | 会话聚合入口 |
| 技术术语 | `Conversation / Message` | 否 | 会话与消息域核心对象 |

### 使用规则

- `群` 不等于 `会话`
- 群可以有会话，会话也可以独立于群存在
- 搜索里优先说 `聊天记录`，不直接暴露 `Conversation`

---

## 9. 助手与搜索

| 术语 | 是否前台使用 | 定义 |
|------|-------------|------|
| `小趣` | 是 | 全局统一助手名 |
| `小趣搜` | 是 | 搜索里的 AI 结果入口 |
| `全局搜索` | 是 | 跨内容、联系人、聊天记录、群组与后续主页的搜索壳层 |
| `最近在搜` | 是 | 搜索首页记录区块 |
| `网络结果页` | 是 | 搜索 query 进入后的综合结果页 |
| `Run / Stream / 学习事件 / 反馈注入` | 否 | 助手运行与学习闭环的内部术语 |

### 使用规则

- `小趣` 是产品词，不直接让用户理解 `assistant-run-learning`
- `小趣搜` 只用于搜索结果语境
- 搜索页的统一用户词是 `群组`，不是 `圈子`

---

## 10. 分发与挂载

| 术语 | 是否前台使用 | 定义 |
|------|-------------|------|
| `话题` | 是 | 内容分发与聚合锚点 |
| `地点` | 是 | 与地理位置相关的分发与浏览锚点 |
| `关联主页` | 是 | 内容与某个具体事物主页的绑定 |
| `相关群组` | 是 | 主页推荐的关系沉淀入口 |
| `发现流` | 是 | 推荐内容主消费流 |

### 使用规则

- `话题`、`地点` 不替代共享主页
- `话题` 不等于标签乱堆砌
- `内容挂载主页` 优先于“创建抽象对象”的用户心智

---

## 11. 角色体系

| 场景 | 正式术语 |
|------|----------|
| 兴趣型关系主页 | `圈主 / 圈管` |
| 组织型关系主页 | `负责人 / 管理员` |
| 群级 | `群主 / 群管` |
| 共享主页维护侧 | `创建者 / 贡献者 / 维护者 / 认领方` |

### 使用规则

- 学校、公司、院系、班级等不得再用 `圈主`
- `群主` 只属于真实的群单元
- 共享主页认领方不等于内容与口碑拥有者

---

## 12. 禁用或仅内部使用术语

| 术语 | 处理方式 | 原因 |
|------|----------|------|
| `实体` | 前台禁用，内部可用 | 太技术，不符合用户语言 |
| `directory` | 不作为整域名和产品主词 | 更像黄页/目录，不适合主页网络 |
| `对象平台` | 禁用 | 用户难以理解，语义过宽 |
| `圈子` 作为全局总称 | 禁用 | 会与学校/公司等组织型主页冲突 |
| `社区` 作为全局总称 | 不建议 | 过泛，弱化组织与群的边界 |
| `发帖` 作为全局动作 | 禁用 | 论坛语义过强，覆盖不了作品/动态/口碑 |
| `官方群` | 不作为前台主词 | 后台感过强 |
| `私有群` | 可作为规则描述，不作为一级产品词 | 用户更容易理解 `自建群` |

## 13. 旧词迁移映射

| 记录叫法 | 新口径 |
|----------|--------|
| 圈子（全局总称） | 群组 |
| 圈子详情页（兴趣型） | 圈子 |
| 学校圈 / 公司圈 | 学校 / 公司主页（组织型关系主页） |
| 实体 / 对象页 | 共享主页 / 具体类目主页 |
| 发帖 | 发布 |
| 帖子流 | 内容流 |
| 官方群 | 公共群 |
| 私有群 | 自建群 |
| 作品/笔记/文章混称为创作条目 | 统一归入内容体系，再按内容类型和表达形式区分 |
| 圈主用于学校、组织场景 | 负责人 |
| 圈子搜索结果 | 群组结果 |

## 14. 服务与 API 命名建议

### 14.1 服务级

| 层级 | 建议名称 |
|------|----------|
| 用户前台 | 个人主页、共享主页、群组、群、发布、口碑、趣聊、小趣 |
| PRD 能力名 | 共享主页、群组体系、内容发布体系、会话体系、助手体系 |
| 云侧服务名 | `entity-service`、`circle-service`、`chat-service`、`content-service`、`assistant-service` |

### 14.2 API 语义建议

可以保留技术稳定性，但在文档说明中补用户语言映射：

- `SearchEntities`：搜索共享主页候选
- `GetEntityPage`：获取共享主页详情
- `SuggestEntity`：补充共享主页
- `RequestEntityClaim`：申请认领共享主页

说明原则：

- API 名可暂保留 `entity`
- 页面文案与 PRD 说明不直接暴露 `entity`

## 15. 记录文档解释规则

阅读记录文档时，如出现以下情况，统一按本表解释：

| 记录文档出现 | 当前应理解为 |
|--------------|--------------|
| 圈子作为全局频道/总称 | 群组 |
| 学校圈、公司圈 | 组织型关系主页 |
| 圈主用于学校、组织场景 | 负责人 |
| 发帖、帖子 | 发布、内容 |
| 实体、对象 | 共享主页 / 具体事物主页 |
| 圈子搜索结果 | 群组结果 |
| 圈子频道分类 | 群组分类 facet（内部仍来自 circle 域） |

## 16. 后续 PRD 检查清单

任何新 PRD、设计稿或 metadata 规划，在落笔前都要先过这几个问题：

1. 这个功能面对用户时，看到的是哪一个正式术语？
2. 这个术语属于前台用户语言、PRD 术语还是技术术语？
3. 是否误把 `圈子` 当作全局总称使用？
4. 是否误把 `实体`、`directory` 暴露给用户？
5. 是否把 `群` 和 `会话` 混成一个对象？
6. 是否把业务内容类型与发现展示维度混成一个维度？
7. 是否违反了“身份 / 主页 / 群组 / 群 / 内容 / 会话 / 小趣”这套全局对象分工？

只要有一项回答不清，就先补术语，不直接进入实现。
